🏷️
What are tags? Start with real life.
A tag is simply a label you stick on a piece of content. That label helps categorize and find the content later.
Real life analogies you already understand
📚
Library books have subject labels. A book about "The History of Cricket in India" has labels on the spine and in the catalog: Sports, Cricket, India, History, Non-fiction. You can find all books about Cricket by searching that label — even books you've never heard of.
📧
Gmail labels. You label emails as "Work", "Family", "Bills". When you click "Work", all work emails appear — regardless of who sent them or when. The label connects them.
📸
Instagram hashtags. #cricket #india #ipl — these are tags. Anyone searching #cricket finds your photo along with millions of others tagged the same way. Tags create connections between unrelated pieces of content.
Tags in AEM — concrete example
An article titled "India beats Australia in final" gets tagged by the editor:
Sports Sports / Cricket Geography / India Geography / Australia Year / 2024
Now the website can automatically:
— Show this article on the "Cricket" topic page
— Show it in "Related Articles" on other cricket articles
— Count: "Sports has 4,521 articles, Cricket has 1,203 articles"
— Let users filter: "Show me cricket articles from India in 2024"
How tags are stored in AEM (the technical reality)
In AEM, tags are not just strings like "#cricket". They are actual nodes (entries) stored in the database at a specific location. Think of it like this:
🗄️
Think of AEM's database (the JCR) as a giant folder system. Tags live in their own special folder called /content/cq:tags. Each tag is a folder inside this folder. When you "apply a tag to a page", AEM stores the tag's address (its path) inside the page's properties — like writing a reference to the tag folder on the page's label.
/content/cq:tags/ ← where all tags live
├── sports/ ← a namespace (big category)
│ ├── cricket/ ← a tag
│ ├── football/
│ └── tennis/
├── geography/
│ ├── india/
│ └── australia/
└── year/
└── 2024/
When an article is tagged with "Sports/Cricket", AEM stores the text "sports:cricket" in that article's properties. That's the tag ID — the address that points to the /content/cq:tags/sports/cricket folder.
What is a namespace?
A namespace is the top-level folder that groups related tags together. It is the very first part of a tag ID, before the colon.

sports:cricket → namespace is sports, tag is cricket
geography:india → namespace is geography, tag is india

Namespaces prevent conflicts. Two different teams can both have a tag called "tennis" — one under sports:tennis and one under indoorgames:tennis — and they don't interfere.

Think of a namespace like a last name. "John Smith" and "John Jones" are different people even though they share the first name "John". sports:tennis and entertainment:tennis are different tags even though they share "tennis".
Tag ID format — the address of a tag
Every tag has a unique ID written in this format:
namespace:path/to/tag

sports:cricket — namespace "sports", tag "cricket"
sports:cricket/ipl — namespace "sports", parent tag "cricket", child tag "ipl"
geography:india/chennai — namespace "geography", parent "india", child "chennai"

Tags can be nested (parents and children), just like folders inside folders.
When stored on a page, the cq:tags property holds an array of these IDs:
cq:tags = ["sports:cricket", "geography:india", "year:2024"]
🗂️
Tag structure in the repository
Let's look at what a tag actually looks like inside AEM's database, and what properties it has. This helps you understand what information you can read from a tag in your Java code.
A tag node — what it looks like in CRXDE
When you open CRXDE Lite (AEM's file explorer, at /crx/de) and navigate to a tag, you see something like this:
/content/cq:tags/sports/cricket
jcr:primaryType = "cq:Tag" ← the node type that identifies this as a tag
jcr:title = "Cricket" ← the display name
jcr:title.hi = "क्रिकेट" ← Hindi title (localised)
jcr:title.ta = "கிரிக்கெட்" ← Tamil title
jcr:description = "All articles about cricket"
sling:resourceType = "cq/tagging/components/tag"
The node type cq:Tag is what makes AEM recognise this as a tag — not just a random folder. The jcr:title is what shows up in the AEM tag picker UI when editors are tagging content. The localised titles (.hi, .ta) let the tag display in different languages depending on the user's locale.
What is stored ON a page when you tag it?
When an editor tags an article in AEM's page editor, this is what gets saved in the page's JCR properties:
/content/news/india-beats-australia/jcr:content
jcr:title = "India beats Australia in final"
cq:tags = [
"sports:cricket", ← tag ID
"geography:india", ← tag ID
"geography:australia", ← tag ID
"year:2024" ← tag ID
]
The cq:tags property is an array of tag ID strings. AEM resolves each ID to its actual tag node at /content/cq:tags/... when needed. The page doesn't copy the tag — it just stores the tag's address (like a shortcut/symlink to the tag folder).
Tags can be nested — parents and children
Tags can contain other tags, just like folders inside folders. This lets you build a hierarchy from broad to specific:
sports: ← namespace
cricket ← parent tag (ID: sports:cricket)
ipl ← child tag (ID: sports:cricket/ipl)
test-matches ← child tag (ID: sports:cricket/test-matches)
womens-cricket ← child tag (ID: sports:cricket/womens-cricket)
football
premier-league
champions-league
If you tag an article with sports:cricket/ipl, it is also considered tagged with the parent sports:cricket. So a query for all cricket articles will include IPL articles too. This is called tag inheritance — very useful for broad searches.
How to create tags in the AEM UI (no code needed)
Go to Tools → Tagging in the AEM navigation. This opens the Tag Manager console where you can:

— Create a new namespace (like "sports")
— Create tags inside a namespace (like "cricket" inside "sports")
— Add titles in multiple languages
— See how many pages/assets are tagged with each tag (the count)
— Move, merge, or delete tags

Merging tags is powerful — if you have "cricket" and "Cricket" as two separate tags (created by different editors), you can merge them so all content tagged with either one now uses just one canonical tag.

The Tagging API — Java code for tags
The Tagging API is a set of Java classes AEM provides so your code can work with tags: resolve a tag from its ID, apply tags to pages, create new tags, read tags from a page. The main class is TagManager.
What is TagManager?
🧑‍💼
Think of TagManager like the librarian at the library.

You say: "I need the book with the label CRICKET." The librarian (TagManager) knows exactly where the cricket section is, fetches the tag object, and hands it to you.

You say: "I want to put the CRICKET label on this book." The librarian does it for you and updates the catalog.

You say: "How many books have the CRICKET label?" The librarian counts and tells you.
You get a TagManager by calling resourceResolver.adaptTo(TagManager.class). This asks AEM to give you the TagManager service that is tied to your current session (and therefore your service user's permissions).
Core operation 1: Resolving a tag (getting a tag object from its ID)
You have the tag ID as a string — like "sports:cricket". You want the actual Tag object with all its properties.
Resolving a tag — every line explained
1// Get the TagManager from your ResourceResolver
2TagManager tagManager = resolver.adaptTo(TagManager.class);
3
4// Resolve a tag by its ID — ID format is namespace:path
5Tag cricketTag = tagManager.resolve("sports:cricket");
6
7// Always check — if the tag doesn't exist, resolve() returns null
8if (cricketTag != null) {
9
10 // Get the display title in English
11 String title = cricketTag.getTitle(Locale.ENGLISH);
12 // title = "Cricket"
13
14 // Get the tag's path in the repository
15 String path = cricketTag.getPath();
16 // path = "/content/cq:tags/sports/cricket"
17
18 // Get how many pages/assets are tagged with this tag
19 long count = cricketTag.getCount();
20 // count = 1203 (1203 articles tagged with cricket)
21}
L2
resolver.adaptTo(TagManager.class) — adaptTo() is AEM's way of getting a specific service or view from an existing object. You have a ResourceResolver (your database connection). By calling adaptTo(TagManager.class) you ask: "give me a TagManager that works within this session." The TagManager will use the same service user permissions as the resolver — so it can only access tags the service user is allowed to see.
L5
tagManager.resolve("sports:cricket") — takes a tag ID string and looks up the corresponding tag node at /content/cq:tags/sports/cricket. Returns a Tag object if found, or null if the tag doesn't exist. The Tag object is like a Java wrapper around the tag node — it gives you convenient methods to read the tag's properties without writing raw JCR code.
L8
if (cricketTag != null) — always check for null before using the tag. If someone deleted the "sports:cricket" tag from the tag manager UI, or if there's a typo in the ID, resolve() returns null. Calling methods on a null object throws a NullPointerException and crashes your code.
L11
cricketTag.getTitle(Locale.ENGLISH) — returns the human-readable display title of the tag in the requested language. If you stored a Tamil title (jcr:title.ta) on the tag, you'd call getTitle(new Locale("ta")) to get the Tamil version. If no localised title exists for that language, AEM falls back to the default jcr:title.
L19
cricketTag.getCount() — returns how many content nodes (pages, assets) across the entire repository are tagged with this tag. AEM maintains this count automatically whenever tags are applied or removed. You use this to display "Cricket (1,203)" in a tag cloud or navigation menu, showing readers which topics have the most content.
Core operation 2: Reading tags FROM a page
You have a page resource. You want to know what tags have been applied to it.
Getting tags from a page resource
1// Get the page's jcr:content node — that's where cq:tags is stored
2Resource pageContent = resolver.getResource(
3 "/content/news/india-beats-australia/jcr:content");
4
5// Read the raw cq:tags property — an array of tag ID strings
6String[] tagIds = pageContent.getValueMap()
7 .get("cq:tags", String[].class);
8// tagIds = ["sports:cricket", "geography:india", "year:2024"]
9
10// OR — use TagManager to get full Tag objects directly
11Tag[] tags = tagManager.getTagsForSubtree(pageContent, false);
12
13// Loop through each tag and print its title
14for (Tag tag : tags) {
15 log.info("Tag: {}", tag.getTitle());
16 // prints: "Tag: Cricket", "Tag: India", "Tag: 2024"
17}
L2-3
resolver.getResource(".../jcr:content") — a page in AEM has two nodes: the page node itself (/content/news/india-beats-australia) and a child node called jcr:content where all the page's actual properties live — title, tags, content, metadata. The cq:tags property is on the jcr:content node, so that's what you need to read.
L6-7
pageContent.getValueMap().get("cq:tags", String[].class) — getValueMap() returns all the properties of this node as a Map (key-value pairs). .get("cq:tags", String[].class) reads the cq:tags property as an array of Strings. The second argument String[].class tells AEM what type to return — AEM handles the conversion automatically. Returns null if the property doesn't exist (no tags applied).
L11
tagManager.getTagsForSubtree(pageContent, false) — a convenience method that reads the cq:tags property, resolves each tag ID to its full Tag object, and returns them as an array. The second argument false means: only return tags directly on this node (not from child nodes). Pass true to also collect tags from all child nodes (useful for getting all tags used within a whole section of the website).
L14-17
for (Tag tag : tags) — a standard Java for-each loop. For each Tag object in the array, you can call any Tag method: tag.getTitle() for the display name, tag.getTagID() for the ID string like "sports:cricket", tag.getPath() for the repository path, tag.getCount() for how many items are tagged with it.
Core operation 3: Applying tags to a page in code
Your code needs to programmatically apply tags to a page — for example, an import job that auto-tags 10,000 product pages based on their category.
Applying tags programmatically
1// First, resolve the tags you want to apply
2Tag cricketTag = tagManager.resolve("sports:cricket");
3Tag indiaTag = tagManager.resolve("geography:india");
4
5// Get the page's jcr:content resource
6Resource pageContent = resolver.getResource(
7 "/content/news/my-article/jcr:content");
8
9// Apply the tags — this sets the cq:tags property on that resource
10tagManager.setTags(
11 pageContent,
12 new Tag[]{cricketTag, indiaTag},
13 true
14);
15
16// Save the changes
17resolver.commit();
L2-3
Resolve before applying. You always need the actual Tag objects — not just strings. If a tag doesn't exist (resolve returns null), you cannot apply it. In a real import job you'd check for null and either skip that tag or create it first (using tagManager.createTag()).
L10-14
tagManager.setTags(resource, tagsArray, mergeWithExisting) — sets the cq:tags property on the given resource.

resource — the jcr:content node of the page you want to tag.
tagsArray — the array of Tag objects to apply. You create this with new Tag[]{tag1, tag2, ...}.
true (third argument) — merge with existing tags. If the page already has tags, keep them and add these new ones. If you pass false, it replaces all existing tags with just these new ones. For an import that auto-adds category tags without removing manually-applied tags, always use true.
L17
resolver.commit() — setTags() modifies the JCR node in memory but doesn't save to disk yet. commit() persists the changes. Without this, tags are lost when the session closes. Same concept as Ctrl+S — always call after writes.
Core operation 4: Creating a new tag in code
Creating a tag programmatically
1Tag newTag = tagManager.createTag(
2 "sports:cricket/womens-t20",
3 "Women's T20",
4 "All articles about Women's T20 cricket",
5 false
6);
7resolver.commit();
L2
"sports:cricket/womens-t20" — the full tag ID path for the new tag. This creates the tag as a child of sports:cricket. AEM creates any missing parent nodes automatically — so if sports:cricket doesn't exist yet, it creates that too.
L3
"Women's T20" — the human-readable title stored as jcr:title. This is what shows in the AEM UI when editors pick tags.
L4
"All articles about Women's T20 cricket" — the description stored as jcr:description. Helps editors understand what to tag with this tag. Shown as a tooltip in the tag picker UI.
L5
false — whether to auto-create if it doesn't exist (true) or fail if the tag already exists (false). Passing false means: if this tag already exists, throw an exception instead of silently doing nothing. Useful in import jobs where you want to know if something already exists vs was freshly created.

Tag Search and Listing
Now the most powerful part: using tags to FIND content. "Give me all articles tagged cricket" or "List all tags under Sports with their article counts."
Real world use case — "Related Articles" component
📰
You are building a "Related Articles" section that appears at the bottom of every article. It should show 5 other articles that share at least one tag with the current article.

How it works:
1. Read the current article's tags → ["sports:cricket", "geography:india"]
2. Search all pages that have ANY of those tags
3. Exclude the current page itself
4. Show the top 5 results

This entire feature runs on the tagging system. No manual curation needed.
Searching for pages by tag — using QueryBuilder
AEM's QueryBuilder lets you search the repository. You build a query as a Map of parameters and execute it. Think of it like building a SQL WHERE clause but in a key-value format.
Find all articles tagged "sports:cricket"
1// Import needed classes
2import com.day.cq.search.QueryBuilder;
3import com.day.cq.search.result.SearchResult;
4import java.util.HashMap;
5import java.util.Map;
6
7@Reference
8private QueryBuilder queryBuilder;
9
10// Build the search query as key-value pairs
11Map<String, String> params = new HashMap<>();
12
13params.put("type", "cq:Page");
14params.put("path", "/content/news");
15params.put("tagid", "sports:cricket");
16params.put("tagid.property", "jcr:content/cq:tags");
17params.put("p.limit", "5");
18
19// Execute the query — need a JCR Session
20Session session = resolver.adaptTo(Session.class);
21Query query = queryBuilder.createQuery(
22 PredicateGroup.create(params), session);
23SearchResult result = query.getResult();
24
25// Loop through matched pages
26for (Hit hit : result.getHits()) {
27 String pagePath = hit.getPath();
28 log.info("Found cricket article: {}", pagePath);
29}
L7-8
@Reference QueryBuilder queryBuilder — just like how you @Reference ResourceResolverFactory, you @Reference QueryBuilder. OSGi injects it automatically. QueryBuilder is AEM's search engine API — it translates your Map of parameters into an actual JCR-SQL2 query and runs it against the repository.
L11
new HashMap<>() — a HashMap is a simple key-value store in Java. Like a dictionary: each key maps to one value. You use it here to build the query parameters. QueryBuilder reads this map and understands: "the user wants type=cq:Page, under path=/content/news, with tagid=sports:cricket". It's like filling in a search form programmatically.
L13
params.put("type", "cq:Page") — only return pages (not assets or other node types). cq:Page is the node type for all AEM pages. If you wanted to search assets in the DAM, you'd use "dam:Asset".
L14
params.put("path", "/content/news") — only search within this path. Without this, QueryBuilder searches the ENTIRE repository — all 500,000+ nodes. This is critical for performance. Always narrow the search scope to only the section you care about.
L15-16
params.put("tagid", "sports:cricket") — find items tagged with this specific tag ID. AEM also searches child tags — so if there's a child tag sports:cricket/ipl, articles tagged with IPL will also be returned (because IPL is a cricket sub-tag).

params.put("tagid.property", "jcr:content/cq:tags") — tells QueryBuilder exactly which property on the page node to check. The tags are stored in jcr:content/cq:tags, not directly on the page node. The tagid.property param is the path to that property relative to the page node.
L17
params.put("p.limit", "5") — return at most 5 results. Always set a limit in production — without it, the query could return thousands of results at once, consuming huge amounts of memory. Use "-1" to get unlimited results only when you truly need all of them (e.g. in a batch job).
L20
resolver.adaptTo(Session.class) — QueryBuilder needs a raw JCR Session (the lower-level connection object that the ResourceResolver wraps). adaptTo(Session.class) extracts this from your existing resolver — no new login needed. The session carries the same service user permissions as the resolver.
L26-29
result.getHits() — returns the list of matching nodes. Each Hit object represents one matched page. hit.getPath() gives you the JCR path of that page. You can also call hit.getResource() to get the full Resource object and read its properties.
Listing child tags — building a tag cloud or navigation menu
You want to display all sub-tags under "Sports" in a navigation menu, each showing how many articles it has.
List all sports tags with their article counts
1// Resolve the parent tag
2Tag sportsTag = tagManager.resolve("sports:");
3// Note: "sports:" with a colon and nothing after = the namespace itself
4
5if (sportsTag != null) {
6 // listChildren() returns an Iterator over all direct child tags
7 Iterator<Tag> children = sportsTag.listChildren();
8
9 while (children.hasNext()) {
10 Tag child = children.next();
11
12 String title = child.getTitle();
13 long count = child.getCount();
14 String id = child.getTagID();
15
16 log.info("{} ({}) — id: {}", title, count, id);
17 // prints: "Cricket (1203) — id: sports:cricket"
18 // prints: "Football (876) — id: sports:football"
19 }
20}
L2
tagManager.resolve("sports:") — to resolve the namespace itself (not a specific tag inside it), you use the namespace name followed by just a colon. This gives you the sports namespace tag node, whose children are cricket, football, tennis etc. Without the colon, it would look for a tag literally named "sports" inside the default namespace.
L7
sportsTag.listChildren() — returns a Java Iterator over the immediate child tags. An Iterator is like a cursor that walks through a list one item at a time. It only gives you direct children — not grandchildren. If you want ALL descendants (cricket, cricket/ipl, cricket/test-matches etc.), you'd need to recurse into each child and call listChildren() on them too.
L9-10
while (children.hasNext()) { Tag child = children.next(); } — this is the standard Java Iterator pattern. hasNext() returns true if there are more items. next() gets the next item and advances the cursor. You loop until hasNext() returns false (no more children).
L12-14
child.getTitle() — the display name like "Cricket".
child.getCount() — how many pages/assets across the whole repository are tagged with this specific tag.
child.getTagID() — the full tag ID string like "sports:cricket". You use this ID to link to a tag listing page or as a query parameter in a URL like /content/news.tag.sports:cricket.html.
Putting it all together — the "Related Articles" component in HTL
This is how all the above comes together in a real AEM component. The Sling Model reads current page tags, finds related articles, and the HTL template renders them.
RelatedArticlesModel.java — Sling Model using tagging
1@Model(adaptables = SlingHttpServletRequest.class)
2public class RelatedArticlesModel {
3
4 @SlingObject
5 private ResourceResolver resolver;
6
7 @ScriptVariable
8 private Page currentPage;
9
10 public List<String> getRelatedArticlePaths() {
11 TagManager tm = resolver.adaptTo(TagManager.class);
12 Tag[] tags = tm.getTagsForSubtree(currentPage.getContentResource(), false);
13
14 if (tags == null || tags.length == 0) return Collections.emptyList();
15
16 // Use the first tag to find related articles
17 Map<String, String> params = new HashMap<>();
18 params.put("type", "cq:Page");
19 params.put("path", "/content/news");
20 params.put("tagid", tags[0].getTagID());
21 params.put("tagid.property", "jcr:content/cq:tags");
22 params.put("p.limit", "5");
23
24 List<String> paths = new ArrayList<>();
25 for (Hit hit : queryBuilder.createQuery(
26 PredicateGroup.create(params),
27 resolver.adaptTo(Session.class))
28 .getResult().getHits()) {
29 if (!hit.getPath().equals(currentPage.getPath()))
30 paths.add(hit.getPath());
31 }
32 return paths;
33 }
34}
L1
@Model(adaptables = SlingHttpServletRequest.class) — this is a Sling Model annotation. It registers this class as a Sling Model — a Java class that can be used by HTL templates. adaptables says "this model is created from a SlingHttpServletRequest" — meaning it runs in the context of a web request (when someone opens a page in their browser). The model has access to the current request, page, and resource.
L4-5
@SlingObject ResourceResolver resolver — in a Sling Model, @SlingObject injects objects from the current Sling context. Here it gives you the ResourceResolver of the current web request — no service user needed because this runs as the page visitor's anonymous session (or logged-in user session).
L7-8
@ScriptVariable Page currentPage — @ScriptVariable injects variables that the Sling engine makes available in every HTL template — like currentPage (the AEM Page object for the page being rendered). currentPage.getContentResource() returns the jcr:content node of the current page — where its tags are stored.
L29-30
if (!hit.getPath().equals(currentPage.getPath())) — exclude the current page from results. Without this check, the "Related Articles" section would include the article you're currently reading as its own related article. The ! means "not equal to" — only add the hit path to results if it's NOT the current page's path.
