Setup
Maven archetype — generate your whole project
Run this one command in your terminal. It creates every module, folder, and config file you need.
Where to run it
locationYour terminal — any folder you want your project created in
prereqsJava 11, Maven 3.6+, AEM running on localhost:4502
The command
runmvn -B archetype:generate -D archetypeGroupId=com.adobe.aem -D archetypeArtifactId=aem-project-archetype -D archetypeVersion=48 -D appTitle="My Site" -D appId="mysite" -D groupId="com.mysite" -D aemVersion="6.5.0"
What each parameter means
appTitle
Human name shown in AEM — e.g. "Nike India"
appId
Short lowercase ID used in folder names — e.g. "nike". No spaces, no capitals.
groupId
Your Java package base — e.g. "com.nike". Becomes your package prefix.
aemVersion
"6.5.0" for AEM 6.5, or "cloud" for AEM as Cloud Service
After it finishes, open the folder in VS Code: code ./mysite
Setup
Project folders — one job per module
The archetype generates these modules. Every file type has exactly one correct home.
Module → what it contains → where it lands in AEM
core/src/main/java/
All Java files — Sling Models, OSGi Services, Servlets. Deploys as a .jar OSGi bundle.
ui.apps/…/apps/mysite/
Components (HTL + dialogs), ClientLibs, static templates. Deploys to /apps/mysite/ in JCR.
ui.content/…/content/
Sample page content. Deploys to /content/mysite/ in JCR.
ui.content/…/conf/
Editable templates, CF models, policies. Deploys to /conf/mysite/ in JCR.
ui.config/…/osgiconfig/
OSGi .cfg.json config files per environment. Deploys as OSGi configuration.
ui.frontend/src/
Raw SCSS, TypeScript, webpack. Built by npm; output goes into ui.apps clientlibs.
Never put Java in ui.apps. Never put HTL in core. Each module has one job — mixing them breaks the build.
Setup
Build & deploy commands
Run from the project root. AEM must be running on 4502 first.
Full deploy (all changes)
mvn clean install -PautoInstallPackage
Only Java changed
mvn clean install -PautoInstallBundle -pl core — faster, rebuilds just the bundle
Only HTL/dialog changed
mvn clean install -PautoInstallPackage -pl ui.apps
Deploy to Publish
mvn clean install -PautoInstallPackagePublish — installs to port 4503
Skip tests
Add -DskipTests to any command above — much faster
After deploying, go to /system/console/bundles and search your bundle name — it must say "Active". If it says "Installed", your Java imports are broken.

Sling Models
Sling Model Java file — where to create it
Sling Models are Java classes that read JCR properties and prepare data for the HTL template. They live in core — the only Java module.
Location
interfacecore/src/main/java/com/mysite/core/models/CardModel.java
impl classcore/src/main/java/com/mysite/core/models/impl/CardModelImpl.java
verify in AEM/system/console/bundles — search bundle name, must say Active
How HTL connects to your model
card.html: data-sly-use.model="com.mysite.core.models.CardModel"
→
AEM instantiates CardModelImpl, injects JCR props
→
${model.title} calls getTitle() in Java
File naming convention
CardModel.java
The interface — defines public getter methods. HTL references this name in data-sly-use.
CardModelImpl.java
The implementation — has all annotations (@Model, @ValueMapValue, @PostConstruct) and actual logic.
Sling Models
Which annotation to use when
Each annotation serves exactly one purpose. Use the most specific one for each field.
@ValueMapValue
Reading a dialog field that an editor filled in. Maps to a JCR property by matching the field name. e.g. @ValueMapValue private String title; reads the title JCR property.
@ValueMapValue(name=)
When the JCR property name differs from your Java field name — e.g. @ValueMapValue(name="jcr:title") private String pageTitle;
@Default
Fallback value if the JCR property is null or empty — e.g. @Default(values="Read More") private String ctaText;
@SlingObject
Sling framework objects — Resource currentResource, ResourceResolver resourceResolver, SlingHttpServletRequest request
@ScriptVariable
AEM page/node context from HTL — Page currentPage, Node currentNode, PageManager pageManager
@OSGiService
Inject another OSGi service — e.g. @OSGiService private EmailService emailService;
@ChildResource
Inject a child JCR node — e.g. @ChildResource private Resource image; reads the child node named "image"
@PostConstruct
Method that runs AFTER all fields are injected. Put computed logic here — format values, call services, build lists.
Quick rule: dialog fields → @ValueMapValue. AEM objects → @ScriptVariable or @SlingObject. Other services → @OSGiService. Logic after injection → @PostConstruct.
Sling Models
POM changes for Sling Models
Two dependencies plus one manifest entry. Without the manifest entry, AEM won't scan your models and data-sly-use will silently return null.
Add to core/pom.xml dependencies
Sling Models API
org.apache.sling / org.apache.sling.models.api / scope: provided
AEM SDK API
com.adobe.aem / aem-sdk-api / scope: provided — gives you Page, Asset, etc.
Add to bnd-maven-plugin config in core/pom.xml
Sling-Model-Packages
Sling-Model-Packages: com.mysite.core.models — tells AEM which Java package to scan for models. Without this, nothing works.
If your model compiles fine but ${model.title} returns null, check this manifest entry first. It is the most common Sling Model mistake.

ClientLibs
ClientLib node — where CSS & JS live
A ClientLib is a special JCR folder that bundles your CSS and JS for AEM to combine, minify, and serve.
Location
in projectui.apps/…/apps/mysite/clientlibs/site/
in JCR/apps/mysite/clientlibs/site/
Files inside the folder
.content.xml
Node definition — jcr:primaryType=cq:ClientLibraryFolder, categories=["mysite.site"], allowProxy={Boolean}true
css.txt
List of CSS files in load order. First line must be #base=css. Then one file per line: main.css
js.txt
List of JS files in load order. First line must be #base=js. Then: main.js
css/ folder
Your actual .css files referenced in css.txt
js/ folder
Your actual .js files referenced in js.txt
Key .content.xml properties
categories
The unique ID — e.g. ["mysite.site"]. This is what you reference in HTL to include the library.
dependencies
Other libraries to load before this one — e.g. ["jquery"]
embed
Other libraries to inline into this one — combines into a single file to reduce HTTP requests.
allowProxy
{Boolean}true — required so the library works on the Publish layer via the /etc.clientlibs/ proxy path.
ClientLibs
Including ClientLibs in your page
You include ClientLibs in the page component's head.html (CSS) and bodyEnd.html (JS) — not in individual component HTL files.
Load CSS only
In head.html: data-sly-use.clientlib="/libs/granite/sightly/templates/clientlib.html" then call ${clientlib.css @ categories='mysite.site'}
Load JS only
In bodyEnd.html: call ${clientlib.js @ categories='mysite.site'}
Load both at once
Call ${clientlib.all @ categories='mysite.site'}
Debug: see individual files
Append ?debugClientLibs=true to any page URL — loads each CSS/JS file individually instead of combined
Inspect what's in a lib
Go to /libs/granite/ui/content/dumplibs.html — lists all categories and their file contents
OSGi
OSGi Service Java file — where and what
Services provide shared functionality (email sending, API calls, caching) to any Java class in AEM. They live in core.
Location
interfacecore/src/main/java/com/mysite/core/services/EmailService.java
implcore/src/main/java/com/mysite/core/services/impl/EmailServiceImpl.java
verify in AEM/system/console/services — search by interface name
Required annotations on the impl class
@Component(service=)
Registers this class as an OSGi component — e.g. @Component(service=EmailService.class)
@Designate(ocd=)
Links to a config interface so the config form appears in OSGi console — e.g. @Designate(ocd=EmailConfig.class)
@Activate
Method called when bundle starts. Reads config values, sets up connections.
@Modified
Called when OSGi config updates without restart. Usually just calls @Activate again.
@Deactivate
Called when bundle stops. Close connections, release resources.
@Reference
Inject another OSGi service — e.g. @Reference private ResourceResolverFactory rrf;

JCR
Important JCR properties — what to look for where
When you're in CRXDE and confused, check these first. They tell you everything about how a node connects to the rest of AEM.
sling:resourceType
The single most important property in AEM. On any content node — tells AEM which component renders it. Check this first for any rendering problem. Path is relative to /apps.
jcr:primaryType
cq:Page = a page. cq:Component = a component definition. nt:unstructured = generic data node (component content). cq:ClientLibraryFolder = a clientlib.
jcr:title
Human name. On pages = the page title. On components = shown in component browser. On templates = shown in template picker.
sling:resourceSuperType
On a component — the parent it inherits from. AEM walks up this chain when looking for rendering scripts or dialog fields that aren't found locally.
cq:template
On a page's jcr:content — which template was used to create this page. Path to the template node under /conf.
categories
On a ClientLib node — the library's unique ID used in HTL includes.
allowedPaths
On a template — which content paths editors can use this template in. e.g. /content/mysite(/.*)$
The fastest way to understand any unknown AEM behavior: go to CRXDE, navigate to the node, click Properties. Everything is visible there — sling:resourceType, component data, configs.
