Replication — moving content from author to publish
AEM always has at least two separate servers: the Author instance and the Publish instance. They are completely separate — different servers, different databases, different URLs. Replication is the bridge between them.
Real life analogy
📰
Think of a newspaper.

The editorial office is where journalists write, editors review, and the layout team formats the pages. It is not open to the public — only staff work there.

The printing press and distribution is separate. Once the paper is approved, it gets sent to the printing press which makes thousands of copies for readers.

Author = editorial office. Editors create and approve content here. Nobody else can see it.
Publish = printing press + distribution. Readers visit here. Only approved content reaches here.
Replication = the process of sending the approved content from editorial to the press.
Author vs Publish — what is different between them
Author Instance
:4502
Where editors log in and create content. Has the full AEM authoring UI — page editor, DAM, Sites console, Workflow inbox. Only staff access. Usually one author server (or a small cluster). Content is NOT visible to the public.
→
Publish Instance
:4503
The live website visitors access. No authoring UI — only serves content. Can have many publish servers behind a load balancer for performance. All public traffic goes here. Only replicated content appears here.
Critical point: When an editor writes a new page on Author, it does NOT automatically appear on Publish. The editor must explicitly click "Activate/Publish" to replicate it. Until then, the page exists only on Author — invisible to the public.

The Author serialises the page's JCR nodes into a content package (like a zip file of JCR data). The Replication Agent sends this package via HTTP POST to the Publish instance's /bin/receive endpoint. Publish installs the package — creating or updating the corresponding JCR nodes. The page is now live.
Triggering replication from Java code
Real use case: your scheduled job finds all pages with scheduledPublishDate = today and activates them automatically at 8 AM without any human clicking Publish.

L5-6
@Reference Replicator replicator — OSGi injects AEM's built-in Replication service. This is the same service that runs when an editor manually clicks Publish in the AEM UI. By @Referencing it, your background code can do exactly what a human editor does — publish or unpublish pages programmatically.
L11
resolver.adaptTo(Session.class) — the Replicator.replicate() method needs a raw JCR Session (not a ResourceResolver). The session identifies WHO is doing the replication — your service user. That service user needs the crx:replicate privilege in its repoinit permissions, otherwise replication will be denied even if the code looks correct.
L14-16
replicator.replicate(session, ReplicationActionType.ACTIVATE, pagePath) — three parameters:

session — identifies the user performing replication. Must have crx:replicate permission.
ACTIVATE — the action. ACTIVATE = send to publish (make it live). DEACTIVATE = remove from publish (take it offline). DELETE = delete from publish.
pagePath — what to replicate. This can be a page path, an asset path, or any JCR node path. AEM replicates the node and all its children.
L20-22
catch ReplicationException — ReplicationException is thrown when replication fails: the publish server is down, network error, the replication agent is disabled, or insufficient permissions. Always catch it and log with the path — so you know exactly which page failed to publish and why.
L30
ReplicationActionType.DEACTIVATE — removes the page from the publish server. Used when content expires or is recalled. After DEACTIVATE, visiting that URL on the publish server returns 404. The page still exists on Author — it's just no longer pushed to Publish.

Replication Agents — the delivery trucks
A Replication Agent is a configuration on the Author server that defines: WHERE to send content (which publish server), HOW to send it (HTTP, credentials), and WHEN to retry if it fails. Think of it as a delivery truck company configuration — you tell it the destination address and the driver credentials.
Real life analogy
🚚
Courier delivery configuration.

You set up a courier account: "Deliver to: 123 Publisher Street. Driver login: replication-user. Transport: HTTPS. If delivery fails: retry 3 times, wait 10 minutes between attempts. Priority: urgent."

Every time you hand a package (published content) to the system, it uses this configured courier account to deliver it. You don't re-configure the delivery each time — the agent handles it every time automatically.
Where replication agents live
Agents are stored in the JCR at /etc/replication/agents.author/. You manage them in the AEM UI at Tools → Replication → Agents on Author. Each agent is a named node with configuration properties.
/etc/replication/agents.author/
├── publish ← default agent to publish server
├── publish2 ← second publish server (if you have one)
├── flush ← dispatcher/CDN cache flush agent
└── static ← static export agent (less common)
The 4 key agent types — what each one does
1
Default Agent (publish) — the main delivery
Sends content from Author to Publish. This is the agent that runs when you click "Activate Page". It POSTs to http://publish-server:4503/bin/receive. The most important agent — without it, nothing gets published. You need one of these per publish server. If you have 5 publish servers, you have 5 default agents pointing to each one.
2
Flush Agent — clearing the CDN/Dispatcher cache
Most AEM setups have a Dispatcher (CDN/cache layer) in front of Publish. When you publish a page, the Dispatcher still serves the OLD cached version to visitors. The Flush Agent tells the Dispatcher: "delete your cached copy of this page" — so the next visitor gets the fresh version. Covered in detail on the next tab.
3
Reverse Replication Agent — content coming back
Moves content FROM Publish back TO Author. Used when visitors submit forms or user-generated content on the live site. That data is first stored on Publish, then this agent pulls it back to Author for processing. Covered in detail on the next tab.
4
Static Agent — exporting as static HTML files
Exports pages as flat .html files to a filesystem instead of to a live server. Used for archiving, regulatory compliance snapshots ("save this page exactly as it was on this date"), or feeding content to legacy systems that can't handle dynamic pages.
What a replication agent config looks like
When you open a replication agent in the AEM UI (Tools → Replication → Agents on Author → publish), you see these settings:
Key replication agent settings and what they mean
Enabled = true ← turn this off to pause replication without deleting the agent
Transport URI = http://publish-server:4503/bin/receive
↑ where to send. Change this per environment
Transport User = admin ← who logs into the publish server to receive
Transport Passwd= admin ← in production use a dedicated replication user
Serialization = Default ← package format (Default = JCR data package)
Retry Delay = 60000 ← wait 60 seconds before retrying on failure
Queue = ordered ← send pages in the order they were published
Testing an agent: Every agent has a "Test Connection" button. Click it and AEM sends a test ping to the publish server. If it returns "Replication test succeeded", your agent is correctly configured and the publish server is reachable. If it fails, check the Transport URI and credentials first.

Reverse Replication & Flush Agent
Two special replication scenarios that solve specific problems: content coming from visitors back to Author (reverse), and clearing the CDN cache after publishing (flush).
Reverse Replication — content flowing backwards
📬
Normal replication: Author → Publish (one way)
Reverse replication: Publish → Author (the other way)

When do you need this? A visitor fills in a contact form on the live bank website. That form submission (their name, email, message) is saved as a JCR node on the Publish server under /content/usergenerated.

But editors and CRM systems are on Author. How does that submission get to Author for processing? The Reverse Replication Agent periodically polls the Publish server, collects new nodes under /content/usergenerated, and copies them to Author.
Real-world example: The bank's contact form. A customer fills in "Call me back about a home loan." That gets stored on Publish. The reverse replication agent picks it up within a minute and copies it to Author at /content/usergenerated/contactforms/2024/01/15/submission-001. A CRM workflow on Author then picks it up, creates a lead in Salesforce, and sends a confirmation email.
The Flush Agent — why your published page might still show old content
🗄️
The problem: Most production AEM setups have a Dispatcher (Apache web server acting as a CDN/cache) in front of the Publish servers. The first time a visitor requests a page, the Dispatcher fetches it from Publish and caches it as a static HTML file on disk. The next 10,000 visitors get that cached file directly — Publish is never hit.

The problem: You publish a page update. It's on Publish. But the Dispatcher is still serving the old cached HTML file to all visitors. They see stale content.

The solution: A Flush Agent. When you publish a page, after the content reaches Publish, the Flush Agent sends a DELETE request to the Dispatcher saying "delete your cached copy of this page." The next visitor gets fresh content from Publish.
The complete publish + flush flow
1. Editor publishes/content/bank/home
→
2. Default Agentsends to Publish server
→
3. Page on Publishupdated ✓
→
4. Flush Agentsends DELETE to Dispatcher
→
5. Dispatcher deletescached HTML file
→
6. Next visitorgets fresh content ✓
If the Flush Agent is missing or misconfigured: Step 4-5 never happen. The Dispatcher keeps serving the old cached HTML. Visitors see stale content for hours until the cache expires naturally. This is one of the most common "why is my published content not showing up?" production problems.
Flush Agent configuration — what makes it different
Flush Agent key settings
Transport URI = http://dispatcher-host/dispatcher/invalidate.cache
↑ NOT the publish server — the Dispatcher's invalidation endpoint
Serialization = Dispatcher Flush ← special type just for cache invalidation
HTTP Method = GET ← uses GET with special headers
// Special headers added automatically by AEM:
CQ-Action = Activate ← what happened (publish/delete)
CQ-Handle = /content/bank/homepage← which page was published
CQ-Path = /content/bank/homepage← Dispatcher uses this to find and delete cached file
The Dispatcher reads the CQ-Handle header to know which cached file to delete. It looks for the corresponding .html file in its cache directory and removes it. This is called "cache invalidation."

Multi-Site Management (MSM)
MSM is AEM's system for managing many websites from a single master source. Changes to the master automatically push to all child sites. Each child site can keep some things unique (local language, local prices) while inheriting everything else.
Real life analogy
🍔
A global fast food chain like McDonald's.

Every McDonald's in the world uses the same: golden arches logo, same packaging design, same menu layout, same ordering system, same kitchen equipment configuration, same cleaning protocols.

But each country has some local items: McAloo Tikki in India, McSpicy Paneer for vegetarians, Maharaja Mac instead of Big Mac. And prices are in local currency.

The global headquarters (Blueprint) defines everything shared. Each country restaurant (Live Copy) inherits from headquarters but can override local things.

When headquarters changes the logo, ALL 40,000 restaurants get the new logo automatically — via a rollout. But the McAloo Tikki in India stays because it's a local override.
The MSM vocabulary — 6 terms you must know
Blueprint
The master/source site. This is where your global team creates and manages content. Think of it as headquarters. For the clothing brand, the English master site at /content/brand/en is the Blueprint. Everything created here can be pushed to all country sites.
Live Copy
A child site that is connected to the Blueprint. The India site at /content/brand/in/en is a Live Copy of the Blueprint. It receives updates from the Blueprint through rollouts. Some of its content is locked (inherits from Blueprint) and some is local (overridden).
Rollout
The action of pushing Blueprint changes to Live Copies. When the global team updates the navigation in the Blueprint, they click "Rollout" — AEM pushes the changes to all Live Copies. Like a software update being pushed to all franchise locations.
Inheritance
A Live Copy property that is "locked" — it receives Blueprint changes automatically. The navigation component is inherited — when Blueprint navigation changes, all Live Copies get the update on next rollout. An inherited property has a lock icon in the page editor.
Cancelled Inheritance
A Live Copy property that is "unlocked" — local editors have overridden it. The India site's hero banner is locally edited — it shows the India-specific campaign image. Even when the Blueprint hero changes, India's banner stays as-is because inheritance is cancelled for that property.
Language Copy
A copy of content translated into another language. Not exactly MSM Live Copy — more of a translation workflow. The English master is translated into Hindi, Tamil, German etc. AEM's Translation Integration Framework connects to translation services (Google Translate, manual translators) and manages the translated copies.
The clothing brand's MSM structure
/content/brand/
├── en/ ← BLUEPRINT (master — global team works here)
│ ├── homepage
│ ├── products
│ └── about-us
│
├── in/en/ ← LIVE COPY of Blueprint (India site)
│ ├── homepage ← inherits from Blueprint EXCEPT hero banner (local)
│ ├── products ← inherits from Blueprint EXCEPT prices (local INR)
│ └── about-us ← fully inherited — same content as Blueprint
│
├── us/en/ ← LIVE COPY of Blueprint (USA site)
│
└── de/en/ ← LIVE COPY of Blueprint (Germany site)

Live Copy and Blueprint — deep dive
Now let's go deeper into exactly how a Blueprint and Live Copy are set up, what the relationship looks like in the JCR, and how inheritance works at the property level.
How to create a Blueprint — step by step (UI)
1
Create the master site
Build your master site normally at /content/brand/en. This is just a regular AEM site at this point — nothing special yet.
2
Create a Blueprint Configuration
Go to Tools → Sites → Blueprints. Click Create. Point it to /content/brand/en. Give it a name: "Brand Master". Now AEM knows this path is a Blueprint — a source that other sites can be Live Copies of.
3
Create a Live Copy from the Blueprint
Select the Blueprint site in Sites console → Click "Create" → "Live Copy". Choose the destination path (/content/brand/in). Choose which Rollout Configuration to use. AEM copies the entire Blueprint structure to the new path and creates live relationship metadata.
4
The Live Copy relationship is stored in the JCR
AEM stores the relationship as JCR properties on the Live Copy's jcr:content node. These properties tell AEM: "this page is connected to this Blueprint path, using this rollout config, with these properties cancelled from inheritance."
What the Live Copy relationship looks like in JCR
/content/brand/in/en/homepage/jcr:content
↑ the Live Copy page's content node
cq:isDeepCopy = true
cq:master = "/content/brand/en/homepage"
↑ which Blueprint page this is connected to
cq:rolloutConfigs = ["/libs/msm/wcm/rolloutconfigs/default"]
↑ which rollout config controls what syncs
cq:propertyInheritanceCancelled = ["heroImage", "heroBannerText"]
↑ these two properties are LOCAL — blueprint changes won't override them
jcr:title = "Homepage" ← inherited from Blueprint
navigationTitle = "Home" ← inherited from Blueprint
heroImage = "/content/dam/brand/in/hero.jpg" ← LOCAL (cancelled)
heroBannerText = "India's favourite fashion brand" ← LOCAL
The cq:propertyInheritanceCancelled array is the key. Any property name listed here is treated as local — Blueprint rollouts will not overwrite it. Everything else is inherited and overwritten on each rollout.
What editors see when editing a Live Copy page
🔒
When a local editor in India opens the homepage in the Page Editor:

Inherited properties (like Navigation Title, Page Title, SEO metadata) have a lock icon 🔒 next to them. The field is greyed out — the editor cannot change it. It will be updated by the next Blueprint rollout.

Cancelled-inheritance properties (like Hero Image, Hero Text) have an unlock icon 🔓. The editor can freely edit these. Blueprint rollouts will not touch them.

An editor can also manually cancel inheritance on any field by clicking the lock icon and confirming. This immediately makes that field local and editable.
Language Copy — translation workflow
A Language Copy is created from the master language version (English) and translated into other languages. The site structure is usually:
/content/brand/
├── en/ ← master (English) — all content created here first
├── hi/ ← Hindi translation of en/
├── ta/ ← Tamil translation of en/
└── de/ ← German translation of en/
When English content changes, AEM's Translation Integration Framework (TIF) detects the change, creates a Translation Job, sends the new text to a translation service (human translators or machine translation like Google Translate), receives the translated text back, and updates the language copies automatically.

Rollout Configuration — controlling what syncs
A Rollout Configuration is a set of rules that controls: WHEN does a rollout happen (what triggers it) and WHAT gets synced (which properties, which components). You can have different rollout configs for different scenarios.
Real life analogy
📦
Think of a software update policy for a company's computers.

"Critical security patches: install immediately on all computers."
"New features: install only on dev team computers first."
"Design theme updates: install on all computers but preserve user's custom wallpaper."

Each rule says: what gets updated, when, and what is preserved locally. That's a Rollout Configuration — it defines the update policy between Blueprint and Live Copies.
Rollout trigger types — when does rollout happen?
1
On Modification (manual)
Rollout only happens when someone explicitly clicks the "Rollout" button in the Blueprint's page properties or the Sites console. Good for: design changes, navigation updates where you want to control exactly when the push happens.
2
On Activation (publish)
Rollout automatically triggers when a Blueprint page is published (Activated). Good for: content that should immediately go to all country sites when the master content is approved and published.
3
On Deactivation / On Deletion
Rollout triggers when a Blueprint page is unpublished or deleted. Ensures that if the global team deletes a product from the master, all country sites also remove it — no orphaned content left on Live Copies.
Rollout actions (sync handlers) — what gets synced
Within a rollout config, you choose which "sync handlers" (actions) run. Each handler controls a specific type of content synchronisation:
A
contentUpdate — syncs content and properties
The main one. Copies all non-cancelled properties from Blueprint pages to Live Copy pages. This is what updates the navigation title, SEO metadata, component configurations etc.
B
contentCopy — creates missing pages
If the Blueprint has a new page that doesn't exist in the Live Copy yet, this handler creates it. Without this, new Blueprint pages would never appear in Live Copies.
C
referencesUpdate — fixes internal links
When a Blueprint page has an internal link pointing to another Blueprint page, this handler automatically updates the link in the Live Copy to point to the Live Copy equivalent. Without this, all internal links in Live Copies would point back to the Blueprint.
D
orderChildren — maintains page order
Ensures the page order in Live Copies matches the Blueprint. If Blueprint reorders its pages (e.g. moves "About Us" before "Products" in the navigation), this handler applies the same order to all Live Copies.
Triggering rollout from Java code
Programmatic rollout — every line explained
1import com.day.cq.wcm.msm.api.RolloutManager;
2import com.day.cq.wcm.msm.api.RolloutManager.RolloutParams;
3
4@Reference
5private RolloutManager rolloutManager;
6
7public void rolloutBlueprintPage(ResourceResolver resolver) {
8

10 Resource blueprintPage = resolver.getResource(
11 "/content/brand/en/homepage");
12

14 RolloutParams params = new RolloutParams();
15 params.setMaster(blueprintPage);
16 params.setDeep(true);
17 params.setReset(false);
18

21

23}
L4-5
@Reference RolloutManager rolloutManager — AEM's MSM rollout engine. OSGi injects it automatically. It handles finding all Live Copies of a Blueprint, applying the rollout configuration rules, and syncing each Live Copy one by one.
L14-17
RolloutParams — a configuration object for this specific rollout.

setMaster(blueprintPage) — which Blueprint page to roll out FROM. AEM finds all Live Copies of this specific page and syncs them.

setDeep(true) — rollout this page AND all its children recursively. For a homepage this would include all sub-pages. false = only this one page, not children.

setReset(false) — if false: merge changes (preserve local overrides). If true: full reset — overwrite everything in the Live Copy including locally-edited content. Very destructive — rarely used unless you explicitly want to wipe local customisations.
L20
rolloutManager.rollout(params) — triggers the rollout synchronously. For a deep rollout of a large site with 40 Live Copies and 1,000 pages, this can take several minutes. In production, this is typically done asynchronously via a background job — triggered by a workflow or scheduled task — not in a synchronous request handler where it would timeout.
Common MSM mistakes
!
Accidentally cancelling inheritance on the wrong property
An editor in India accidentally clicks the lock icon on the "SEO Title" field — now all future global SEO updates will not apply to India. This is a silent problem — no error, just the field stops syncing. Always educate editors about what the lock/unlock icons mean.
!
Running deep rollout on a large site synchronously
Rolling out 40 Live Copies × 500 pages = 20,000 page updates. If done synchronously in a request thread, it times out (30+ minutes). Always trigger large rollouts as a background workflow or scheduled job.
!
Not configuring referencesUpdate in the rollout config
Without referencesUpdate, all internal links in Live Copies point to Blueprint paths. A navigation link on the India site says "go to /content/brand/en/products" instead of "/content/brand/in/en/products" — taking Indian visitors to the English master site.
