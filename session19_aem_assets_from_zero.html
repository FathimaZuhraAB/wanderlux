What is the DAM?
DAM stands for Digital Asset Management. It is the part of AEM where all files — images, videos, PDFs, documents — are stored, organised, and managed. Think of it as a very powerful, searchable, version-controlled shared drive built into AEM.
Real life analogy
📸
Think of a professional photo agency.

They receive thousands of photos every week from photographers all over the world. Each photo needs to be:

— Filed in the right folder (by date, topic, photographer)
— Given metadata: title, description, copyright, photographer name
— Resized into multiple formats: web size, print size, thumbnail
— Made searchable so editors can find "cricket match, India, 2024"
— Distributed to the right publications on request

AEM's DAM is that photo agency system, built into your website.
Where assets live in AEM
All assets are stored in the JCR (AEM's database) under a special path called /content/dam. Think of this as the root folder of the shared drive.
/content/dam/ ← root of the DAM
├── clothing-brand/ ← your project folder
│ ├── summer-2024/
│ │ ├── shoe-001-front.jpg ← asset node
│ │ ├── shoe-001-back.jpg
│ │ └── size-guide.pdf
│ └── winter-2024/
│ └── jacket-hero.jpg
└── shared/
└── brand-logo.svg
Just like a shared network drive, you can create folders and subfolders. Editors navigate this structure in AEM's Assets console (at /assets.html/content/dam) — it looks just like Windows Explorer or Mac Finder but inside the browser.
Types of assets AEM supports
AEM can handle many file types. Each is processed differently when uploaded:
Images — JPEG, PNG, GIF, WebP, TIFF, SVG
Most common. AEM auto-generates thumbnails and multiple sizes. EXIF metadata (camera, location, date) is auto-extracted.
Videos — MP4, MOV, AVI
AEM generates video thumbnails and can transcode to different formats. Duration, resolution auto-extracted.
Documents — PDF, DOCX, XLSX, PPTX
AEM generates page previews so you can see page thumbnails without downloading. Full text is indexed for search.
Design Files — Adobe InDesign (.indd), Illustrator (.ai), Photoshop (.psd)
AEM integrates with Adobe tools. Designers can check assets in/out and publish directly from Creative Cloud.
3D Models — GLB, OBJ, FBX
Used in e-commerce for 360° product views. AEM generates preview renders.

Asset structure — what's inside a single asset?
When you upload a single photo to AEM, it doesn't just save the file. It creates an entire node structure in the JCR database around that file. Understanding this structure is essential because your Java code reads from it.
Real life analogy
🗃️
Think of a physical filing cabinet in a professional studio.

For each photo, there is a folder (the asset node). Inside that folder:
— The original photo (the original binary file)
— A contact sheet of smaller versions (thumbnails and web renditions)
— A label on the front of the folder with: photographer, date, copyright, subject (metadata)

AEM creates all of this automatically every time you upload one image.
The exact JCR node structure of one asset
When you upload shoe-001.jpg, this is what AEM creates in the database:
/content/dam/clothing-brand/shoe-001.jpg [dam:Asset]
↑ The asset node itself. Type dam:Asset tells AEM this is a file.
└── jcr:content [dam:AssetContent]
↑ All data about the asset lives inside here
├── metadata [nt:unstructured]
│ ├── dc:title = "Shoe 001 Front View"
│ ├── dc:format = "image/jpeg"
│ ├── dam:fileSize = 245760
│ ├── dam:Physicalwidth = 1920
│ ├── dam:Physicalheight = 1080
│ ├── dam:season = "Summer 2024" ← custom
│ └── dam:sku = "SHOE-001" ← custom
└── renditions [nt:folder]
├── original ← the actual uploaded file
├── cq5dam.thumbnail.48.48.png ← tiny thumbnail
├── cq5dam.thumbnail.140.100.png ← DAM UI thumbnail
└── cq5dam.web.1280.1280.jpeg ← web-optimised version
Breaking down each part
A
The asset node — dam:Asset
The top-level node representing the file. Its path is the "address" of the asset in AEM: /content/dam/clothing-brand/shoe-001.jpg. The node type dam:Asset is what tells AEM "this is a file, not just a regular content node." Every asset in the DAM has this type.
B
jcr:content — the container
A child node of the asset that holds ALL data about the asset. Just like pages have a jcr:content node for their properties, assets do too. Type dam:AssetContent.
C
metadata — the label on the folder
A child of jcr:content that stores all metadata as properties. Standard properties use Dublin Core prefixes (dc:title, dc:description) and DAM-specific prefixes (dam:fileSize, dam:Physicalwidth). You add custom properties here too — like dam:season or dam:sku.
D
renditions — the different sizes
A child of jcr:content that stores all versions of the file. original is always the file exactly as uploaded — never modified. Other renditions are generated versions: thumbnails for the AEM UI, web-optimised versions for website pages. Each rendition is a binary node containing the actual file bytes.
What are renditions and why do they exist?
🖼️
A photographer uploads a 24-megapixel RAW photo — 45MB file, 6000×4000 pixels.

Problem: You can't use this 45MB file on a product page — it would load extremely slowly for website visitors. But you also don't want to throw away the original high-quality file.

Solution — renditions: AEM keeps the original intact and auto-generates:
— A 140×100px thumbnail (3KB) for the DAM grid view
— A 1280×853px web version (180KB) for product pages
— A 400×400px square crop (40KB) for category listings

Your website components use the right-sized rendition. The original is preserved for print or future use.

Asset processing workflow
When you drag a photo into AEM's DAM, something powerful happens automatically: a workflow runs behind the scenes and does a series of processing steps on the file — all without you doing anything.
Real life analogy
🏭
Think of a factory assembly line.

Raw material (your uploaded photo) enters one end. It passes through automated stations: quality check, cutting, shaping, coating, packaging, labelling. Out the other end comes a finished, ready-to-use product — multiple sizes, metadata filled in, indexed for search.

The "DAM Update Asset" workflow is that assembly line. It runs automatically every time a file is uploaded or updated.
The DAM Update Asset workflow — step by step
This is the default workflow AEM runs on every uploaded asset. You can see it at Tools → Workflow → Models → DAM Update Asset.
Uploadphoto added
Extract metadataEXIF, IPTC, XMP
Generate thumbnails48px, 140px, 319px
Web rendition1280px JPEG
Index for searchfull text, tags
Doneasset ready
Each step explained in plain English
1
Metadata extraction
AEM reads all embedded metadata from the file itself. For a JPEG, this means reading EXIF data — camera make/model, focal length, aperture, GPS coordinates, timestamp. This information is automatically stored in the asset's metadata node. No manual entry needed — the data was already in the file from the camera.
2
Thumbnail generation
AEM generates multiple small versions of the image and stores them as renditions. The 48×48 thumbnail is used for search results in the DAM UI. The 140×100 thumbnail is used in the card view. The 319×319 is used for larger previews. These are created by resizing and cropping the original image automatically.
3
Web rendition creation
AEM creates an optimised version suitable for use on web pages — typically a JPEG at 1280px wide, compressed for web delivery. This is what your website's product component references. When a visitor loads the page, their browser downloads this rendition, not the original 45MB file.
4
Full-text indexing
AEM extracts all text from the asset (title, description, metadata, and for PDFs the actual text content) and adds it to the search index. This is why when you search "summer collection" in AEM's asset search, you find images that have "summer collection" in their title or metadata — even though the image itself contains no searchable text.
5
Smart tagging (if configured)
If Adobe Sensei (AEM's AI) is enabled, it automatically analyses the image content and applies descriptive tags. Upload a photo of a red shoe and it auto-applies tags like "footwear", "red", "shoe", "leather", "fashion". No human needed to label the image.
How to add a custom step to the workflow
You can add your own processing step to this workflow. For example, the clothing brand needs to auto-apply a "season" tag based on the folder the image was uploaded to.
Custom workflow process step — auto-tag by folder
1// Register as a WorkflowProcess — AEM shows this in workflow model editor
2@Component(service = WorkflowProcess.class,
3 property = {"process.label=Auto Tag By Folder"})
4public class AutoTagByFolderProcess implements WorkflowProcess {
5
6 @Override
7 public void execute(WorkItem workItem,
8 WorkflowSession wfSession,
9 MetaDataMap args) {
10
11 // Get the path of the asset being processed
12 String assetPath = workItem.getWorkflowData()
13 .getPayload().toString();
14 // e.g. "/content/dam/clothing-brand/summer-2024/shoe-001.jpg"
15
16 // Extract folder name — "summer-2024" from the path
17 String[] parts = assetPath.split("/");
18 String folderName = parts[parts.length - 2];
19 // folderName = "summer-2024"
20

28}
L2-3
@Component(service=WorkflowProcess.class, property={"process.label=..."}) — by registering as a WorkflowProcess OSGi service with a process.label property, this class appears as a selectable step in the Workflow Model editor UI. A developer can drag it into the DAM Update Asset workflow diagram without writing any XML config.
L7-9
execute(WorkItem, WorkflowSession, MetaDataMap) — AEM calls this method when this step runs in a workflow. Three parameters: WorkItem = the current task being processed, contains the payload (what asset is being processed). WorkflowSession = the session for this workflow execution, lets you access the JCR. MetaDataMap args = any configuration arguments you set in the workflow step dialog (like a custom folder-to-season mapping).
L12-13
workItem.getWorkflowData().getPayload().toString() — a chain of calls to dig into the work item and get what it's processing. The "payload" in a DAM workflow is always the path of the asset being processed. This gives you a String like "/content/dam/clothing-brand/summer-2024/shoe-001.jpg".
L17-18
assetPath.split("/") — splits the path string by the "/" character into an array of parts. For "/content/dam/clothing-brand/summer-2024/shoe-001.jpg", the array is: ["", "content", "dam", "clothing-brand", "summer-2024", "shoe-001.jpg"]. parts.length - 2 gets the second-to-last part — the folder name "summer-2024".
L22-26
session.getNode(...) and setProperty() — gets the metadata node of the asset directly via the JCR API (lower-level than ResourceResolver). metaNode.setProperty("dam:season", folderName) writes the folder name as a custom metadata property. session.save() commits this to the database — equivalent to resolver.commit() but at the lower JCR session level.

Assets API — Java code for working with assets
The Assets API lets your Java code work with the DAM programmatically: upload new assets, read asset metadata, get specific renditions, update metadata, and delete assets. The main class is AssetManager.
What is AssetManager?
🧑‍💼
Think of AssetManager as the DAM librarian.

You say: "Store this photo file at this path." The librarian files it, triggers processing, and confirms it's done.

You say: "Give me the asset at this path." The librarian retrieves it with all its metadata and renditions.

You say: "Delete this asset." The librarian removes it and all its generated versions.
You get an AssetManager by calling resourceResolver.adaptTo(AssetManager.class) — same pattern as TagManager from Session 18.
Operation 1: Uploading an asset from a file or stream
Real use case: an integration job pulls product photos from a PIM (Product Information Management) system every night and stores them in AEM's DAM.
Uploading an asset programmatically
1import com.day.cq.dam.api.Asset;
2import com.day.cq.dam.api.AssetManager;
3import java.io.InputStream;
4import java.io.FileInputStream;
5
6// Get the AssetManager from your ResourceResolver
7AssetManager assetManager =
8 resolver.adaptTo(AssetManager.class);
9
10// Open the file as a stream — InputStream reads bytes one at a time
11try (InputStream is = new FileInputStream("/tmp/shoe-001.jpg")) {
12
13 // Upload the file to DAM — createAsset does everything
14 Asset asset = assetManager.createAsset(
15 "/content/dam/clothing-brand/summer-2024/shoe-001.jpg",
16 is,
17 "image/jpeg",
18 true
19 );
20
21 log.info("Asset created: {}", asset.getPath());
22 // Output: "Asset created: /content/dam/clothing-brand/summer-2024/shoe-001.jpg"
23
24} catch (Exception e) {

L7-8
resolver.adaptTo(AssetManager.class) — same pattern as TagManager. The resolver gives you an AssetManager that operates within your service user's permissions. If your service user doesn't have jcr:write on /content/dam, this will fail when you try to upload — so make sure the service user's repoinit grants write access to the DAM folder your code needs.
L11
try (InputStream is = new FileInputStream("/tmp/shoe-001.jpg")) — an InputStream is a Java way to read a file as a stream of bytes, without loading the entire file into memory at once. For a 45MB photo, loading it fully into memory could cause OutOfMemoryError if you're processing hundreds of photos in a loop. The try-with-resources ensures the file is closed after use.
L14
assetManager.createAsset(...) — this single method call does everything: creates the dam:Asset node at the given path, stores the file as the "original" rendition, and triggers the DAM Update Asset workflow (which generates thumbnails, extracts metadata etc.). It returns the created Asset object so you can immediately read its properties or add metadata.
L15
The destination path — where to store the asset in the DAM. If the parent folder doesn't exist, AEM creates it. The filename at the end (shoe-001.jpg) becomes the asset's node name. If an asset already exists at this path, it is replaced (versioned if versioning is enabled).
L17
"image/jpeg" — the MIME type. This tells AEM what kind of file this is so it knows how to process it. Common types: "image/jpeg", "image/png", "image/svg+xml", "video/mp4", "application/pdf". AEM uses this to choose which workflow steps to run (PDF processing is different from image processing).
L18
true — the fourth parameter is doSave. When true, AEM immediately commits the asset to the JCR database after creating it. When false, it waits — you'd need to call resolver.commit() manually afterwards. In most cases pass true for immediate persistence. Use false only when you need to create many assets and want to batch-commit them all at once for better performance.
Operation 2: Reading an asset and its metadata
Real use case: a product page component reads the product photo's metadata (SKU, season, alt text) to render the page correctly.
Reading asset metadata
1// Get the asset from its path
2Resource assetResource = resolver.getResource(
3 "/content/dam/clothing-brand/shoe-001.jpg");
4
5// Adapt the resource to an Asset object to use the Asset API
6Asset asset = assetResource.adaptTo(Asset.class);
7
8if (asset != null) {
9 // Read metadata via the metadata child node's ValueMap
10 ValueMap meta = assetResource
11 .getChild("jcr:content/metadata")
12 .getValueMap();
13
14 String title = meta.get("dc:title", "");
15 String mimeType = meta.get("dc:format", "");
16 long fileSize = meta.get("dam:fileSize", 0L);
17 String season = meta.get("dam:season", "");
18 String sku = meta.get("dam:sku", "");
19
20 log.info("Title={} SKU={} Season={}", title, sku, season);
21}
L2-3
resolver.getResource(assetPath) — gets the asset's root node as a Resource. This is the dam:Asset node. From here you can navigate to child nodes using getChild("jcr:content") or getChild("jcr:content/metadata").
L6
assetResource.adaptTo(Asset.class) — converts the generic Resource into a typed Asset object that has convenient methods like getRendition(), getPath(), and getMimeType(). Returns null if the resource is not actually a dam:Asset node.
L10-12
assetResource.getChild("jcr:content/metadata").getValueMap() — navigates from the asset node down into the metadata child node. getChild() accepts a path relative to the current resource — "jcr:content/metadata" means "go to my child jcr:content, then to its child metadata." getValueMap() returns all properties of that node as a key-value Map.
L14-18
meta.get("property:name", defaultValue) — reads a property from the metadata map. The second argument is the default value returned if the property doesn't exist. dc:title, dc:format are Dublin Core standard properties auto-populated by AEM. dam:fileSize is AEM's own property. dam:season, dam:sku are your custom properties (defined in the metadata schema, covered in the next tab).
Operation 3: Getting a specific rendition
Real use case: your product component wants to use the 1280px web rendition for the main image, not the original 45MB file.
Getting a rendition from an asset
1Asset asset = assetResource.adaptTo(Asset.class);
2
3// Get the web rendition by its exact name
4Rendition webRendition =
5 asset.getRendition("cq5dam.web.1280.1280.jpeg");
6
7// If web rendition doesn't exist, fall back to original
8if (webRendition == null) {
9 webRendition = asset.getOriginal();
10}
11
12// Get the URL path for use in an HTML img tag
13String renditionPath = webRendition.getPath();
14// = "/content/dam/clothing-brand/shoe-001.jpg/jcr:content/renditions/cq5dam.web.1280.1280.jpeg"
15
16// Read the actual bytes as a stream (for processing or sending)
17InputStream stream = webRendition.getStream();
L4-5
asset.getRendition("cq5dam.web.1280.1280.jpeg") — gets a specific rendition by its exact name. The naming convention for AEM-generated renditions is: cq5dam.[type].[width].[height].[format]. "cq5dam" is historical AEM naming. "web" = the rendition type. "1280.1280" = max width and height. "jpeg" = format. Returns null if that rendition hasn't been generated yet (e.g. the workflow hasn't finished yet).
L8-10
Null check and fallback to original — always handle the case where a rendition doesn't exist. This happens when: the workflow hasn't finished yet (race condition), the workflow failed, or the file type doesn't support that rendition (you can't make a 1280px rendition of a 200px image). Falling back to asset.getOriginal() ensures your component always has something to render.
L13
webRendition.getPath() — returns the JCR path of the rendition node. In an HTL template, you'd use this path as the src of an img tag: <img src="${renditionPath}">. AEM's servlet automatically streams the binary file when a browser requests this path.
L17
webRendition.getStream() — returns the actual file bytes as an InputStream. Use this when you need to actually process the image bytes in Java — for example, sending it to an external CDN, running image analysis, or converting the format. Always close this stream after use to avoid resource leaks.
Operation 4: Updating metadata on an existing asset
Real use case: your nightly PIM sync updates product SKUs and season labels on 500 existing assets.
Updating asset metadata
1// Get the metadata node as a ModifiableValueMap
2Resource metaResource = resolver.getResource(
3 "/content/dam/clothing-brand/shoe-001.jpg/jcr:content/metadata");
4
5ModifiableValueMap mvm =
6 metaResource.adaptTo(ModifiableValueMap.class);
7
8// Update properties — just like writing to a Map
9mvm.put("dc:title", "Shoe 001 — Redesigned for Summer 2024");
10mvm.put("dam:season", "Summer 2024");
11mvm.put("dam:sku", "SHOE-001-V2");
12
13// Save — without this, changes are lost when session ends
14resolver.commit();
15log.info("Metadata updated for shoe-001");
L5-6
metaResource.adaptTo(ModifiableValueMap.class) — a ValueMap is read-only. A ModifiableValueMap is the writable version — it lets you use .put() to set or update properties. It directly modifies the underlying JCR node's properties. Think of ValueMap as reading a form, and ModifiableValueMap as filling in and editing the form.
L9-11
mvm.put(propertyName, value) — sets a property. If the property already exists, it is overwritten. If it doesn't exist yet, it is created. You can update standard AEM properties (dc:title) and custom ones (dam:season, dam:sku) the same way. The property name must match exactly what is defined in the metadata schema (covered in the next tab).
L14
resolver.commit() — same as always: writes in-memory changes to the actual JCR database. Without this, all your mvm.put() calls are lost when the resolver closes. In a batch update of 500 assets, you might call commit() every 50 assets to avoid keeping too many uncommitted changes in memory at once.

Custom metadata schema
By default, every asset has standard metadata fields: title, description, tags, copyright. But your clothing brand needs extra fields: Season, SKU, Collection Name, Photographer. A custom metadata schema lets you define and display your own fields in AEM's asset properties panel.
Real life analogy
📝
Think of a form you fill in when you admit a patient to a hospital. The default fields are: name, age, blood type, allergies.

But a specialist cancer hospital also needs: cancer type, stage, previous treatment, clinical trial number. They add these custom fields to their admission form.

A custom metadata schema is exactly that — adding your own fields to the AEM asset properties form. Editors fill them in. Your code reads them.

Default — only standard fields

Title
Description
Tags
Copyright
... that's it. No way to record which season, which product SKU, who photographed it.

With custom schema — your fields added

Title
Description
Tags
Copyright
Season (dropdown)
Collection Name (text)
Product SKU (text)
Photographer (text)

The custom fields appear in the asset properties panel in AEM's UI. When an editor opens any image and clicks "Properties", they see your custom fields ready to fill in.
How to create a custom metadata schema — step by step
1
Open the Metadata Schema Editor
Go to Tools → Assets → Metadata Schemas. You'll see a list of existing schemas. The "default" schema is the one applied to all assets unless overridden. Click "Create" to make a new schema for your specific asset type (e.g. "Product Photos"), or edit the default to add fields to everything.
2
Add your custom fields by dragging components
The schema editor shows a form builder. Left panel has field types: Single Line Text, Multi Line Text, Number, Date, Dropdown, Tags, Checkbox. Drag them onto the form. Each field has two key settings: the Label (what the editor sees) and the Map to Property (the JCR property name where the value is stored).
3
Set the "Map to property" for each field
This is the most important setting. For the "Season" field, you'd set Map to property = ./jcr:content/metadata/dam:season. The ./ means "relative to the asset node". This is the exact JCR property path where the value gets stored — and where your Java code reads it from using meta.get("dam:season", "").
4
Apply the schema to a folder
Right-click any DAM folder → Properties → Metadata Schema → choose your schema. All assets in that folder and its subfolders now use your custom schema. Editors opening any of those assets' properties will see your custom fields.
Field types you can add
Single Line Text
For short values like Product SKU ("SHOE-001"), Photographer name, Collection name. Stores as a String property.
Dropdown (single select)
For fixed options like Season (Spring, Summer, Autumn, Winter) or Asset Type (Hero, Detail, Lifestyle). Stores the selected option as a String.
Date
For shoot date, expiry date, publication date. Stores as a JCR Date value. Java reads it as a Calendar object.
Tags field
Links to AEM's tagging system. Stores selected tags as tag IDs in cq:tags. Connects the metadata schema to the tagging framework from Session 18.
Checkbox (boolean)
For yes/no fields like "Approved for web use", "Contains model release". Stores as a Boolean property.
Reading custom metadata in a Java component — full example
Your product page Sling Model reads the custom metadata from the product photo and uses it to render alt text, SKU, and season badge.

L4-7
@ValueMapValue String fileReference — @ValueMapValue injects a property from the current resource's (the component's) ValueMap. fileReference is the standard AEM property name for an asset path chosen via an image/asset picker in a component dialog. The author picks a product photo in the page editor, and that photo's DAM path is stored as fileReference on the component node.
L12-22
getSeason(), getSku(), getAltText() — each public getter method in a Sling Model becomes accessible in the HTL template. The HTL template calls ${model.season}, ${model.sku}, ${model.altText} — Sling automatically calls the corresponding getter method and returns the value.
L24-30
private helper method getMetaValue(propertyName) — instead of repeating the same resolver.getResource + getValueMap logic three times, it's extracted into a private helper method. The method takes the property name as a parameter and returns its value. Line 25: guard against null fileReference (no asset selected by author). Line 28: guard against null metadata node. Line 29: read and return the property with "" as default (never return null to HTL — it renders as "null" string).

${model.altText @ context='attribute'} — the alt text from the asset's dc:title metadata property is used as the HTML alt attribute for accessibility. Search engines also read this. Without proper alt text on product images, the page fails WCAG accessibility standards.
L6-7
${model.season} and ${model.sku} — your custom metadata values rendered directly into the HTML. The season badge could be styled with CSS to show "Summer 2024" in a coloured label. The SKU is shown for users who know their product number. All this data came automatically from the asset's custom metadata — no hardcoding in page content.
