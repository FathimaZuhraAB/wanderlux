Session 1

What is a CMS and WCM?

Before AEM, understand what problem it solves. Every website needs content — text, images, pages. Without a CMS, a developer has to manually edit HTML files every single time something changes.

The problem CMS solves

Imagine a bank's website. They have 500 pages — loan rates, branch hours, FAQs. Every week something changes. Without a CMS, a developer has to edit HTML files, redeploy, and push to production for every single change. This takes days and costs money.

A CMS (Content Management System) lets non-technical people — marketing managers, content editors — log into a website, click on text, change it, and publish it themselves. No developer needed.

Real example: HDFC Bank's website has thousands of pages. Their marketing team updates interest rates every week, adds new offers every day, and publishes new articles every hour — all without a single developer writing code. They use a CMS to do this.

CMS vs WCM — what's the difference?

CMS — Content Management System
General term for any system that manages content. WordPress, Drupal, Joomla are all CMS platforms. Suitable for blogs, small websites, simple use cases.
WCM — Web Content Management
An advanced CMS built for enterprise. Handles multiple sites, multiple languages, workflows, approvals, personalization, digital assets — all at scale. AEM is a WCM.

Three things every CMS has

Content repository
Where everything is stored — pages, images, videos, configs. In AEM this is the JCR (Java Content Repository).
Authoring interface
The UI editors use to create and edit content. Can be a form, a rich text editor, or a drag-and-drop canvas.
Delivery layer
Converts stored content into HTML pages that end users see in their browser.
Session 1

What is AEM — Adobe Experience Manager

AEM is Adobe's enterprise CMS. It is used by large companies — banks, airlines, retailers — to manage massive websites across multiple countries, languages, and brands from one platform.

What AEM is built on

AEM is not built from scratch. It is assembled from several open-source Apache projects layered on top of each other:

The technology stack inside AEM
Apache Felix (OSGi) — the Java runtime. Manages Java bundles (plugins) that can be installed and updated without restarting the server.

Apache Sling — the web framework. Maps URLs to JCR content nodes and finds the right script to render them.

Apache Jackrabbit Oak — the content repository. Stores everything as a tree of nodes and properties (like a file system but with typed data).

Adobe Granite — Adobe's layer on top. Adds the authoring UI, workflows, DAM, and all the enterprise features.

What AEM can do

AEM Sites
Create and manage web pages. Drag-and-drop components, edit content inline, manage templates, run A/B tests.
AEM Assets (DAM)
Manage all digital files — images, videos, PDFs. Auto-generate image renditions, add metadata, search assets.
AEM Forms
Build dynamic digital forms with workflows and e-signatures. Used by banks and governments.
Headless CMS
Deliver content as JSON to React apps, mobile apps, or any channel via GraphQL and REST APIs.
Real example: Marriott hotels uses AEM to manage their website in 50 languages across 200 countries. One team in corporate HQ manages the brand standards and global templates. Local teams in each country manage their own pages and offers — all from the same AEM instance.

WYSIWYG — What You See Is What You Get

WYSIWYG means the editing interface looks exactly like the final published page. You click on a title directly on the page, type the new text, and the page immediately shows what it will look like when live.

How it works in AEM

When an editor opens a page in AEM's editor, they see the actual rendered page — with all the CSS, images, and layout applied. They can then click on any component (text block, image, banner) and a blue toolbar appears. Clicking "Edit" activates the editor directly on the component.

Open page in editor
→
Click on a component
→
Blue toolbar appears
→
Click Edit / Configure
→
Edit content directly on page

Two types of editing in AEM

Inline editing (in-place)
Double-click on a text component and a rich text editor activates directly on the page. You type changes and they appear immediately where the text is. Used for simple text components.
Dialog editing (configure)
Click the wrench icon and a popup dialog opens with form fields. Used for complex components with many settings — image paths, link URLs, dropdown selections, checkboxes.
Real example: A content editor at Zomato needs to update the hero banner text for Diwali. She opens the homepage in AEM editor, double-clicks the headline text, types the new text, clicks save. She sees exactly what customers will see — the Diwali text in the correct font, size, and color. No preview button needed — it's already showing the real result.

Author mode vs Preview mode vs Publish

Edit mode
Shows the page with blue authoring overlays on components. Editors can drag, drop, configure. This is the working mode.
Preview mode
Shows the page exactly as visitors will see it — no overlays, no edit toolbars. Used to check how the page looks before publishing.
Published page
The real public page on the Publish server. Only activated content appears here. Visitors never see the author environment.

Author, Publish, Dispatcher — the 3-layer setup

AEM always runs as three separate layers. This is not optional — it is the core architecture. Understanding this is essential before doing anything else in AEM.

The three layers

Author → Publish → Dispatcher
Author (port 4502)
Editors work here
→ activate →
Publish (port 4503)
Content delivered here
→ cache →
Dispatcher (port 80)
Users see this

Author layer — where you work as a developer

The Author server runs at http://localhost:4502. Default login is admin / admin on local. This is where:

  • Editors create and edit pages
  • Developers deploy code using Maven
  • Templates are created and configured
  • Assets are uploaded to DAM
  • Workflows run for approvals

Author is NEVER accessible to the public. It is behind a VPN or firewall. If the public could reach Author, anyone could delete all your content.

Publish layer — what users see

The Publish server runs at http://localhost:4503. This is what your website visitors hit. Content only appears here after an editor clicks "Publish" on the Author server. This triggers replication — Author copies the content to Publish.

In production, you typically have 2 or more Publish servers behind a load balancer, so if one goes down the site stays up.

Dispatcher — cache and shield

The Dispatcher is an Apache web server module. It sits in front of Publish and does two things. First, it caches rendered HTML pages so the same page doesn't have to be re-generated by AEM for every visitor. Second, it acts as a security filter — blocking URLs that should never be public (like /crx/de or /system/console).

Real example: When Flipkart runs a sale and 2 million people visit the homepage simultaneously, the Dispatcher serves the cached HTML to 99% of them without touching AEM at all. Only when the content changes (price update, new banner) does AEM regenerate the cache. Without the Dispatcher, AEM would crash under that load.

How content moves from Author to users

1

Editor finishes editing a page on Author and clicks "Publish"

2

AEM's Replication Agent copies the page's JCR nodes from Author's repository to Publish's repository

3

AEM also sends a "flush" request to the Dispatcher, telling it to delete the old cached HTML for that page

4

The next user who visits the page gets a fresh HTML from Publish, which is then cached by Dispatcher for all subsequent visitors

JCR — the content repository

The JCR (Java Content Repository) is where AEM stores absolutely everything — pages, components, configurations, users, templates, assets — as a tree of nodes. Think of it like a file system, but instead of files you have nodes with typed properties.

What the JCR tree looks like

Every piece of data in AEM is a node. Nodes have properties (key-value pairs). Nodes can have child nodes. The tree structure mirrors a file system path.

# A page in the JCR tree:
/content/mysite/en/home          ← this is a "cq:Page" node
  └── jcr:content                ← this is a "cq:PageContent" node
       jcr:title = "Home Page"   ← property
       sling:resourceType = "mysite/components/page/base"  ← property
       └── root                  ← layout container node
            └── banner           ← banner component node
                 title = "Welcome to our site"
                 imageRef = "/content/dam/images/hero.jpg"

Key JCR paths you need to know

/apps/mysite/ — your code (components, templates, clientlibs)
/content/mysite/ — your page content (what editors create)
/content/dam/mysite/ — your assets (images, videos, PDFs)
/conf/mysite/ — your configuration (editable templates, CF models)
/libs/ — AEM's own built-in code. Never edit this.
/var/ — runtime data (workflow instances, audit logs)

How to see the JCR

Open CRXDE Lite at http://localhost:4502/crx/de. The left panel is the JCR tree. Click any node and the right panel shows all its properties. You can see exactly what an editor saved in a dialog by navigating to the component's content node.

Real example: An editor sets the title of a card component to "Summer Sale 2024". This gets saved as a JCR property title = "Summer Sale 2024" on the node at /content/mysite/en/home/jcr:content/root/maincontent/card. The developer's Sling Model reads this property, and the HTL template outputs it as HTML. This is the entire data flow.

AEM interfaces — what each one is for

AEM has several different interfaces. Each is for a different job. As a developer you will use all of them regularly.

Sites console

URL: http://localhost:4502/sites.html/content

This is the main screen. It shows a tree/card view of all your pages. From here you create pages, delete pages, publish pages, open the page editor, see version history, and manage workflows.

Page editor (WYSIWYG editor)

URL: http://localhost:4502/editor.html/content/mysite/en/home.html

The drag-and-drop page editing interface. This is what content editors use all day. Drag components from the left panel onto the page, click to configure, switch between Edit/Layout/Preview modes.

CRXDE Lite

URL: http://localhost:4502/crx/de

The JCR browser. Use it to inspect nodes and their properties, edit HTL files directly, verify what an editor saved, check component structure. This is the developer's most-used debugging tool.

Any time something is not working, open CRXDE, navigate to the component's content node, and look at the Properties tab. You can see exactly what values are saved there and what sling:resourceType is set.

OSGi console

URL: http://localhost:4502/system/console

The Felix web console. Shows all Java bundles (your deployed code). Go here to check if your bundle is "Active" after deploying. Also used to manage OSGi configurations for services. The two sub-pages you use most:

  • /system/console/bundles — check your bundle status
  • /system/console/configMgr — manage service configurations

Package manager

URL: http://localhost:4502/crx/packmgr

Packages are zip files containing JCR content. Use Package Manager to install packages (WKND, service packs, 3rd party tools) and to export content from one AEM instance to move to another.

Tools menu

URL: http://localhost:4502/aem/start.html → click the hammer icon

A large set of admin screens for: creating editable templates, managing tag taxonomies, configuring replication agents, managing users and permissions, setting up cloud services, and much more.

Session 2

What is Maven and why AEM uses it

Maven is a build tool for Java projects. AEM is a Java application, so all AEM development uses Maven to compile code, manage libraries, and deploy to AEM. Think of it like npm for Java.

What Maven does for you

Manages dependencies
You declare which libraries you need in pom.xml. Maven automatically downloads them from Maven Central. No manual JAR downloading.
Compiles and packages
Runs javac to compile your .java files, then packages everything into a .jar bundle or .zip content package.
Deploys to AEM
With the right Maven profile (-PautoInstallPackage), Maven installs your built package directly into your running AEM instance.
Manages multi-module builds
AEM projects have multiple sub-projects (core, ui.apps, ui.content). Maven coordinates building all of them in the right order.

Maven build lifecycle — the phases in order

When you run mvn clean install, Maven runs these phases in sequence:

clean
→
validate
→
compile
→
test
→
package
→
verify
→
install

You almost always run mvn clean install which cleans any previous build output, then runs all phases up to install. The -PautoInstallPackage profile adds the AEM deploy step after install.

Maven dependency scopes — important for AEM

provided — AEM already has this library. Don't bundle it. Use for all AEM and Sling APIs.

compile — Bundle this library into your JAR. Use for 3rd party libraries AEM doesn't have.

test — Only needed for running unit tests. JUnit, Mockito, AEM Mocks.

runtime — Not needed at compile time but needed when running. Rare in AEM.
If you mark an AEM API dependency as compile instead of provided, your bundle will contain duplicate classes and cause ClassLoader conflicts at runtime. Always use provided for anything that AEM already ships with.
Session 2

POM.xml — every element explained

pom.xml (Project Object Model) is the configuration file for a Maven project. Every Maven module has one. The root pom.xml is the parent — it manages all sub-modules.

Root pom.xml structure

<project>
  <!-- Who this project is -->
  <groupId>com.mycompany.mysite</groupId>   ← like a Java package
  <artifactId>mysite</artifactId>             ← project name
  <version>1.0-SNAPSHOT</version>            ← SNAPSHOT = in development
  <packaging>pom</packaging>                ← "pom" = this is the parent

  <!-- Sub-projects to build -->
  <modules>
    <module>core</module>         ← Java code
    <module>ui.apps</module>      ← components, templates
    <module>ui.content</module>   ← page content, conf
    <module>ui.config</module>   ← OSGi configs
    <module>all</module>         ← master installer
  </modules>

  <!-- Reusable variables -->
  <properties>
    <aem.host>localhost</aem.host>
    <aem.port>4502</aem.port>
    <java.version>11</java.version>
  </properties>
</project>

core/pom.xml — the Java module

<packaging>bundle</packaging>   ← builds as OSGi bundle (.jar)

<dependencies>
  <dependency>
    <groupId>com.adobe.aem</groupId>
    <artifactId>aem-sdk-api</artifactId>
    <scope>provided</scope>    ← AEM has it, don't bundle
  </dependency>
  <dependency>
    <groupId>org.apache.sling</groupId>
    <artifactId>org.apache.sling.models.api</artifactId>
    <scope>provided</scope>
  </dependency>
</dependencies>

<!-- Tells AEM where to scan for Sling Models -->
<plugin>
  <artifactId>bnd-maven-plugin</artifactId>
  <configuration>
    <bnd>Sling-Model-Packages: com.mysite.core.models</bnd>
  </configuration>
</plugin>

Generate your AEM project with Maven archetype

The AEM Maven Archetype is an official Adobe template that generates a complete, production-ready project structure with one terminal command. You never create this structure manually.

Prerequisites before running

  • Java 11 installed — verify with java -version
  • Maven 3.6+ installed — verify with mvn -version
  • AEM running on http://localhost:4502

The command — run this in your terminal

mvn -B archetype:generate \
  -D archetypeGroupId=com.adobe.aem \
  -D archetypeArtifactId=aem-project-archetype \
  -D archetypeVersion=48 \
  -D appTitle="My Site" \
  -D appId="mysite" \
  -D groupId="com.mysite" \
  -D aemVersion="6.5.0"

What each parameter means

appTitle — Human name: "Nike India". Shown in AEM UI.
appId — Short ID: "nike". Used in folder names. Lowercase, no spaces.
groupId — Java package base: "com.nike". All your Java classes start with this.
aemVersion — Your AEM version: "6.5.0" for local, "cloud" for AEM as Cloud Service.

After it runs

A folder called mysite/ is created with all modules, pom.xml files, and starter code. Open it in VS Code: code ./mysite

VS Code extensions to install: "AEM Sync" (by Adobe) for fast file sync, "HTL" for syntax highlighting in .html files, "XML" for dialog XML editing.
Session 2

Project folder structure — one job per module

The archetype creates 5 modules. Each module has exactly one job. Never mix file types between modules — Java in core, HTML in ui.apps, content in ui.content.

The complete structure

mysite/
├── core/                      ← Java code only
│   └── src/main/java/com/mysite/core/
│       ├── models/            ← Sling Models (.java)
│       ├── services/          ← OSGi Services (.java)
│       └── servlets/          ← Sling Servlets (.java)
│
├── ui.apps/                   ← JCR code (deploys to /apps)
│   └── src/main/content/jcr_root/apps/mysite/
│       ├── components/        ← component folders (HTL + dialogs)
│       ├── clientlibs/        ← CSS and JS library nodes
│       └── i18n/              ← translation strings
│
├── ui.content/                ← JCR content (deploys to /content, /conf)
│   └── src/main/content/jcr_root/
│       ├── content/mysite/    ← sample page content
│       └── conf/mysite/       ← editable templates, CF models
│
├── ui.config/                 ← OSGi config files (deploys to /apps/mysite/osgiconfig)
│   └── src/main/content/jcr_root/apps/mysite/osgiconfig/
│       ├── config/            ← all environments
│       ├── config.author/     ← author only
│       └── config.publish/    ← publish only
│
└── all/                       ← master installer (bundles all other modules)
    └── pom.xml

The golden rule

core/ → only .java files
ui.apps/ → only .html, .xml, .css, .js files that are component code
ui.content/ → only .xml files that are page content or configuration
ui.config/ → only .cfg.json OSGi config files
Session 2

Build and deploy commands

Run all commands from the project root folder in your terminal. AEM must be running on port 4502 first.

The commands

# Deploy everything — use this when in doubt
mvn clean install -PautoInstallPackage

# Only Java changed — faster, only rebuilds core bundle
mvn clean install -PautoInstallBundle -pl core

# Only HTL/dialog changed — only rebuilds ui.apps
mvn clean install -PautoInstallPackage -pl ui.apps

# Deploy to Publish server (port 4503) as well
mvn clean install -PautoInstallPackagePublish

# Skip unit tests to build faster
mvn clean install -PautoInstallPackage -DskipTests

Verify deployment worked — 3 checks

1

Go to http://localhost:4502/system/console/bundles → search for your bundle name (e.g. "mysite.core") → must say "Active". If "Installed", your Java imports are broken.

2

Go to CRXDE at http://localhost:4502/crx/de → navigate to /apps/mysite/components/ → your component folders should be there.

3

Open the page editor → your component should appear in the component browser under your group name.

Common errors and what they mean

BUILD FAILURE — Java compile error. Read the error message for which file and line.

Connection refused — AEM is not running. Start AEM first.

Bundle shows "Installed" not "Active" — Missing Java import. Go to /system/console/bundles → click your bundle → look for "Missing Dependencies".

Component not in browser — componentGroup in .content.xml doesn't match what's allowed in the template policy.

Templates — static vs editable

A template defines the starting structure and rules for a page. Every page is created from a template. There are two kinds — static (old) and editable (modern).

Static templates (legacy, before AEM 6.2)

Static templates are stored as code under /apps/mysite/templates/. They define a fixed structure — the developer hardcodes which layout containers are on the page. Template authors and editors cannot change the structure. To add a new template, a developer has to write code and deploy.

Stored at: /apps/mysite/templates/article-page/
Files: .content.xml (jcr:primaryType = cq:Template)
Changed by: developers only
Layout: fixed in code

Editable templates (modern, AEM 6.2+)

Editable templates are stored as content under /conf/mysite/settings/wcm/templates/. A template author (a power user, not a developer) creates and edits them through a visual template editor in the AEM UI. No code deployment needed to create a new template.

Stored at: /conf/mysite/settings/wcm/templates/article-page/
Created via: Tools → Sites → Templates
Changed by: template authors (no code needed)
Layout: editable in a visual editor

Inside an editable template — 3 layers

Structure
The locked parts — editors cannot change these. Use for the global header, footer, and navigation that must be the same on every page.
Initial content
Pre-populated content that appears on new pages. Editors can change it per page. A starting point, not a lock.
Policies
Rules — which components are allowed in which containers, which styles are available. Template authors set these. Editors cannot override them.
Real example: A news site creates a "Breaking News" template. The template author locks the header and side ad panel (structure — editors cannot remove them). Sets the main column as an editable Layout Container where journalists can add Text, Image, Video components (initial content + policies). Journalists create breaking news pages from this template — the structure is consistent but the content is fully flexible.
Session 3

Layout Container and Parsys

A Layout Container is the droppable area on a page where editors drag components. Without a Layout Container, editors cannot add any components. It is the foundation of page building.

What is Parsys?

Parsys (paragraph system) is the old name from AEM's predecessor (CQ5/CQ). It served the same purpose — a droppable zone. In modern AEM (6.2+) we use the Layout Container from Core Components, which is more powerful because it also supports the responsive grid.

The 12-column responsive grid

The Layout Container divides its width into 12 equal columns. Editors can set how many columns each component spans — from 1 (very narrow) to 12 (full width). This is the same grid system as Bootstrap.

# A page with a 2-column layout using the grid:
Layout Container (12 columns total)
  ├── Text component   → set to 8 columns (left, wider)
  └── Image component  → set to 4 columns (right, narrower)

# The rendered HTML:
<div class="aem-Grid aem-Grid--12">
  <div class="text aem-GridColumn--default--8">...text...</div>
  <div class="image aem-GridColumn--default--4">...image...</div>
</div>

How to use the Layout Container as an editor

1

In Edit mode: open the left panel (component browser icon) → drag a component onto the Layout Container area on the page

2

Click the component → click Configure (wrench icon) → fill in the dialog → save

3

Switch to Layout mode (toolbar) → click the component → drag the resize handle to change column span

4

Use the breakpoint selector to set different column spans for desktop, tablet, and mobile separately

Session 3

Policies and Design mode

Policies control what editors can and cannot do on pages. They are the governance layer — a template author sets them, and editors cannot override them.

What a policy controls

Allowed components
Which components editors can drag into a Layout Container. If a component is not in the allowed list, it is invisible in the component browser for that container.
Style options
Which visual styles editors can apply to a component (the Style System). Template authors define named styles with CSS classes. Editors pick from them.
Component configuration
Some components have template-level settings accessible through the policy — e.g. which heading levels are allowed in a Text component's rich text editor.
Responsive grid settings
Min/max column spans, number of columns in the container. Controls how much layout flexibility editors have.

Where policies are stored

Policies are stored in JCR at /conf/mysite/settings/wcm/policies/. When a template author edits a policy in the template editor, AEM writes the configuration to this path.

Design mode (legacy)

Design mode is the old equivalent of policies for static templates. In Design mode, a developer or admin visits a page with ?wcmmode=design in the URL and can configure which components are allowed in parsys. Design mode is replaced by policies in modern AEM but you may still see it in older projects.

Real example: An e-commerce company wants all product pages to always have the "Add to Cart" component visible and locked. They put it in the template structure (editors cannot remove it). The main content area allows Text, Image, and Specifications components (policy). Marketing cannot add a Video or Social Feed component — they are not in the allowed list. This keeps product pages consistent across the catalog.
Session 3

Sling URL resolution — how AEM renders pages

When someone visits a URL on your AEM site, Sling (the web framework inside AEM) figures out which component to use to render the response. Understanding this is critical — it explains how everything connects.

The URL anatomy

http://localhost:4502/content/mysite/en/home.article.wide.html
                      |_________________________|  |_____| |__| |___|
                               JCR path           selector ext  (suffix)

How Sling resolves a URL step by step

1

Sling takes the JCR path part (/content/mysite/en/home) and finds that JCR node

2

It reads the sling:resourceType property on that node — e.g. mysite/components/page/article

3

It looks for a rendering script at /apps/mysite/components/page/article/

4

It checks for the most specific script first: article.wide.html (selector.ext), then article.html (ext only), then GET.html

5

It runs the found .html file and returns the output to the browser

This is why the HTL filename must match the component folder name. If your component is at /apps/mysite/components/card/, the file must be card.html — that's what Sling looks for when it finds a node with sling:resourceType = mysite/components/card.

Selectors in practice

Selectors are the parts between the last slash and the extension in a URL. /content/mysite/page.print.html has a selector print. AEM looks for page.print.html first. This lets you have multiple renderings of the same page — normal, print, PDF, JSON — by adding different selector files to the component folder.

Real example: A news site has articles at /content/news/article/story.html (normal). Adding .amp.html selector gives Google AMP version at /content/news/article/story.amp.html. Adding .print.html gives a printer-friendly version. Same content, same JCR node, different renderings based on selector.

AEM Core Components — the built-in component library

Adobe provides a library of pre-built, accessible, production-ready components called "Core Components". You should always extend these rather than building from scratch — they handle image rendering, accessibility, SEO, and more.

Key Core Components

Page
The page wrapper component. Generates the HTML document structure — doctype, head, body. Every page goes through this. Extend it for your site's global header/footer.
Text
Rich text editing with configurable toolbar. Output is semantic HTML (h1-h6, p, ul, ol, a, strong, em). Auto-sanitizes HTML on save.
Image
Smart image component. Auto-generates responsive srcset with multiple sizes. Lazy loading, WebP conversion, alt text, and caption built in.
Teaser
An image + title + description + CTA button. The most used component in enterprise sites. Used for article teasers, product cards, promo blocks.
Navigation
Reads the AEM page tree to generate site navigation. Auto-updates when pages are created or deleted. Multiple levels, mobile-friendly.
List
Generates a list of pages, assets, or tagged content. Query-driven — can auto-populate based on page path or tags. Used for article lists, product listings.
Experience Fragment
Embeds a reusable Experience Fragment into a page. The XF is a standalone AEM page used as a component — update once, updates everywhere it's used.
Content Fragment
Renders a structured Content Fragment (headless content) on a page. The bridge between headless content and traditional page rendering.

All Core Components are at /libs/core/wcm/components/. The WKND project shows every one of them in use with proper configuration and styling.

Session 4

What is an AEM component

A component is a reusable piece of UI — a card, a banner, a navigation bar, a product listing. Editors drag them onto pages. Developers build them. Each component is a folder with several files.

The simplest way to think about it

Think of a Lego brick. AEM components are Lego bricks. Each brick has a specific shape and function. Editors pick bricks from a panel and snap them together to build a page. The developer designs the bricks — what they look like and what settings they have.

What makes up a component — the 4 essentials

1. Definition (.content.xml)
Registers the component with AEM — its name, which group it belongs to, which parent it extends. AEM reads this to show the component in the browser.
2. Dialog (_cq_dialog)
The popup form editors fill in to configure the component — title, image, link. Each field saves a property in JCR when saved.
3. Sling Model (Java)
Java class that reads the JCR properties and exposes them as getter methods. The bridge between data storage and the template.
4. HTL template (card.html)
The HTML template that calls the model and renders the final HTML output. Uses AEM's templating language with special data-sly-* attributes.

The complete data flow

Editor fills dialog
→
Saves to JCR node
title="Summer Sale"
→
Sling Model reads JCR
@ValueMapValue title
→
HTL calls model
${model.title}
→
HTML output
<h2>Summer Sale</h2>
Real example: A retail site has a "Product Card" component. The developer builds it once. Editors use it 500 times across the site — one per product. Each instance is configured with different data (product name, price, image) through the dialog. The same component code renders 500 different cards. That's the power of components.

Component folder structure — every file

Here is the exact folder structure for a component called "card". Every file has a specific job. None of them are optional except the design dialog.

Where to create it

ui.apps/src/main/content/jcr_root/apps/mysite/components/
└── card/                        ← create this folder
    ├── .content.xml             ← component definition
    ├── card.html                ← HTL rendering template
    ├── _cq_dialog/              ← subfolder for dialog
    │    └── .content.xml        ← author dialog definition
    ├── _cq_editConfig/          ← optional: author toolbar config
    │    └── .content.xml
    ├── _cq_design_dialog/       ← optional: template-level settings
    │    └── .content.xml
    └── thumbnail.png            ← optional: 80x80px icon in browser

What each file does

.content.xml — The component node metadata. Sets jcr:title (browser name), componentGroup (browser tab), sling:resourceSuperType (parent component).

card.html — The rendering template. Filename MUST match the folder name. This is what AEM runs to produce HTML when the component is on a page.

_cq_dialog/.content.xml — Defines the Configure dialog. Contains all the form fields editors use. Each field writes to a JCR property via name="./propertyname".

_cq_editConfig/.content.xml — Controls the blue edit bar in author mode. Which buttons show, what refreshes after editing, empty placeholder text.

_cq_design_dialog/.content.xml — Template-level policy form. Used by template authors, not editors. Settings here apply to all instances on all pages using the template.
Start with just .content.xml, card.html, and _cq_dialog/.content.xml. Add editConfig and design dialog later when you need them. Don't create empty files.
Session 4

.content.xml — the component definition

This file is placed directly inside the component folder. It defines the JCR node for the component itself — not the component's content, but its identity and metadata.

The complete file

<?xml version="1.0" encoding="UTF-8"?>
<jcr:root
  xmlns:cq="http://www.day.com/jcr/cq/1.0"
  xmlns:jcr="http://www.jcp.org/jcr/1.0"
  xmlns:sling="http://sling.apache.org/jcr/sling/1.0"

  jcr:primaryType="cq:Component"

  jcr:title="Card"

  jcr:description="A card with title, description and CTA button"

  componentGroup="My Site - Content"

  sling:resourceSuperType="core/wcm/components/teaser/v2/teaser"
/>

Every property explained

jcr:primaryType="cq:Component" — Required. Tells JCR this node is a component definition. Without this, AEM does not recognize the folder as a component.

jcr:title="Card" — The name shown in the component browser when editors drag components. Make it descriptive: "Product Card", "Hero Banner", "Testimonial Slider".

jcr:description="..." — Tooltip shown on hover in the component browser. Helps editors understand what the component is for.

componentGroup="My Site - Content" — Which tab in the component browser this appears under. All your custom components should share the same group so editors know which are custom. This MUST match what's configured in the template policy.

sling:resourceSuperType="core/wcm/components/teaser/v2/teaser" — The parent component to inherit from. By inheriting from Core Components' Teaser, you get all its image handling, accessibility, and dialog fields for free. You only override what you need to change.
The componentGroup value is case-sensitive and must exactly match what you allow in the template policy. If it doesn't match, your component will be invisible in the component browser even after deployment.
Session 4

HTL file — the HTML template

HTL (HTML Template Language) is AEM's templating language. It is HTML with special data-sly-* attributes that add logic. It replaced JSP in AEM 6.1. It is secure by default — all output is automatically escaped against XSS attacks.

Why HTL instead of JSP?

In JSP you could write arbitrary Java code inside the HTML. This led to messy code mixing logic and markup, and created XSS vulnerabilities when developers forgot to escape output. HTL separates logic (Sling Models) from markup (HTL) completely, and escapes everything automatically.

A complete card.html

<!-- Step 1: Load the Sling Model and bind it to "model" variable -->
<div data-sly-use.model="com.mysite.core.models.CardModel"
     class="card">

  <!-- data-sly-test: only renders this element if model.title is not null -->
  <h3 class="card__title"
       data-sly-test="${model.title}">
    ${model.title}
  </h3>

  <!-- data-sly-test with "not": renders when description IS null -->
  <p class="card__empty"
     data-sly-test="${!model.description}">
    No description provided.
  </p>

  <!-- @ context='uri': use URI encoding for links, not HTML encoding -->
  <a class="card__cta"
     href="${model.ctaLink @ context='uri'}"
     data-sly-test="${model.hasLink}">
    ${model.ctaText}
  </a>

  <!-- data-sly-list: loop over a collection -->
  <ul data-sly-test="${model.tags}">
    <li data-sly-list.tag="${model.tags}">${tag}</li>
  </ul>

</div>

All HTL expressions and attributes

${model.title} — outputs the value from a getter, auto HTML-escaped
${model.link @ context='uri'} — URL-encode instead of HTML-encode
${model.html @ context='html'} — output raw HTML (only for trusted content)
data-sly-use.model="..." — load and bind a Sling Model or Java class
data-sly-test="${condition}" — hide element if condition is falsy
data-sly-list.item="${collection}" — repeat element for each item in a list
data-sly-include="partial.html" — include another HTL file
data-sly-resource="${'path'}" — render another JCR resource
data-sly-text="${value}" — replace element content with value
data-sly-attribute.href="${value}" — set an attribute dynamically
Always use @ context='uri' on href and src attributes. Without it, AEM HTML-encodes the URL and links break with special characters.

Dialog (_cq_dialog) — in complete detail

The dialog is the popup form that appears when an editor clicks the Configure (wrench) icon on your component. It is defined as a single XML file inside the _cq_dialog subfolder.

The complete dialog.xml with all field types

<?xml version="1.0" encoding="UTF-8"?>
<jcr:root xmlns:sling="..." xmlns:jcr="..." xmlns:nt="..."
  jcr:primaryType="nt:unstructured"
  sling:resourceType="cq/gui/components/authoring/dialog"
  jcr:title="Card Settings">

  <content jcr:primaryType="nt:unstructured"
    sling:resourceType="granite/ui/components/coral/foundation/container">
    <items jcr:primaryType="nt:unstructured">

      <!-- TEXT FIELD: single line input -->
      <title
        sling:resourceType="granite/ui/components/coral/foundation/form/textfield"
        fieldLabel="Title"
        fieldDescription="The heading of the card"
        name="./title"
        required="{Boolean}true"/>

      <!-- TEXTAREA: multi-line input -->
      <description
        sling:resourceType="granite/ui/components/coral/foundation/form/textarea"
        fieldLabel="Description"
        name="./description"
        rows="3"/>

      <!-- PATHFIELD: browse and select a page or asset -->
      <ctaLink
        sling:resourceType="granite/ui/components/coral/foundation/form/pathfield"
        fieldLabel="Button Link"
        name="./ctaLink"
        rootPath="/content"/>

      <!-- SELECT: dropdown with fixed options -->
      <theme
        sling:resourceType="granite/ui/components/coral/foundation/form/select"
        fieldLabel="Card Theme"
        name="./theme">
        <items jcr:primaryType="nt:unstructured">
          <default jcr:title="Default" value="" selected="{Boolean}true"/>
          <dark    jcr:title="Dark"    value="theme-dark"/>
          <light   jcr:title="Light"   value="theme-light"/>
        </items>
      </theme>

      <!-- CHECKBOX: true/false toggle -->
      <openInNewTab
        sling:resourceType="granite/ui/components/coral/foundation/form/checkbox"
        fieldLabel="Open link in new tab"
        name="./openInNewTab"
        value="true"
        uncheckedValue="false"/>

    </items>
  </content>
</jcr:root>

The most important rule — the name property

Every field has a name property. This tells AEM where to save the value in JCR when the editor saves the dialog. The ./ prefix means "relative to the component's content node".

So name="./title" saves the value as a property called title on the JCR node at /content/mysite/en/home/jcr:content/root/card. Your Sling Model then reads it with @ValueMapValue private String title;. The field name in XML and the Java field name must match.

_cq_editConfig — the author toolbar

When an editor clicks on your component in author mode, a blue bar appears at the top of the component. The editConfig defines what buttons appear in that bar and what happens after certain actions.

The complete editConfig

<?xml version="1.0" encoding="UTF-8"?>
<jcr:root xmlns:cq="..." xmlns:jcr="..."
  jcr:primaryType="cq:EditConfig"

  <!-- Which buttons to show in the blue toolbar -->
  cq:actions="[edit,delete,insert,copymove]"

  <!-- Style of the toolbar: "editbar" (above) or "rollover" (overlay) -->
  cq:layout="editbar"

  <!-- Shown when component has no content yet -->
  cq:emptyText="Configure the Card component">

  <!-- Event handlers -->
  <cq:listeners
    jcr:primaryType="cq:EditListenersConfig"
    afteredit="REFRESH_PAGE"
    aftercopy="REFRESH_SELF"
    afterdelete="REFRESH_PARENT"/>

  <!-- Enable click-to-edit directly on the component -->
  <cq:inplaceEditing
    cq:childEditors="[title]"
    active="{Boolean}true"/>
</jcr:root>

cq:actions — all available options

edit — Opens the configure dialog (wrench icon)
delete — Deletes the component
insert — Insert a new component after this one
copymove — Copy and move handles
EDIT — Uppercase means it's a primary action (larger button)

afteredit listeners — what happens after

REFRESH_PAGE — Reload the entire page. Use when your component affects page structure or other components.
REFRESH_SELF — Reload only this component. Faster. Use for most cases where only this component's output changes.
REFRESH_PARENT — Reload the parent container. Use when your component affects siblings.
Session 4

Design dialog — template-level settings

The design dialog (also called the policy dialog) is opened by template authors — not page editors. Settings here apply to ALL instances of the component on ALL pages using that template.

Design dialog vs regular dialog

Regular dialog (_cq_dialog)
Per-instance settings. Every card on every page can have different values. The editor sets it when they configure that specific component on that specific page.
Design dialog (_cq_design_dialog)
Template-wide settings. Applied to all instances of this component on all pages using this template. Set by template authors in the template editor.

Example use cases for design dialog

  • Which heading levels (H1-H6) are allowed in the rich text editor
  • Maximum number of items in a list component
  • Whether the "dark theme" option should be available to editors
  • Which image ratios are allowed (1:1, 16:9, free)

Where design dialog settings are saved

Settings from the design dialog are saved to /conf/mysite/settings/wcm/policies/ — not to /content/. This is an important difference. Regular dialog values go to /content/. Design/policy values go to /conf/.

How to read design dialog values in Sling Model

@ScriptVariable
private Style currentStyle;  // inject the policy/design object

@PostConstruct
protected void init() {
  // Read a policy value — e.g. max heading level
  int maxLevel = currentStyle.get("headingLevels", 3);
}

Inheritance in AEM — sling:resourceSuperType

Inheritance lets your component extend another component. You get all its files (dialog, HTL, logic) for free and only override what you need to change. This is the standard pattern for building on top of Core Components.

How inheritance works

Set sling:resourceSuperType in your .content.xml. When AEM resolves your component, it checks your folder first. If a file is not found there, it walks up to the parent and uses that file instead.

# Your component .content.xml:
sling:resourceSuperType = "core/wcm/components/teaser/v2/teaser"

# AEM resolution order for card.html:
1. Check /apps/mysite/components/card/card.html        ← YOUR file
2. If not found: /apps/core/wcm/components/teaser/v2/teaser/teaser.html
3. If not found: walk up superType chain further

# AEM resolution order for _cq_dialog:
1. Check /apps/mysite/components/card/_cq_dialog/      ← YOUR dialog
2. If not found: use parent's _cq_dialog (teaser's dialog)

Practical example — overriding only what you need

Your "BankTeaser" extends Core Teaser. You only need to add one extra field to the dialog (a "disclaimer text" field for regulatory compliance). You only create:

bankteaser/.content.xml — set sling:resourceSuperType to Core Teaser
bankteaser/_cq_dialog/.content.xml — your dialog with the extra field

You do NOT copy card.html, clientlibs, or models. AEM inherits them all automatically from Core Teaser. You only override the dialog. This keeps your code small and ensures you pick up Core Component updates.
Real example: A pharmaceutical company needs a standard Teaser component plus a "Safety information" field with a specific disclaimer. Instead of copying 1000 lines of Core Teaser code, they create a component that inherits from Core Teaser and adds only 20 lines of XML for the new field. Their developers focus on the business requirement, not rebuilding what Adobe already built.

Overlay and Override

Overlay and override are two techniques for customizing AEM's built-in behavior without touching the /libs directory — which is sacred and must never be edited.

The rule: never edit /libs

/libs contains AEM's own code — Core Components, admin UIs, default scripts. When Adobe releases a Service Pack or upgrade, they overwrite /libs. If you edited files there, your changes are gone. All customization must be done under /apps.

Overlay — copy to the same path under /apps

AEM always resolves /apps before /libs. So if you copy a file from /libs/some/path/file.html to /apps/some/path/file.html, AEM will use YOUR version, ignoring the one in /libs.

# Original in /libs — DO NOT TOUCH:
/libs/cq/gui/components/authoring/clientlibs/rte/js/plugins/myPlugin.js

# Overlay in /apps — your customized version:
/apps/cq/gui/components/authoring/clientlibs/rte/js/plugins/myPlugin.js
# AEM uses this one. libs version is ignored.

Override — different path, different sling:resourceType

Override means pointing to a completely different component rather than the inherited parent. You change sling:resourceType on a content node to point to your custom component instead of the default one.

# Default page uses core page component:
/content/mysite/en/home/jcr:content
  sling:resourceType = "core/wcm/components/page/v3/page"

# Change it to YOUR page component (override):
sling:resourceType = "mysite/components/page/base"
# Now your page component renders this page

When to use which

Use Overlay when
Customizing AEM's own UI — admin screens, RTE plugins, author clientlibs. You are modifying something in /libs that has no inheritance mechanism.
Use Override/Inheritance when
Building your own components that extend Core Components. You extend via sling:resourceSuperType and only override specific files in your component folder.
Session 4

Style System

The Style System lets template authors define named CSS class groups as "styles". Editors pick a style from a paint palette on each component instance. Zero code needed to switch a component's visual appearance.

The concept

Imagine a "Teaser" component. As a developer you build one Teaser with its HTML and CSS. But the marketing team wants three visual variants — Standard, Featured (bigger with a highlight), and Minimal (text only, no image). Without the Style System, you'd need three separate components. With the Style System, you build one component and add CSS classes for variants.

How it works — 3 steps

1

Developer: Add CSS classes for each variant to the component's clientlib. .card--featured { border: 2px solid gold; font-size: 1.2em; }

2

Template Author: In the template editor → click the component → Policy → Add Style Group → name it "Theme" → add options: "Standard" (class: empty), "Featured" (class: card--featured), "Minimal" (class: card--minimal)

3

Editor: On the page, click the component → click the paint palette icon in the toolbar → pick "Featured" → the CSS class is added automatically

Where the CSS class is added

AEM adds the style CSS class to the outermost wrapper div of the component. If the editor picks "Featured", the rendered HTML becomes:

<div class="card card--featured">
  <!-- component content -->
</div>

Where styles are stored in JCR

Style groups and options are stored in the component's policy at /conf/mysite/settings/wcm/policies/. When an editor picks a style for a specific component instance on a page, the chosen class is stored as a cq:styleIds property on that component's content node.

Real example: A telecom company has a "Plan Card" component showing mobile plans. The Standard style shows the card normally. The "Best Value" style adds a golden border and a "Recommended" badge via CSS. The "Budget" style shows a minimal grey card. The marketing team can switch plans between these styles any time through the editor — no developer, no deployment, no waiting. One component, three looks.
