Part 1 — why Maven

The problem Maven solves

Before learning Maven, you need to understand the painful reality of Java development without it. Every concept in Maven makes perfect sense once you see what it replaces.

Java development before tools like Maven

You write a CardModel.java file. To compile it, your code imports classes from Sling, from AEM, from Apache Commons — all external libraries stored as JAR files. Without Maven, you must do everything manually:

1
Download every JAR manually
Go to Apache's website. Download sling-api-2.24.0.jar. Go to Adobe's portal. Download aem-api-6.5.0.jar. Go to Apache Commons. Download commons-lang3-3.12.0.jar. Repeat for every library your code needs — could be 30+ libraries.
2
Manage version compatibility yourself
Library A version 2.0 requires Library B version 1.5. But Library C requires Library B version 1.8. These conflict. You read through documentation to figure out which versions are compatible. This is called "dependency hell".
3
Compile with a huge classpath
Run: javac -classpath ./libs/sling-api.jar:./libs/aem-api.jar:./libs/commons-lang3.jar:... CardModel.java. One typo in the path and it fails. Every time you add a library, you update this command.
4
Package everything manually
Zip all compiled .class files into a JAR. Write a MANIFEST.MF file by hand. Package the JAR into a content package ZIP following AEM's vault format. Upload the ZIP to AEM Package Manager. Do this every time you make any change.

This was slow, error-prone, and inconsistent between developers. The same code compiled differently on different machines because everyone had downloaded slightly different library versions.

What Maven changes — one command, everything automated

With Maven, you write one XML file (pom.xml) that declares what your project is and what it needs. Then you run mvn clean install -PautoInstallPackage. Maven handles everything: downloading the right version of every library, compiling your code, running tests, packaging into the right format, and deploying to AEM. One command. Every time. Consistently on every developer's machine.

The npm analogy — most important
If you have done web development with Node.js, Maven is exactly like npm. The pom.xml is like package.json. Maven Central is like the npm registry. Running mvn install is like running npm install. The core idea is identical — declare what you need and the tool fetches it.
Part 1 — why Maven

What Maven is — four jobs in one tool

Maven does four distinct jobs. Understanding each one separately helps you understand why Maven commands are written the way they are.

Job 1 — dependency management

You declare in pom.xml that you need org.apache.sling:org.apache.sling.models.api:1.5.0. Maven goes to Maven Central (a public registry of millions of JARs), downloads that exact version, caches it at ~/.m2/repository/ on your machine, and puts it on the classpath when compiling. Maven also handles transitive dependencies — if Library A needs Library B which needs Library C, Maven downloads all three. You only declared Library A.

Job 2 — standardised project structure

Maven defines a layout that every Maven project follows. Because it is standardised, any developer who has used Maven can instantly understand any Maven project without a setup guide:

src/main/java/ ← your production Java files (.java) src/main/resources/ ← non-Java files bundled with the code src/test/java/ ← your unit test files src/test/resources/ ← test resource files pom.xml ← the Maven config file target/ ← compiled output (never edit this)
Job 3 — build lifecycle

Maven runs builds in a standardised sequence of phases: validate → compile → test → package → verify → install → deploy. When you run mvn install, Maven automatically runs all phases before it in order. You never skip compile and jump to package — the sequence is enforced.

Job 4 — plugin execution

Maven itself does almost nothing. All real work is done by plugins declared in pom.xml. The Java compiler plugin compiles code. The bnd plugin creates OSGi bundles. The content-package plugin creates and installs AEM packages. Plugins run automatically at the right lifecycle phase.

Real example
A developer at a company runs mvn clean install -PautoInstallPackage after fixing a bug in a Sling Model. Maven: cleans the previous build, compiles 47 Java files, runs 23 unit tests, creates an OSGi bundle JAR, creates a content package ZIP containing the JAR and all HTL/XML files, uploads the ZIP to AEM's Package Manager API, and waits for AEM to activate the bundle. All from one command. Without Maven, each step would be manual work.
Part 1 — why Maven

Build lifecycle and phases

Every Maven command triggers a sequence of phases. Understanding each phase tells you exactly what Maven is doing when you run a command — and why certain commands are faster than others.

The default lifecycle — all phases in order
validate
Checks that pom.xml is valid and the project structure is correct. Runs in milliseconds.
compile
Runs javac on all .java files in src/main/java/. Produces .class files in target/classes/. Java syntax errors are caught here. The build stops with a "compilation failure" message if any file has an error.
test
Compiles test code from src/test/java/ and runs unit tests with JUnit. If any test fails, the build stops. Add -DskipTests to skip this phase when you need speed.
package
Takes compiled .class files and packages them into the output format declared by <packaging> — either a .jar (OSGi bundle) or a .zip (content package). Output goes to target/.
verify
Runs integration checks — validates the package structure, checks OSGi manifest correctness.
install
Copies the built artifact into your local Maven cache at ~/.m2/repository/. Other modules in the same project can now use it as a dependency.
deploy
Uploads artifact to a remote Maven repository (Nexus, Artifactory). Used in CI/CD. Not used for AEM deployment.
clean — the separate lifecycle

The clean lifecycle is separate. mvn clean deletes the target/ folder — removing all previous build output. You almost always combine it with install: mvn clean install. This ensures you are building from scratch, not from stale compiled files. Without clean, Maven only recompiles changed files — faster but can leave old compiled classes that cause subtle bugs.

The AEM deploy profile — what -PautoInstallPackage actually does

The -PautoInstallPackage flag activates a Maven profile defined in the archetype's pom.xml. This profile configures the content-package-maven-plugin to run after the install phase — it takes the built .zip and uploads it to your AEM instance via the Package Manager API at http://localhost:4502/crx/packmgr/service.jsp. This is not a standard Maven lifecycle phase — it is AEM-specific plugin configuration activated by the profile. That is why deployment only happens when you include -PautoInstallPackage.

Part 1 — why Maven

Installing Maven on your machine

Maven must be installed before any AEM development. Here is the exact process with all verification steps.

Prerequisites
Java 11 (JDK)
AEM 6.5 requires Java 11 specifically. Not 8, not 17, not 21. Download from adoptium.net — Temurin 11 LTS. After installing: java -version must show 11.x.x
Maven 3.6+
Download from maven.apache.org/download.cgi — Binary zip archive, e.g. apache-maven-3.9.6-bin.zip. After installing: mvn -version must show 3.6.x or higher
Installation steps
1
Check if already installed
Run mvn -version in your terminal. If you see Maven version info — it is already installed. If "command not found" — proceed.
2
Extract Maven
Mac/Linux: extract to /opt/maven/apache-maven-3.9.6/. Windows: extract to C:\Program Files\Maven\apache-maven-3.9.6\
3
Set environment variables
Mac/Linux — add to ~/.zshrc or ~/.bash_profile:
export M2_HOME=/opt/maven/apache-maven-3.9.6
export PATH=$PATH:$M2_HOME/bin
Windows: set system variable M2_HOME and add %M2_HOME%\bin to PATH.
4
Verify both Java and Maven
Restart terminal. Run both:
java -version → must show 11.x.x
mvn -version → must show 3.6.x or higher AND Java 11
If mvn -version shows a different Java version than java -version, Maven is using a different JDK. Set JAVA_HOME to your Java 11 installation folder and restart the terminal.
Part 2 — POM.xml

What is pom.xml — the project's DNA

POM stands for Project Object Model. The pom.xml file defines everything about a Maven project — its identity, its dependencies, how it should be built, and how it should be deployed. Every single Maven module has exactly one pom.xml.

The minimum possible pom.xml
pom.xml — absolute minimum
<project> <modelVersion>4.0.0</modelVersion> ← always 4.0.0, never change this <groupId>com.mysite</groupId> <artifactId>mysite.core</artifactId> <version>1.0-SNAPSHOT</version> <packaging>bundle</packaging> ← what format to produce </project>

This alone tells Maven: "this module's identity is com.mysite:mysite.core:1.0-SNAPSHOT and it should produce an OSGi bundle JAR file." That is the minimum Maven needs.

The six main sections of a pom.xml
coordinates
groupId + artifactId + version — the unique identity of this module (the GAV coordinates)
parent
Which parent pom.xml this module inherits from. Child modules inherit all dependency versions and plugin configurations from the parent.
properties
Reusable variables referenced anywhere in the pom.xml with ${propertyName} syntax
dependencies
Libraries your code needs. Each has its own GAV coordinates and a scope (how it is used)
build/plugins
Which Maven plugins to run and how to configure them — compile settings, bnd OSGi generation, content package creation
modules
Only on the root pom.xml — lists all child modules that Maven should build as part of this project
Part 2 — POM.xml

GAV coordinates — the unique identity of every artifact

Every JAR or ZIP in the Maven world has a unique identity made of three parts: GroupId, ArtifactId, and Version. These are called GAV coordinates. Maven uses them to know exactly which library to download and to identify your own modules.

GroupId — the organisation

GroupId identifies the organisation or project family. Convention is a reversed domain name — just like Java packages:

  • com.mycompany — your company's projects
  • org.apache.sling — the Apache Sling project
  • com.adobe.aem — Adobe's AEM libraries
  • com.day.cq — older CQ/AEM libraries (Day Software was acquired by Adobe)

For your AEM project: if your company's domain is mybank.com, use com.mybank. This becomes the root Java package for all your classes too.

ArtifactId — the project name

Identifies this specific project within the group. Becomes the base filename of the built artifact. AEM conventions:

  • mysite — the root parent module
  • mysite.core — the Java bundle module
  • mysite.ui.apps — the component package module
  • mysite.all — the master installer

Keep it lowercase with dots or hyphens. No spaces, no uppercase.

Version — the release identifier
1.0-SNAPSHOT
SNAPSHOT = in development. Maven always fetches the latest SNAPSHOT from remote even if cached. Use during active development. The -SNAPSHOT suffix is a convention, not a number.
1.0.0
A release version. Maven caches this permanently — once downloaded, never fetches again. Use for stable, released artifacts.
How Maven uses GAV to build a download URL

When you declare a dependency with GAV coordinates, Maven constructs the download URL automatically:

GAV coordinates → download URL
GroupId: org.apache.sling ArtifactId: org.apache.sling.models.api Version: 1.5.0 → Downloads from: https://repo.maven.apache.org/maven2/ org/apache/sling/ ← groupId (dots→slashes) org.apache.sling.models.api/ ← artifactId 1.5.0/ ← version org.apache.sling.models.api-1.5.0.jar ← artifactId-version.jar
Part 2 — POM.xml

Dependencies and scopes — the most important concept for AEM

Dependencies are the external libraries your code needs. Each dependency has a scope that tells Maven how to use it. Getting scopes wrong in AEM causes runtime errors that are confusing to debug.

The four scopes
provided
The library is available at compile time so your code compiles, but do NOT bundle it into your JAR. AEM already ships this library — it will be available at runtime. Use this for all AEM APIs, Sling APIs, OSGi APIs. This is the most common scope in AEM development.
compile
Bundle this library inside your JAR. Use for third-party libraries that AEM does NOT provide — custom PDF tools, external SDK clients, etc. If you mark an AEM API as compile, you get ClassLoader conflicts at runtime because the same class loads twice.
test
Only available when compiling and running unit tests. Never bundled into your JAR. Use for JUnit, Mockito, AEM Mocks. These are only needed to test your code, not at runtime.
runtime
Not needed at compile time but needed at runtime. Rare in AEM. Used for JDBC drivers or libraries your code references only by string name via reflection.
The golden rule — provided vs compile

This is the single most common mistake in AEM Java development:

Correct — scope: provided
Wrong — scope: compile
All AEM and Sling APIs: aem-sdk-api, sling-api, osgi-core, jcr, cq-wcm-api — all must be provided because AEM already ships them.
Using compile for AEM APIs bundles the same classes twice — once in AEM and once in your bundle. At runtime, AEM throws ClassCastException because two ClassLoaders each loaded "the same" class. Very confusing to debug.
Complete dependencies section for core/pom.xml
core/pom.xml — dependencies section
<dependencies> <!-- AEM SDK API — provided: AEM already has this --> <dependency> <groupId>com.adobe.aem</groupId> <artifactId>aem-sdk-api</artifactId> <scope>provided</scope> </dependency> <!-- Sling Models API — provided: AEM has this --> <dependency> <groupId>org.apache.sling</groupId> <artifactId>org.apache.sling.models.api</artifactId> <scope>provided</scope> </dependency> <!-- Apache Commons Lang — provided: AEM bundles this --> <dependency> <groupId>org.apache.commons</groupId> <artifactId>commons-lang3</artifactId> <scope>provided</scope> </dependency> <!-- 3rd party library AEM doesn't have — compile: bundle it --> <dependency> <groupId>com.some.thirdparty</groupId> <artifactId>custom-pdf-tool</artifactId> <version>2.1.0</version> <scope>compile</scope> ← WILL be bundled in your JAR </dependency> <!-- JUnit — test: only for unit tests, never bundled --> <dependency> <groupId>org.junit.jupiter</groupId> <artifactId>junit-jupiter</artifactId> <scope>test</scope> </dependency> <!-- AEM Mocks for testing — test only --> <dependency> <groupId>io.wcm.testing</groupId> <artifactId>io.wcm.testing.aem-mock.junit5</artifactId> <scope>test</scope> </dependency> </dependencies>
Part 2 — POM.xml

Plugins — what actually does the work

Maven itself is just an engine. All real work — compiling, packaging, deploying — is done by plugins declared in pom.xml. Each plugin has one specific job.

Key plugins in AEM projects
maven-compiler-plugin
Compiles your .java files using javac. Must be configured with <source>11</source> and <target>11</target> for AEM 6.5. Runs at the compile phase.
bnd-maven-plugin
Creates the OSGi bundle JAR with the correct MANIFEST.MF. Processes your @Component, @Service, @Model annotations and generates OSGi metadata from them. The most critical plugin for the core module.
content-package-maven-plugin
Creates the AEM content package ZIP from the JCR content in src/main/content/. Also handles installing the package to AEM when the autoInstall profile is active. Used in ui.apps, ui.content, ui.config, and all modules.
maven-surefire-plugin
Runs your unit tests during the test phase. Reports which tests pass and fail. Build stops if any test fails (unless you use -DskipTests).
frontend-maven-plugin
Used in ui.frontend only. Runs npm install and npm run build to compile SCSS and TypeScript into CSS and JS output files.
The bnd plugin — the most important AEM plugin

The bnd plugin turns a regular JAR into an OSGi bundle. It reads your Java source annotations and generates a MANIFEST.MF that registers your classes with OSGi. Without it, AEM does not recognise any of your components or services. The most critical configuration:

core/pom.xml — bnd plugin configuration
<plugin> <groupId>biz.aQute.bnd</groupId> <artifactId>bnd-maven-plugin</artifactId> <configuration> <bnd> # This tells AEM which Java package to scan for Sling Models. # Without this one line, data-sly-use returns null silently. # This is the most forgotten configuration in AEM projects. Sling-Model-Packages: com.mysite.core.models </bnd> </configuration> </plugin>
The Sling-Model-Packages entry is the single most commonly forgotten configuration. Without it, AEM never scans your models package and every data-sly-use in HTL returns null with no error message. Always add this when creating Sling Models.
Part 2 — POM.xml

Properties and profiles

Properties are reusable variables in pom.xml. Profiles are sets of configuration that activate based on conditions. Together they let one pom.xml work for all environments.

Properties — define once, use everywhere
pom.xml — properties section
<properties> <aem.host>localhost</aem.host> ← reference as ${aem.host} <aem.port>4502</aem.port> <aem.publish.host>localhost</aem.publish.host> <aem.publish.port>4503</aem.publish.port> <java.version>11</java.version> <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding> </properties> <!-- Override at command line: mvn install -Daem.host=staging.mysite.com -->
Profiles — configuration sets that activate on demand

A profile is a block of configuration activated when you add -PprofileName to your Maven command. The archetype creates these profiles:

autoInstallPackage
Activated with -PautoInstallPackage. Configures the content-package plugin to install the built package to AEM Author on port 4502.
autoInstallPackagePublish
Same but targets AEM Publish on port 4503. Use when testing published content rendering.
autoInstallBundle
Installs only the OSGi bundle (core module JAR) to AEM. Faster than installing the full package. Use when only Java code changed.

You can also override properties at the command line to deploy to remote servers: mvn install -PautoInstallPackage -Daem.host=staging.company.com -Daem.port=4502

Part 2 — POM.xml

Parent POM and multi-module projects

An AEM project is not one Maven module — it is several. The root pom.xml is the parent that coordinates all sub-modules. Understanding this relationship is essential for working with AEM projects.

The parent declares the children

The root pom.xml has <packaging>pom</packaging> — it produces no artifact itself. It just organises. It lists all child modules in the <modules> section. Running mvn install from the root builds all child modules in the correct order.

Each child declares its parent
core/pom.xml — the parent declaration
<project> <!-- Declare who the parent is --> <parent> <groupId>com.mysite</groupId> <artifactId>mysite</artifactId> <version>1.0-SNAPSHOT</version> <relativePath>../pom.xml</relativePath> </parent> <!-- Only set what is different from the parent --> <artifactId>mysite.core</artifactId> <packaging>bundle</packaging> <!-- Dependencies — no version needed if declared in parent's dependencyManagement section --> <dependencies> <dependency> <groupId>com.adobe.aem</groupId> <artifactId>aem-sdk-api</artifactId> <!-- no <version> needed — inherits from parent --> </dependency> </dependencies> </project>
What children inherit from the parent
  • All <dependencyManagement> version declarations — so children can omit version numbers
  • All <properties> variables like ${aem.host}
  • Plugin configurations and versions declared in <pluginManagement>
  • Build profiles like autoInstallPackage
Build order: Maven analyses inter-module dependencies and builds them in the right order automatically. all/ is always last. core/ is always early since ui.apps may depend on the core bundle being installed first.

Full root pom.xml — annotated line by line

The complete root pom.xml with every section explained. This is the file at mysite/pom.xml — the parent of all other modules.

mysite/pom.xml — complete annotated root POM
<project> <modelVersion>4.0.0</modelVersion> <!-- ① Identity of this root project --> <groupId>com.mysite</groupId> <artifactId>mysite</artifactId> <version>1.0-SNAPSHOT</version> <packaging>pom</packaging> ← "pom" = aggregator, no artifact produced <!-- ② Reusable variables --> <properties> <aem.host>localhost</aem.host> <aem.port>4502</aem.port> <aem.publish.host>localhost</aem.publish.host> <aem.publish.port>4503</aem.publish.port> <java.version>11</java.version> <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding> </properties> <!-- ③ Sub-projects to build (in dependency order) --> <modules> <module>core</module> ← Java code → OSGi bundle (.jar) <module>ui.apps</module> ← components, clientlibs → .zip <module>ui.content</module> ← page content, conf → .zip <module>ui.config</module> ← OSGi configs → .zip <module>ui.frontend</module> ← SCSS/TS → compiled into ui.apps <module>all</module> ← master installer, always last </modules> <!-- ④ Shared dependency version declarations Children reference these without specifying versions --> <dependencyManagement> <dependencies> <dependency> <groupId>com.adobe.aem</groupId> <artifactId>aem-sdk-api</artifactId> <version>2023.11.14227.20231114T064801Z-231100</version> <scope>provided</scope> </dependency> </dependencies> </dependencyManagement> <!-- ⑤ Profiles — activated with -P flag --> <profiles> <profile> <id>autoInstallPackage</id> <activation><activeByDefault>false</activeByDefault></activation> <build> <plugins> <plugin> <groupId>com.day.jcr.vault</groupId> <artifactId>content-package-maven-plugin</artifactId> <configuration> <serviceURL> http://${aem.host}:${aem.port}/crx/packmgr/service.jsp </serviceURL> ← where Maven uploads the package </configuration> </plugin> </plugins> </build> </profile> </profiles> </project>
Part 3 — packaging

Packaging types — what each module produces

The <packaging> element tells Maven what output format to produce. AEM projects use three distinct types across their modules.

pom
Produces no file artifact. Used for the root parent module and the all module. The root just organises child builds. all just pulls in all other packages as embedded dependencies.
bundle
Produces an OSGi bundle — a JAR file with a special MANIFEST.MF declaring Import-Package and Export-Package. Used only by the core module. AEM's Felix container loads this JAR at runtime. Your Sling Models and Services live here.
content-package
Produces an AEM content package — a ZIP file in Apache Jackrabbit FileVault format containing JCR content serialised as XML. Used by ui.apps, ui.content, ui.config, and all. AEM installs this via Package Manager.
How each type gets into AEM
1
bundle (core) — indirect install
The core bundle JAR is embedded inside the all content package at path /apps/mysite/install/mysite.core-1.0-SNAPSHOT.jar. When AEM installs the all package, it scans /apps/*/install/ folders and automatically installs any JARs found there as OSGi bundles into Felix.
2
content-package (ui.apps, etc.) — direct install
Maven uploads the ZIP to AEM's Package Manager API at http://localhost:4502/crx/packmgr/service.jsp. AEM reads the vault metadata inside the ZIP and writes the JCR content (XML files) into the repository at the correct paths.
Part 3 — packaging

Bundle vs content package — the key difference

New developers often confuse these two. They are completely different things that work in completely different ways. Understanding the difference tells you immediately where each file type belongs in your project.

OSGi bundle — the Java runtime artifact

An OSGi bundle is a JAR file containing compiled Java code (.class files) and a MANIFEST.MF that tells Felix what this bundle provides and needs. Think of it like a browser extension — you can install it, start it, stop it, and update it without restarting the browser (AEM).

MANIFEST.MF inside an OSGi bundle JAR
Bundle-Name: My Site Core Bundle-SymbolicName: com.mysite.core Bundle-Version: 1.0.0.SNAPSHOT # Packages this bundle makes available to other bundles: Export-Package: com.mysite.core.models,com.mysite.core.services # Packages this bundle needs from other bundles: Import-Package: org.apache.sling.api, org.apache.sling.models.api, com.day.cq.wcm.api, javax.inject # Tell AEM which Java package to scan for Sling Models: Sling-Model-Packages: com.mysite.core.models
Content package — the JCR content artifact

A content package is a ZIP file containing your JCR content nodes serialised as XML using Apache Jackrabbit FileVault format. When installed, AEM reads these XMLs and writes the corresponding nodes into the JCR repository.

mysite.ui.apps-1.0-SNAPSHOT.zip ├── META-INF/vault/ │ ├── filter.xml ← which JCR paths this package manages │ ├── config.xml ← vault configuration │ └── properties.xml ← package metadata └── jcr_root/ └── apps/mysite/ └── components/ └── card/ ├── .content.xml ← component node definition ├── card.html ← HTL file (stored as a JCR node) └── _cq_dialog/ └── .content.xml
The golden rule — one type per module
Goes in core/ → becomes bundle
Goes in ui.apps/ → becomes content package
Java files only: Sling Models, OSGi Services, Sling Servlets, Workflow Processes, Scheduled Jobs
Everything else: HTL templates, dialog XML, component .content.xml, clientlib CSS/JS, i18n translation files
Part 4 — archetype

What is the AEM Maven Archetype

A Maven archetype is a project template. The AEM Maven Archetype is Adobe's official template that generates a complete, correctly configured, production-ready AEM project in under a minute. You never create this structure by hand.

What the archetype generates for you
  • Root pom.xml with all profiles, dependencyManagement, and plugin configs pre-configured correctly
  • core/ module with an example Sling Model, example Servlet, unit test setup, and the correct bnd plugin configuration including Sling-Model-Packages
  • ui.apps/ module with an example HelloWorld component, clientlib structure, and vault filters
  • ui.content/ module with sample pages and editable template starter nodes under /conf
  • ui.config/ module with example OSGi config files organised by runmode
  • ui.frontend/ module with webpack, SCSS, and TypeScript configured
  • all/ module that correctly embeds all other modules for single-package deployment
  • A correct .gitignore file excluding target/ and other generated folders

Without the archetype, an experienced AEM developer would spend 2–3 days setting all of this up correctly from scratch. The archetype does it in 30 seconds.

Archetype version vs AEM version

The archetype has its own version number (currently 48+). Higher version = newer AEM patterns and features. This is separate from your AEM version. Archetype version 48 can generate projects for AEM 6.5 or AEM as Cloud Service — you specify the AEM version with the aemVersion parameter. Always use the latest archetype version for new projects.

Part 4 — archetype

Running the archetype — step by step

The complete process to generate a new AEM project from scratch. Run this in your terminal. AEM does not need to be running to generate the project — only to deploy it.

The command — copy and customise
terminal — run from any folder where you want the project created
mvn -B archetype:generate \ -D archetypeGroupId=com.adobe.aem \ -D archetypeArtifactId=aem-project-archetype \ -D archetypeVersion=48 \ -D appTitle="My Bank Site" \ -D appId="mybank" \ -D groupId="com.mybank" \ -D aemVersion="6.5.0" \ -D frontendModule="general" \ -D includeExamples="y"

The \ at end of each line is a line-continuation character so the command fits on screen. You can also write it all on one line. On Windows, replace \ with ^.

The -B flag means "batch mode" — Maven proceeds without prompting you to confirm parameters interactively.

What happens while it runs
1
Maven downloads the archetype
Maven connects to Maven Central and downloads the archetype JAR. Takes a few seconds on first run. Cached for future use at ~/.m2/repository/com/adobe/aem/.
2
Archetype generates all files
The archetype uses your parameters to generate every file — replacing appId, groupId, and appTitle placeholders throughout all pom.xml files, Java package names, and XML files.
3
Project folder created
A folder named after your appId (e.g. mybank/) is created in your current directory. This is your complete AEM project.
4
Open in VS Code
Run code ./mybank. All modules appear in the VS Code Explorer panel. You are ready to start development.
Part 4 — archetype

Every archetype parameter explained

Every parameter has a specific impact on the generated project. Understanding each one prevents mistakes that require regenerating the project.

appTitle
Human readable name. Used as: site display name in AEM UI, browser page title, package descriptions. Can have spaces and mixed case. Example: "HDFC Bank Digital"
appId
Short identifier. Must be lowercase, no spaces, only hyphens. Used as: JCR folder names (/apps/mybank/), Maven artifact IDs (mybank.core), component group names. Example: "mybank" or "hdfc-digital"
groupId
Maven GroupId and Java package base. Reversed domain convention. Used as: root Java package for all your classes (e.g. com.mybank.core.models.CardModel). Must be lowercase with dots, no hyphens. Example: "com.mybank"
aemVersion
Which AEM version to target. "6.5.0" for AEM 6.5 on-premise (most enterprises). "cloud" for AEM as Cloud Service. Controls which AEM API version is added as a dependency.
frontendModule
"general" — generates a webpack/npm frontend build with SCSS and TypeScript. "none" — skips ui.frontend entirely. Use "general" unless you have a specific reason not to.
includeExamples
"y" — includes example components (HelloWorld, Byline) and sample content pages. "n" — cleaner starting point. Use "y" when learning — the examples show correct patterns to copy from.
version
Your project's starting version. Default "1.0-SNAPSHOT" is fine. Do not overthink this.
singleCountry
"y" — content at /content/mybank/. "n" — content at /content/mybank/us/en/ (multi-country structure). Use "n" for enterprise multi-region projects.

Part 5 — project structure

core/ module — where all Java lives

The core module is the only place in the entire project where Java code belongs. It compiles into an OSGi bundle JAR. Everything here gets compiled by javac and runs inside AEM's Felix runtime.

What goes inside core/src/main/java/
models/
Sling Model interfaces and their implementations (in a sub-package called impl/). These read JCR properties saved by editors and expose them as Java getters for HTL templates to call. Example: CardModel.java (interface) + impl/CardModelImpl.java (implementation with @Model annotation).
services/
OSGi Service interfaces and implementations. Shared business logic used across multiple components — email sending, external API calls, caching, data transformation. Example: EmailService.java + impl/EmailServiceImpl.java.
servlets/
Sling Servlets that handle HTTP requests. AJAX endpoints, form submission handlers, search APIs, data export endpoints. Example: SearchServlet.java registered at /bin/mybank/search.
workflows/
AEM Workflow Process step implementations. Custom logic in content approval workflows — auto-tagging, notification sending, content validation.
schedulers/
Scheduled job implementations. Code that runs on a cron schedule — cache warming, data sync, report generation, cleanup tasks.
What goes in core/src/test/java/

Unit tests for all your Java classes. Use JUnit 5 and AEM Mocks (io.wcm.testing.aem-mock) to test Sling Models without a running AEM instance. Structure mirrors main: a test for models/impl/CardModelImpl.java goes at src/test/java/.../models/impl/CardModelImplTest.java.

core/pom.xml — the complete annotated version
core/pom.xml — complete

ui.apps/ module — component code and clientlibs

ui.apps contains all the code-level JCR content that lives under /apps — component folders with HTL and dialogs, ClientLib nodes with CSS and JS, i18n translation files. This is deployed to /apps/mybank/ in the JCR.

The path structure
ui.apps/src/main/content/jcr_root/apps/mybank/ │ ├── components/ ← one subfolder per component │ ├── card/ │ │ ├── .content.xml ← component definition │ │ ├── card.html ← HTL rendering template │ │ └── _cq_dialog/ │ │ └── .content.xml ← author dialog │ └── page/base/ │ ├── .content.xml │ ├── base.html ← page component │ └── head.html │ ├── clientlibs/ ← one subfolder per library │ └── site/ │ ├── .content.xml ← categories, dependencies, allowProxy │ ├── css.txt ← list of CSS files in order │ ├── js.txt ← list of JS files in order │ ├── css/main.css │ └── js/main.js │ └── i18n/ ← translation strings └── en.json
The vault filter — what this module manages in AEM

Every ui.apps has a META-INF/vault/filter.xml that declares which JCR paths it "owns". When the package installs, AEM manages nodes under these paths. Anything in your project source that maps under a filter path gets written to JCR. Anything in JCR under a filter path that is NOT in your package gets deleted on install.

ui.apps/src/main/content/META-INF/vault/filter.xml
<workspaceFilter version="1.0"> <!-- This package owns everything under /apps/mybank --> <filter root="/apps/mybank"/> </workspaceFilter>
Part 5 — project structure

ui.content/ — page content and configuration

ui.content contains JCR content under /content and /conf — sample pages and editable template configuration. The crucial distinction: the /content (sample pages) is typically not deployed to production; the /conf (templates, policies) is deployed everywhere.

What lives here
/content/mybank/
Sample pages demonstrating the site. Homepage, example articles, example products. Used in development and staging. Do NOT deploy to production — production has real content authored by editors.
/conf/mybank/
Site configuration — editable template structure and initial content, Content Fragment model definitions, site-specific settings. Deploy to ALL environments including production because templates and configs must exist before editors can create content.
Why sample content is separate from code

Because you want different things deployed to different environments:

  • Development: deploy everything including sample pages (need content to test with)
  • Staging: deploy ui.apps + ui.config + the /conf part of ui.content (templates, not sample pages)
  • Production: deploy ui.apps + ui.config + /conf templates only (never overwrite editors' real content)

The separation of code (ui.apps) from content (ui.content) makes this selective deployment possible.

Part 5 — project structure

ui.config/ — OSGi service configurations

ui.config contains .cfg.json files that configure your OSGi services. Different environments need different values — different SMTP servers, API endpoints, and cache settings. This module manages all of it through AEM's runmode folder system.

The folder structure and runmode resolution
ui.config/src/main/content/jcr_root/apps/mybank/osgiconfig/ ├── config/ ← loaded in ALL environments (baseline) │ └── com.mybank.core.services.impl.EmailServiceImpl.cfg.json │ ├── config.author/ ← loaded ONLY on Author instances │ ├── config.publish/ ← loaded ONLY on Publish instances │ ├── config.prod/ ← loaded ONLY in production runmode │ └── config.author.prod/ ← most specific: Author in production only

Resolution priority — most specific wins: config.author.prod beats config.author beats config.prod beats config.

The .cfg.json file — naming and format

The filename must exactly match the fully-qualified class name of the OSGi service implementation:

config/com.mybank.core.services.impl.EmailServiceImpl.cfg.json
{ "smtp.host": "smtp.mybank.com", "smtp.port": 587, "from.address": "no-reply@mybank.com", "enable.tls": true, "allowed.recipients": ["@mybank.com", "@partner.com"] }

Property names must exactly match those in the service's @ObjectClassDefinition interface. JSON value types are automatically converted to the correct Java type (String, int, boolean, String[]).

Part 5 — project structure

ui.frontend/ — SCSS and TypeScript

The ui.frontend module provides a webpack-based build environment for your front-end code. Write modern SCSS and TypeScript and get compiled CSS and JavaScript automatically placed into your ui.apps clientlibs.

What ui.frontend gives you
SCSS compilation

Write CSS with variables, nesting, mixins, and imports. Webpack compiles SCSS to browser-compatible CSS and outputs it to the right clientlib in ui.apps.

TypeScript

Write TypeScript with type safety. Compiled to ES5 JavaScript for browser compatibility. Output goes to the clientlib in ui.apps.

Watch mode

Run npm run watch in ui.frontend for a dev server that proxies to your local AEM and hot-reloads CSS changes in the browser without a page refresh.

Bundling

Webpack combines, minifies, and tree-shakes all CSS and JS for production. Much more powerful than AEM's built-in clientlib minification.

How it connects to ui.apps

When Maven builds ui.frontend, the frontend-maven-plugin runs npm install then npm run build. The webpack output — compiled CSS and JS — is written directly into the clientlib folder inside ui.apps. When ui.apps is then packaged, it includes these compiled assets. The Maven build handles all of this automatically in sequence.

If you are just starting to learn AEM and do not need a complex frontend build, you can skip ui.frontend and write CSS/JS directly in the ui.apps clientlibs folder. ui.frontend is for teams who want a modern frontend developer experience.

The all module produces one single ZIP package that, when installed in AEM, deploys your entire project. It has no source files — it just pulls in all other modules as embedded packages.

Part 6 — VS Code & deploy

What happens internally when you deploy

When you run mvn clean install -PautoInstallPackage, Maven runs a long sequence of steps. Understanding each step helps you understand why things fail and where to look for errors.

1
clean — delete old build output
Deletes the target/ directory in every module. Removes all previously compiled .class files and previously built JARs and ZIPs. Ensures a fresh build with no stale artifacts.
2
compile — core module
Runs javac on all .java files in core/src/main/java/. If any file has a Java syntax error, the build stops here with "COMPILATION ERROR" and shows you the exact file and line number.
3
test — core module
Compiles and runs all JUnit tests in core/src/test/java/. If any test assertion fails, the build stops here. Add -DskipTests to skip this if you are in a hurry.
4
package — core module
bnd-maven-plugin creates the OSGi bundle JAR at core/target/mybank.core-1.0-SNAPSHOT.jar with the correct MANIFEST.MF including the Sling-Model-Packages entry you configured.
5
package — ui.apps, ui.content, ui.config
content-package-maven-plugin zips each module's src/main/content/ folder together with the vault metadata into content package ZIP files at [module]/target/[artifactId]-1.0-SNAPSHOT.zip.
6
package — all module
The all module's pom.xml embeds all other module ZIPs (and the core JAR) into one master ZIP at all/target/mybank.all-1.0-SNAPSHOT.zip. This is the single file that deploys everything.
7
deploy to AEM — autoInstallPackage profile
content-package-maven-plugin uploads mybank.all-1.0-SNAPSHOT.zip to http://localhost:4502/crx/packmgr/service.jsp via HTTP POST. AEM extracts the ZIP, installs each embedded package, and activates the OSGi bundle. Maven prints "BUILD SUCCESS" when complete.

Verifying deployment worked — five checks

After every deployment, verify these five things. Do not assume the build succeeded — check. Each check validates a different part of the deployment.

Check 1 — terminal shows BUILD SUCCESS

Look at the last lines of Maven output. You want to see:

expected output
[INFO] BUILD SUCCESS [INFO] Total time: 42.831 s

If you see BUILD FAILURE, scroll up in the terminal to find the first [ERROR] line. That is where it stopped and why.

Check 2 — bundle is Active in OSGi console
openhttp://localhost:4502/system/console/bundles

Search for your bundle symbolic name (e.g. mybank.core). Status must say "Active" with a green indicator. If it says "Installed" — the bundle failed to start. Click the bundle name to see which imports are unsatisfied.

Check 3 — component files are in CRXDE
openhttp://localhost:4502/crx/de

Navigate to /apps/mybank/components/. Your component folders must be there. Click any component folder and verify that card.html, .content.xml, and _cq_dialog/ are present. If the folder is missing, ui.apps did not deploy correctly.

Check 4 — package installed in Package Manager
openhttp://localhost:4502/crx/packmgr

Find mybank.all package in the list. Status must show "Installed". If it shows "Uploaded" but not installed, click it and click the Install button manually.

Check 5 — component appears in the page editor

Open any page in the AEM page editor. Open the component browser panel (left side). Look for your component group (e.g. "My Bank - Content"). Your component must appear there. If it does not, the componentGroup value in your .content.xml does not match what is allowed in the template's Layout Container policy.

Common errors and exactly how to fix them

These are the errors every AEM developer encounters repeatedly. Here is what each one means and the exact steps to fix it.

Error 1 — BUILD FAILURE: compilation error
terminal
[ERROR] COMPILATION ERROR [ERROR] CardModelImpl.java:[25,5] error: cannot find symbol symbol: class Resource

Java syntax error or missing import. The numbers are [line, column]. Fix: open CardModelImpl.java at line 25. Add the missing import: import org.apache.sling.api.resource.Resource;. Run Maven again.

Error 2 — bundle stuck in "Installed" state

Bundle compiled and deployed but will not start because a Java package it needs is not exported by any other active bundle.

1
Go to /system/console/bundles. Click your bundle name.
2
In the details section, find "Unsatisfied Imports" — these are the Java packages your bundle needs but cannot find.
3
The missing package is usually something you imported in Java but forgot to add as a provided dependency in core/pom.xml. Or you used compile scope for an AEM API causing a ClassLoader conflict.
Error 3 — connection refused
terminal
java.net.ConnectException: Connection refused (Connection refused)

Maven tried to upload to AEM at localhost:4502 but nothing is listening. Fix: AEM is not running. Start it: java -Xmx4096m -jar aem-author-p4502.jar -gui. Wait for the login page to be accessible. Then run Maven again.

Error 4 — Sling Model returns null in HTL

${model.title} renders nothing even though the editor filled in the dialog. HTL silently outputs nothing when the model is null — no error shown.

1
In core/pom.xml, find the bnd plugin config. Verify: Sling-Model-Packages: com.mybank.core.models is present. This is the most common cause — the model package is not declared so AEM never scans it.
2
In your CardModelImpl.java, verify: @Model(resourceType = "mybank/components/card"). The path must exactly match the component folder path relative to /apps/.
3
In card.html, verify: data-sly-use.model="com.mybank.core.models.CardModel" has the exact fully-qualified interface class name.
Error 5 — component not in component browser
1
In CRXDE at /apps/mybank/components/card, check Properties tab — find componentGroup and note its exact value including capitalisation.
2
Open the template in the Template Editor → click the Layout Container → Policy → Allowed Components. Find your group. Is it checked?
3
Most common fix: componentGroup in .content.xml has a different spelling or capitalisation than what is allowed in the policy. They must match exactly character for character.
