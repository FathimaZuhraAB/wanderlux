Part 1 — what & why

What is a Sling Model and why does it exist

A Sling Model is a Java class that automatically reads data from the JCR when a component is rendered. It bridges the content repository and the HTL template. Without Sling Models, there would be no clean way to get JCR data into HTML.

The problem before Sling Models

Before Sling Models (introduced in 2012), AEM developers wrote JSP files. To get data from the JCR, you had to write code like this inside the template:

old JSP way — the pain
// In card.jsp — mixing Java data-access logic inside HTML template <% Resource resource = resourceResolver.getResource(currentNode.getPath()); ValueMap props = resource.adaptTo(ValueMap.class); String title = props != null ? props.get("title", String.class) : ""; String link = props != null ? props.get("link", String.class) : ""; if (link == null) link = ""; // ... 40 more lines of setup code before any HTML %> <h2><%= xssAPI.encodeForHTML(title) %></h2>

This was hard to read, impossible to unit test, mixed concerns badly, and created security vulnerabilities when developers forgot to escape output. Every JSP was different. There was no standard.

What Sling Models provide
Unit testable

A plain Java class. You can test it with JUnit without a running AEM instance using AEM Mocks. The most important improvement over JSP.

Automatic injection

JCR properties, Sling objects, OSGi services — all injected automatically using annotations. No boilerplate setup code.

Clean separation

Business logic stays in Java. Presentation stays in HTL. Templates become simple — just call getters, no logic.

Reusable

Any Java code can call resource.adaptTo(CardModel.class) to get a populated model — not just HTL. Export as JSON, use in servlets, use in other models.

The simplest mental model
A Sling Model is a POJO (Plain Old Java Object) that reads from the JCR when instantiated. You declare what you want with annotations. AEM fills it in automatically. Your HTL template just calls the getters.
Part 1 — what & why

MVC in AEM — where each piece lives

AEM follows the Model-View-Controller pattern. Once you understand where each AEM concept maps to MVC, the whole system makes sense.

The three layers
Model
The Sling Model. Reads data from the JCR (the editor's input), computes derived values, calls services. Knows nothing about HTML. Returns pure data through getters.
View
The HTL template (card.html). Renders data from the Model as HTML. Contains zero business logic — only calls getters and uses the 13 data-sly-* block statements for display logic.
Controller
Apache Sling itself. Receives the HTTP request, resolves the resource, finds the right model and template, passes control to them. Developers never write controller code — Sling handles it automatically.
What belongs where — the rule
Belongs in the Sling Model
Belongs in the HTL template
Reading JCR properties, null checks, string formatting, URL mapping, permission checks, calling OSGi services, computing derived values (e.g. "is this card featured?"), reading child nodes (multifield items)
Looping over lists (data-sly-list), conditional rendering (data-sly-test), outputting strings (${model.title}), choosing elements (data-sly-element), including partials (data-sly-include)
Never put an if statement, a string split, a null check, or a service call inside an HTL template. The template should only call getters. All logic belongs in @PostConstruct.

Full lifecycle — what happens when a component renders

When a browser requests a page, here is the exact sequence from HTTP request to HTML response, showing where the Sling Model fits in.

1
Browser requests /content/mysite/en/home.html
HTTP GET request reaches AEM's Sling request processing layer.
2
Sling resolves the JCR resource
Finds the cq:Page node at /content/mysite/en/home and its jcr:content child.
3
Sling reads sling:resourceType
Reads sling:resourceType = "mysite/components/page/base" from jcr:content. Finds the rendering script at /apps/mysite/components/page/base/base.html.
4
HTL processes data-sly-use
HTL finds data-sly-use.model="com.mysite.core.models.CardModel". Calls request.adaptTo(CardModel.class).
5
Sling Models factory instantiates CardModelImpl
Sling Models scans all @Model classes registered (from Sling-Model-Packages in MANIFEST.MF). Finds CardModelImpl because its resourceType matches the current resource's sling:resourceType. Creates a new instance.
6
All annotated fields are injected
For each @ValueMapValue field: reads the JCR property with that name. For each @SlingObject field: injects the right AEM object. For each @OSGiService field: looks up the OSGi service. All done automatically.
7
@PostConstruct runs
After all fields are injected, AEM calls the method annotated with @PostConstruct. This is where your business logic runs — null checks, derived computations, string formatting. All injected fields are available here.
8
Model is bound to "model" variable in HTL
The populated CardModelImpl instance is now available in the HTL template as the variable name from data-sly-use.model.
9
HTL calls getters
${model.title} calls getTitle(). ${model.hasLink} calls isHasLink(). The return values are escaped and inserted into the HTML output.
10
HTML is returned to browser
The Sling Model instance is garbage collected. No state persists between requests.

@Model annotation — every parameter explained

The @Model annotation on a class registers it as a Sling Model. It has four parameters. Understanding each one precisely prevents the most common Sling Model bugs.

@Model — all four parameters
@Model( // ① adaptables — what this model can be created FROM // Resource.class = can be created from a Sling Resource // SlingHttpServletRequest.class = can be created from an HTTP request // Use both to cover all scenarios (see next topic) adaptables = { Resource.class, SlingHttpServletRequest.class }, // ② adapters — which interfaces this impl provides // Links this implementation to its interface. // Required for resource.adaptTo(CardModel.class) to work. // Without this, adaptTo() returns null. adapters = { CardModel.class }, // ③ resourceType — which component uses this model // Path relative to /apps. Case-sensitive. No leading slash. // When a resource with this resourceType is adapted, // Sling automatically picks this model — no class name needed in HTL. resourceType = "mysite/components/card", // ④ defaultInjectionStrategy — what happens when a field has no value // OPTIONAL = null is injected if the property does not exist (recommended) // REQUIRED = throws MissingElementException if any field has no value defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL )
Required imports
imports needed for @Model

Part 2 — @Model annotation

adaptables — Resource vs SlingHttpServletRequest

The adaptables parameter tells Sling which object this model can be created from. This determines which injectors are available to the model. Getting this wrong silently prevents certain fields from being injected.

Resource.class — the simpler adaptable

When adaptables = Resource.class, the model is created from the JCR resource node. Available injectors:

  • @ValueMapValue — reads JCR properties from the resource's ValueMap
  • @ChildResource — reads child nodes of the resource
  • @SlingObject fields: Resource, ResourceResolver, SlingScriptHelper, BundleContext
  • @OSGiService — OSGi services from the BundleContext

NOT available: @ScriptVariable (needs request context), request-based fields like currentPage, currentStyle.

SlingHttpServletRequest.class — the request adaptable

When adaptables = SlingHttpServletRequest.class, the model is created from the HTTP request. Available injectors: everything from Resource.class PLUS:

  • @ScriptVariable — page manager, current page, current style, XSS API
  • @RequestAttribute — request attributes set by other components
  • Request parameters via request.getParameter()
Using both — the recommended approach
adaptables — use both for maximum flexibility
// RECOMMENDED for most components: adaptables = { Resource.class, SlingHttpServletRequest.class } // This means: // - When adapted from HTL (request context): request-based injection works // - When adapted from other Java code (resource context): resource-based injection works // - The @ScriptVariable fields only work in request context, but // with OPTIONAL strategy they just remain null in resource context // Use Resource.class ONLY when: // - The model is used for multifield items (child resources) // - The model is called from other Java code, never from HTL directly // - The model never needs currentPage or currentStyle
Part 2 — @Model annotation

adapters — linking the impl to the interface

The adapters parameter lists which interfaces (or classes) this implementation satisfies. It is what makes resource.adaptTo(CardModel.class) return your CardModelImpl instance.

adapters — what it does
// Without adapters parameter — adaptTo returns null: @Model(adaptables = Resource.class) public class CardModelImpl { } Resource r = ...; CardModel model = r.adaptTo(CardModel.class); // ← returns null! No adapter registered // WITH adapters parameter — adaptTo works: @Model(adaptables = Resource.class, adapters = { CardModel.class }) public class CardModelImpl implements CardModel { } CardModel model = r.adaptTo(CardModel.class); // ← returns CardModelImpl instance // Multiple adapters — one impl satisfies multiple interfaces: @Model( adaptables = Resource.class, adapters = { CardModel.class, Exportable.class } ) public class CardModelImpl implements CardModel, Exportable { }
The adapters parameter is the connection between the interface and the implementation. Without it, AEM does not know that CardModelImpl can be used as a CardModel. Always include it.
Part 2 — @Model annotation

resourceType — automatic model selection

The resourceType parameter in @Model allows Sling to automatically select the correct model for a resource without needing to specify the fully-qualified class name in HTL.

resourceType — how it removes class names from HTL
// Without resourceType — must specify full class name in HTL: // data-sly-use.model="com.mysite.core.models.CardModel" // With resourceType in @Model: @Model( adaptables = { Resource.class, SlingHttpServletRequest.class }, adapters = { CardModel.class }, resourceType = "mysite/components/card" ) public class CardModelImpl implements CardModel { } // Now in HTL you can write just the interface name: // data-sly-use.model="com.mysite.core.models.CardModel" // Sling finds CardModelImpl because: // 1. The current resource has sling:resourceType = "mysite/components/card" // 2. CardModelImpl has resourceType = "mysite/components/card" in @Model // 3. CardModelImpl's adapters includes CardModel.class // They match → CardModelImpl is used automatically // CRITICAL: resourceType value rules: // ✓ "mysite/components/card" — relative path, no /apps prefix // ✗ "/apps/mysite/components/card" — WRONG: do not include /apps // ✗ "Mysite/Components/Card" — WRONG: case-sensitive, must be lowercase
Part 2 — @Model annotation

defaultInjectionStrategy — OPTIONAL vs REQUIRED

The injection strategy controls what happens when Sling cannot find a value for an annotated field. This is a global setting for the entire model, with per-field overrides available.

OPTIONAL (recommended)
If a JCR property does not exist or an object cannot be injected, the field receives null (or the primitive default like false for boolean). No exception is thrown. Your @PostConstruct must handle nulls. This is the right choice for almost all components.
REQUIRED
If any field cannot be injected, Sling throws org.apache.sling.models.factory.MissingElementException and the model cannot be created. The component renders nothing. Use only when every field is truly mandatory for the component to make sense.
Per-field strategy overrides
per-field strategy overrides
// Model defaults to OPTIONAL — but specific fields can be required: @Model(..., defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL) public class CardModelImpl { @ValueMapValue private String title; // optional: null if not set @ValueMapValue @Required // THIS field is required even though model is OPTIONAL private String productCode; // exception thrown if not present @OSGiService @Optional // THIS service is optional even if model defaults to REQUIRED private AnalyticsService analytics; // null if service not available }
Part 3 — all injectors

@ValueMapValue — reading JCR properties

@ValueMapValue is the most-used injector. It reads a JCR property from the component's content node and injects it into a Java field. The field name maps directly to the JCR property name.

@ValueMapValue — complete reference
@Model(adaptables = Resource.class, ..., defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL) public class CardModelImpl { // ── Basic usage — field name = JCR property name ────────────── @ValueMapValue private String title; // Reads the "title" JCR property. If title="Summer Sale" in JCR → title = "Summer Sale" // If property does not exist → title = null (with OPTIONAL strategy) @ValueMapValue private String description; @ValueMapValue private String ctaLink; // ── Supported types — AEM auto-converts ────────────────────── @ValueMapValue private boolean openInNewTab; // reads Boolean JCR property @ValueMapValue private long viewCount; // reads Long JCR property @ValueMapValue private double price; // reads Double JCR property @ValueMapValue private String[] tags; // reads String[] (multi-value) JCR property @ValueMapValue private Calendar publishDate; // reads Date JCR property as Calendar // ── Override the property name to read ─────────────────────── @ValueMapValue(name = "jcr:title") private String pageTitle; // Field is named "pageTitle" but reads the "jcr:title" JCR property. // Required when the JCR property name has special characters // (colons: jcr:title, cq:tags) or is a Java reserved word. @ValueMapValue(name = "fileReference") private String imageRef; // field "imageRef" reads JCR property "fileReference" // ── Inject via ResourceResolver for multi-value ───────────── @ValueMapValue(name = "cq:tags") private String[] cqTags; // reads the cq:tags array — multiple tag paths as strings }
How the field name maps to the dialog

In the dialog XML, each field has name="./title". The ./ prefix means "relative to this component's node". So name="./title" saves to a JCR property called title. Your @ValueMapValue private String title; reads that same property. The field name in Java and the last part of the dialog name attribute MUST match exactly.

@Default — fallback values when properties are not set

@Default combined with @ValueMapValue provides a fallback value when the JCR property does not exist. Avoids null checks in @PostConstruct for simple defaults.

@Default — all type variants
// String default @ValueMapValue @Default(values = "Learn More") private String ctaText; // If "ctaText" JCR property is missing: ctaText = "Learn More" // If "ctaText" is present in JCR: uses that value // Boolean default @ValueMapValue @Default(booleanValues = false) private boolean openInNewTab; // Long default @ValueMapValue @Default(longValues = 0L) private long itemCount; // Double default @ValueMapValue @Default(doubleValues = 0.0) private double price; // String array default @ValueMapValue @Default(values = { "technology", "general" }) private String[] categories; // When to use @Default vs handling in @PostConstruct: // Use @Default when: the fallback is a simple constant value // Use @PostConstruct when: the fallback is computed from other values // Example — computed fallback in @PostConstruct: @PostConstruct protected void init() { // Can't use @Default here because the fallback depends on other fields: if (StringUtils.isBlank(ctaText)) { ctaText = StringUtils.isNotBlank(title) ? "Read about " + title : "Read More"; } }

@ChildResource — injecting child JCR nodes

@ChildResource injects a direct child node of the current resource. The field name maps to the child node name. Use it for structural sub-components or as the starting point for reading multifield items.

@ChildResource — all usage patterns
// ── Pattern 1: Inject a child node as Resource ───────────────── @ChildResource private Resource image; // Injects the child JCR node named "image" as a Resource object. // Use when you need to pass the resource to another model or // read its properties manually. // ── Pattern 2: Inject a child node as another Sling Model ────── @ChildResource private ImageModel image; // If ImageModel is a Sling Model, AEM automatically adapts the // "image" child resource to ImageModel. Very clean pattern. // ── Pattern 3: Override the child node name ──────────────────── @ChildResource(name = "jcr:content") private Resource pageContent; // Field is named "pageContent" but reads child node "jcr:content" // ── Pattern 4: Container for multifield items ────────────────── @ChildResource private Resource items; // Injects the "items" container node. // Then in @PostConstruct, iterate over items.getChildren() // to read each multifield item. // ── Manual iteration pattern (for multifield) ───────────────── @SlingObject private Resource currentResource; private List<TeamMember> members = new ArrayList<>(); @PostConstruct protected void init() { Resource teamContainer = currentResource.getChild("team"); if (teamContainer != null) { for (Resource child : teamContainer.getChildren()) { TeamMember m = child.adaptTo(TeamMember.class); if (m != null) members.add(m); } } }

Part 3 — all injectors

@Inject — the generic injector (avoid in new code)

@Inject is the original, generic injector from before the specific injectors existed. It tries all available injectors in order until one works. Avoid it in new code — use the specific injectors instead.

@Inject — what it does and why to avoid it
// OLD pattern (before specific injectors existed): @Inject private String title; // tries all injectors until one works @Inject private Resource currentResource; @Inject private Page currentPage; // PROBLEM: @Inject is ambiguous — hard to know which injector won. // It is also slightly slower because it tries all injectors. // And it is harder to read — other developers cannot tell at a glance // where each value comes from. // MODERN equivalent — explicit and clear: @ValueMapValue private String title; // clearly: from JCR @SlingObject private Resource currentResource; // clearly: from Sling @ScriptVariable private Page currentPage; // clearly: from script context // Rule: never use @Inject in new code. // If you see it in existing code, it works but is a code smell. // Replace it with specific injectors during refactoring.
Part 4 — lifecycle methods

@PostConstruct — where your logic lives

@PostConstruct marks a method that AEM calls after all injected fields have been populated. This is the correct place for ALL your business logic. It is called exactly once per model instance.

@PostConstruct — complete patterns
@PostConstruct protected void init() { // ── At this point, ALL injected fields have their values ────── // @ValueMapValue fields: have JCR values (or null if OPTIONAL) // @SlingObject fields: have their AEM objects // @ScriptVariable fields: have their scripting objects // @OSGiService fields: have service references (or null if @Optional) // ── Null safety — always check @ValueMapValue fields first ──── if (StringUtils.isBlank(title)) { title = ""; // normalise: never let title be null in getters } // ── Computed booleans ───────────────────────────────────────── hasLink = StringUtils.isNotBlank(ctaLink); hasDescription = StringUtils.isNotBlank(description); hasTags = tags != null && tags.length > 0; // ── Derived string values ───────────────────────────────────── themeClass = StringUtils.isNotBlank(theme) ? "card--" + theme.trim().toLowerCase() : ""; displayTitle = StringUtils.defaultIfBlank(title, currentPage != null ? currentPage.getTitle() : "Untitled"); // ── URL processing ──────────────────────────────────────────── if (hasLink && ctaLink.startsWith("/content")) { // Append .html to internal AEM paths processedLink = ctaLink + (".html".equals( ctaLink.substring(ctaLink.lastIndexOf('.') + 1)) ? "" : ".html"); } else { processedLink = ctaLink; } // ── Reading multifield items ────────────────────────────────── Resource itemsNode = currentResource.getChild("items"); if (itemsNode != null) { for (Resource child : itemsNode.getChildren()) { FaqItem item = child.adaptTo(FaqItem.class); if (item != null) faqItems.add(item); } } } // ── Method name can be anything — "init" is just convention ─── // The annotation, not the name, tells Sling to call it. // You can have only ONE @PostConstruct per class. // It must be non-static, return void, and have no parameters.
Do NOT do anything in the constructor. The constructor runs before injection. All injected fields are null when the constructor runs. All logic must be in @PostConstruct.

@PreDestroy — cleanup after rendering

@PreDestroy marks a method called just before the model instance is garbage collected. Use it to close any resources you opened in @PostConstruct.

@PreDestroy — when and how to use it
// @PreDestroy is rarely needed. Most models use injected services // which manage their own resources. Use it only when YOU open something. // Example — model that opens a JCR session directly: private Session jcrSession; @PostConstruct protected void init() { // Sometimes you need a privileged JCR session for special operations: try { jcrSession = resourceResolver.adaptTo(Session.class); // ... use session for queries } catch (Exception e) { log.error("Failed to get JCR session", e); } } @PreDestroy protected void cleanup() { if (jcrSession != null && jcrSession.isLive()) { jcrSession.logout(); // always close what you open } } // NOTE: In modern AEM development, most operations go through // the injected ResourceResolver (which is managed by Sling). // Directly opening JCR sessions is rarely needed. // If you do not open resources yourself, do not write @PreDestroy.
Part 4 — lifecycle methods

Lifecycle order — exact sequence

Here is the complete, precise sequence of every step from HTL requesting a model to the model being garbage collected.

1
HTL evaluates data-sly-use
HTL processes data-sly-use.model="com.mysite.core.models.CardModel". Calls request.adaptTo(CardModel.class).
2
Sling Models factory scans registered models
Finds CardModelImpl because: its adapters includes CardModel.class, and its resourceType matches the current resource's sling:resourceType.
3
Constructor runs
Java creates a new CardModelImpl instance. The constructor runs. All fields are null at this point. Do NOT access injected fields here.
4
@ValueMapValue fields are injected
For each @ValueMapValue field, Sling reads the corresponding JCR property from the resource's ValueMap and sets the field. Type coercion happens automatically.
5
@SlingObject fields are injected
AEM runtime objects (Resource, ResourceResolver, Request) are injected.
6
@ScriptVariable fields are injected
Script-context objects (Page, PageManager, Style) are injected from the request context.
7
@ChildResource fields are injected
Named child nodes are resolved and injected as Resource objects or adapted to their own Sling Models.
8
@OSGiService fields are injected
OSGi services are looked up in the bundle context and injected.
9
@PostConstruct method runs
AEM calls your init() method. All injected fields are available. This is where your business logic runs.
10
Model returned to HTL
The populated model instance is bound to the variable name in HTL (e.g. model).
11
HTL calls getters during rendering
${model.title} calls getTitle(). Returns the pre-computed value. Getters should be fast — no logic, just return the field.
12
@PreDestroy runs, model garbage collected
After the request completes, @PreDestroy runs (if defined). The instance is then eligible for garbage collection. Sling Models are request-scoped — a new instance per request.
Part 5 — interface & impl pattern

Why interface + implementation — the pattern explained

Every Sling Model should have two Java files: an interface and an implementation class. This is not just a Java convention — it has specific benefits in AEM that make production code significantly better.

The four reasons
Unit testable

In tests, you can swap the real CardModelImpl for a mock that implements CardModel. Other models that depend on CardModel can be tested in isolation without the real implementation.

HTL stays clean

HTL uses the interface class name: data-sly-use.model="com.mysite.core.models.CardModel". The interface has no annotations, no imports, no AEM dependencies — it is a pure Java contract.

Swappable impls

You can provide a second implementation (e.g. for an A/B test) by creating another class with adapters = { CardModel.class }. Switch between them with OSGi service ranking — no HTL changes needed.

Clear contract

The interface declares exactly what data the component exposes. Other developers reading the interface immediately know what data is available without reading the implementation details.

Part 5 — interface & impl pattern

The interface — what goes in it

The interface is a pure Java contract. It declares only getter method signatures. No AEM imports, no annotations, no field declarations, no logic. Keep it minimal and readable.

core/src/main/java/com/mysite/core/models/CardModel.java
package com.mysite.core.models; import java.util.List; /** * Sling Model interface for the Card component. * Exposes data for the card.html HTL template. * All methods are getters — no logic, no AEM imports. */ public interface CardModel { /** The main headline of the card. Never null — returns empty string if not set. */ String getTitle(); /** The optional eyebrow label above the title. Null if not configured. */ String getEyebrow(); /** The body description. May contain HTML from the RTE. */ String getDescription(); /** The heading element level (h2, h3, h4) from the design dialog policy. */ String getHeadingLevel(); /** The DAM path of the card image. Null if not set. */ String getImageUrl(); /** The alt text for the image. Falls back to title if not explicitly set. */ String getImageAlt(); /** The target URL for the card's CTA button. Null if not set. */ String getCtaLink(); /** The label for the CTA button. Defaults to "Read More" if not set. */ String getCtaText(); /** True if the CTA link is configured and non-empty. */ boolean isHasLink(); /** True if the link should open in a new browser tab. */ boolean isOpenInNewTab(); /** The CSS modifier class for the selected theme (e.g. "card--dark"). */ String getThemeClass(); /** List of string tags applied to this card. Interface rules
  • Only getter method signatures — no implementations
  • No @Model, @ValueMapValue, or any other Sling/AEM annotations
  • No AEM imports except for return types (e.g. List, String, primitive types)
  • Write Javadoc on every method — describe what it returns and any defaults
  • Keep it in the models/ package, not models/impl/

Part 6 — HTL integration

Getter naming convention — HTL to Java

HTL converts property names in expressions to Java getter method names following the JavaBeans convention. Getting this wrong means the expression silently returns null.

${model.title}
Calls getTitle(). Rule: capitalise first letter, prepend get.
${model.firstName}
Calls getFirstName(). CamelCase stays as-is.
${model.hasLink}
Calls isHasLink() first, then getHasLink() as fallback. Boolean getters use is prefix.
${model.empty}
Calls isEmpty() first. AEM convention: booleans like empty use is.
${model.items}
Calls getItems(). Returns List<?> which HTL can loop over with data-sly-list.
${model.items.size}
Calls getItems().size(). HTL supports chained calls for collection metadata.
${model.ctaLink}
Calls getCtaLink(). Multi-word camelCase: each word capitalised.
naming table — complete reference
// HTL expression → Java method called ${model.title} → getTitle() ${model.ctaText} → getCtaText() ${model.imageUrl} → getImageUrl() ${model.hasLink} → isHasLink() (boolean: "is" prefix) ${model.isEmpty} → isEmpty() (boolean: "is" prefix) ${model.isActive} → isActive() (or getIsActive() as fallback) ${model.openInNewTab}→ isOpenInNewTab() (boolean: "is" prefix) ${model.empty} → isEmpty() (special: "empty" → "isEmpty") ${model.items} → getItems() (returns List/Collection/array) ${model.items.size} → getItems().size() // If HTL cannot find getXxx() or isXxx(), expression returns null // — no error, no warning, just empty output

Part 8 — testing & errors

Common errors — every one with an exact fix

These are the errors every AEM developer hits with Sling Models. Every one has a specific cause and a specific fix.

Error 1 — data-sly-use returns null (model is null)

${model.title} renders nothing. Adding ${model.class.name} to the template also renders nothing.

1
Check Sling-Model-Packages in bnd config
In core/pom.xml, find the bnd plugin. Verify Sling-Model-Packages: com.mysite.core.models is present. This is the cause in 60% of cases.
2
Verify the bundle is Active
Go to /system/console/bundles. Find your bundle (e.g. mysite.core). Status must be "Active". If "Installed" — the bundle failed to start. Fix the missing imports first.
3
Check resourceType in @Model
@Model(resourceType = "mysite/components/card") — must exactly match sling:resourceType on the content node. Verify in CRXDE: navigate to the component node, check Properties → sling:resourceType. No /apps prefix. Case-sensitive.
4
Check adapters
@Model(adapters = { CardModel.class }) must include the interface used in data-sly-use. If you changed the interface name, update this parameter.
Error 2 — @ValueMapValue field is always null

Model is not null but ${model.title} is always empty even after saving in the dialog.

1
Check CRXDE first
Navigate to the component's content node. Click Properties. Is the title property there with a value? If not, the dialog did not save it. Check the dialog XML — the field's name attribute.
2
Check field name matches JCR property name
Dialog field name="./title" saves as JCR property title. Java field @ValueMapValue private String title; reads property title. They must match exactly. Case-sensitive.
3
Check for special property names
If JCR property is jcr:title (has colon), use @ValueMapValue(name="jcr:title") — you cannot name a Java field jcr:title.
Error 3 — @ScriptVariable fields are null

currentPage or currentStyle is always null causing a NullPointerException in @PostConstruct.

Cause: @ScriptVariable only works when the model is adapted from a SlingHttpServletRequest. If adaptables only contains Resource.class, script variables cannot be injected.

Fix: Add SlingHttpServletRequest.class to adaptables. Also guard with null checks in @PostConstruct: if (currentStyle != null) { ... }

Error 4 — NullPointerException in @PostConstruct

Model throws NPE during rendering. Stack trace points to your init() method.

Cause: You are calling a method on an injected field that is null because the editor has not filled in that dialog field yet, or the JCR property does not exist.

Fix: Always check for null before using injected fields. With DefaultInjectionStrategy.OPTIONAL, any field can be null if the JCR property is not set. Use StringUtils.isNotBlank(title) before using title, if (currentPage != null) before calling page methods, etc.

Error 5 — wrong getter name means null in HTL

${model.ctaLink} renders nothing but getCtaLink() clearly returns a value in the debugger.

Check the getter name convention. ${model.ctaLink} calls getCtaLink(). If your method is named getCTALink() (uppercase CTA) — that's a mismatch. HTL capitalises the first letter of the property name and prepends get. The second letter stays lowercase if it was lowercase in the expression. ctaLink → getCtaLink(). Use standard Java camelCase.
