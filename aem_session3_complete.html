Part 1 — authoring

What is page authoring

Page authoring is the process of building web pages inside the AEM editor — adding components, configuring their content, arranging layout, and publishing. It requires zero technical knowledge. A marketing manager can do everything a developer would do in a page builder tool.

Who does authoring and why it matters to developers

Content editors — marketing managers, copywriters, product managers — do the authoring. They are not developers. They do not know HTML, CSS, or Java. They need a tool that is as simple as Microsoft Word but produces professional web pages.

As a developer, you are not the one who builds pages. You are the one who builds the tools editors use to build pages — the components, the templates, the dialog forms. Your job is to make authoring easy and safe. When a component's dialog is confusing or a template is too restrictive, editors suffer. When your work is good, editors can build any page they need without calling you.

What authoring produces in the JCR

Every action an editor takes in the authoring interface writes data into the JCR repository. When they drag a component onto the page, a new JCR node is created. When they fill in a dialog and save, the field values are written as JCR properties on that node. When they publish, those nodes are replicated to the Publish server.

Real example
A content editor at Swiggy needs to create a "Diwali 2024" campaign page. She opens the Sites console, navigates to the Campaigns section, creates a new page using the "Campaign Page" template, and opens the editor. She drags a Hero Banner component onto the page, clicks the wrench icon, fills in the headline "Diwali Dhamaka Offers", uploads the Diwali-themed image, and sets the CTA button text to "Order Now". She saves, switches to Preview mode to check, then clicks Publish. The page is live in under 10 minutes. No developer involved.
Part 1 — authoring

Creating a page — exact steps

Here is the precise, step-by-step process to create a new page in AEM from start to finish. Every click and every field is explained.

Step 1 — open the Sites console
URLhttp://localhost:4502/sites.html/content

After logging in, this is the main screen. You see a tree or card view of all your pages. You need to navigate to the parent folder where you want to create the new page.

Step 2 — navigate to the right folder

Click into your site folder. For example, if your site is at /content/mysite/en/, click through the folders until you are in the section where the new page should live. The URL in the Sites console changes as you navigate — it reflects the JCR path.

Step 3 — click Create

Click the blue "Create" button in the top-right toolbar. A dropdown appears. Select "Page". This opens the page creation wizard.

Step 4 — the template picker

The first screen of the wizard shows all templates that are allowed in this location. Templates are shown as cards with a thumbnail and title. The allowed templates are controlled by the template's allowedPaths property and the folder's cq:allowedTemplates property.

Select the appropriate template for this page type — for example "Article Page", "Campaign Page", or "Product Page". Click "Next".

Step 5 — fill in the page properties

The second screen has required fields:

Title
The page's display title. Shown in browser tab, page header, navigation, and search results. Example: "Home Loans 2024". Required.
Name
The URL slug — the last part of the page's URL. Auto-generated from the Title but you can change it. Must be URL-safe: lowercase, hyphens instead of spaces, no special characters. Example: "home-loans-2024". If Title is "Home Loans 2024", Name auto-fills as "home-loans-2024".
Tags
Optional. Categorise the page with taxonomy tags for filtering and search.
Hide in Navigation
Optional checkbox. If checked, this page won't appear in automatically-generated navigation components.

Click "Create" to create the page.

Step 6 — what AEM creates in the JCR

AEM creates a new cq:Page node at the path /content/mysite/en/home-loans-2024 with a jcr:content child node. The jcr:content node gets the Title from your wizard input, a reference to the selected template, and the initial content from the template's "Initial Content" layer pre-populated.

Step 7 — click Open to go to the editor

After creation, AEM shows a "Page Created" confirmation with two options: "Open" (go to the editor now) and "Done" (stay in the Sites console). Click "Open" to immediately start editing your new page.

What the Name field does to your URL
If your site is at /content/mysite/en/ and you create a page with Name "summer-sale", the page lives at JCR path /content/mysite/en/summer-sale. AEM renders it at the URL http://localhost:4502/content/mysite/en/summer-sale.html. On the live site through the Dispatcher, it might appear as https://www.mysite.com/en/summer-sale.html.

Edit, Layout, and Preview modes — deep explanation

The page editor has three modes that completely change what you see and what you can do. You switch between them using the dropdown in the top toolbar.

Edit mode — the content creation mode

Edit mode is the default. Every component has a blue selection handle. Clicking a component shows its edit toolbar. This is the mode where all content changes happen:

  • Drag components from the left panel onto the page
  • Click a text component and double-click to activate inline text editing
  • Click the wrench icon on any component to open its configure dialog
  • Drag components to reorder them within a container
  • Select a component and press Delete to remove it

In Edit mode, the blue outlines are visible even on non-selected components. This helps editors see the page structure clearly — which areas are separate components, which components are inside which containers.

Layout mode — the responsive grid editor

Layout mode activates the 12-column grid system. You use it to control how wide each component is at different screen sizes. Switch to it by clicking "Layout" in the mode dropdown.

In Layout mode:

  • Each component shows a blue resize handle on its right edge
  • Drag the handle left or right to change how many columns the component spans (1 to 12)
  • A column counter shows the current span as you drag
  • The breakpoint selector in the toolbar switches between Desktop (default), Tablet, and Mobile views
  • You can set completely different column spans for each breakpoint

For example: a Text component and an Image component sitting side by side. On desktop you might set Text to 8 columns and Image to 4 columns. On mobile you set both to 12 columns so they stack vertically. This is controlled entirely in Layout mode, no CSS needed.

Preview mode — see exactly what visitors see

Preview mode removes all authoring overlays. No blue outlines, no edit toolbars, no left panel. The page looks and behaves exactly as it will on the live website. Links are clickable, animations play, carousels slide.

Use Preview mode to:

  • Verify the page looks right before publishing
  • Test hover effects and interactive elements
  • Check that all links go to the right destinations
  • Simulate mobile by combining Preview mode with the Emulator
  • Show the page to a stakeholder for review without the authoring chrome
Preview mode is rendered by the Author server using your real CSS and templates. In rare cases the live site through the Dispatcher may look slightly different due to Dispatcher cache configurations. Always check the actual Publish URL for final sign-off.
Timewarp mode — see the past

A fourth mode available via the page info menu. Timewarp lets you set a date and time to see what the page looked like at that point in history. Useful for audits and debugging when content was changed accidentally. AEM stores all published versions with timestamps.

For example, if the editor fills in "Summer Sale 2024" in the Title field (which has name="./title" in the dialog XML), AEM creates a property title = "Summer Sale 2024" on the JCR node at /content/mysite/en/home/jcr:content/root/main/card.

You can see this immediately in CRXDE by navigating to that path and clicking the Properties tab.

Publishing pages — everything about it

Publishing is the act of making a page live — copying it from the Author server to the Publish server so real users can see it. Until a page is published, no visitor can see it regardless of how complete it looks in the editor.

Three ways to publish a page
From the page editor
Click the Page Info icon (top-left of editor) → "Publish Page". This publishes the current page immediately. A green notification confirms success.
From the Sites console
Select one or more pages (checkboxes) → click "Manage Publication" in the top toolbar → "Publish" → choose "Now" (immediate) or a scheduled future date/time → Next → Publish.
Quick Publish
From Sites console, hover over a page → three-dots menu → "Quick Publish". Fastest method, no wizard. No option to schedule or include references.
Publishing with references

When you publish a page, you often also need to publish the assets it references — images from the DAM, Content Fragments, Experience Fragments. AEM's "Manage Publication" wizard has a "Include referenced items" option that detects all referenced content and publishes everything in one go.

Scheduled publishing

In "Manage Publication", instead of "Now", select "Later". Set a specific date and time. AEM creates a replication job scheduled for that time. At the exact moment, AEM publishes the page automatically. This is how editors schedule content for specific launch dates — a product launch at midnight, a news article at 6:00 AM, a sale banner on sale day.

Page publication states
Not published
Grey indicator. Page exists on Author only. Visitors cannot see it.
Published
Green indicator. Page is live on Publish. Visitors can see the current version.
Modified
Yellow indicator. Page was published before, but has been edited since. The live version is the old version. Publish again to push the changes live.
Deactivated
Red indicator. Page was published but has been unpublished (deactivated). It existed on Publish but has been removed. Visitors now get a 404.
Real example — scheduled bank rate update
The Reserve Bank of India announces a rate change effective next Monday. HDFC Bank's content team updates all 47 loan pages on Author on Thursday. They set Scheduled Publication for Monday 00:01 AM. On Monday at 00:01, AEM automatically publishes all 47 pages simultaneously. The website shows the correct rate the moment it is officially effective. Zero manual work on Monday.
Part 2 — templates

What is a template — from scratch

A template is the blueprint for a page. When an editor creates a new page, they pick a template. The template determines: the page's initial structure (which components are already on it), which components editors are allowed to add, and what the page looks like before any content is added.

The restaurant menu analogy

Think of a template as a restaurant menu format. Every menu at a restaurant follows the same structure — starter section, main course section, dessert section. The format is fixed. But the actual food items inside each section are different for every restaurant and change every day. The template is the format. The page content is the food items.

An "Article Page" template always has a hero image at the top, a title, a body text area, and a related articles section at the bottom. Every article page has this structure. But each article has different photos, different headlines, different body text. The template enforces consistency. The editors provide variety.

What a template controls
Structure (locked)

Components that are always on every page made from this template and cannot be moved or deleted by editors. Used for global header, global footer, required legal disclaimers.

Initial content

Components that are pre-added to new pages but CAN be edited or deleted by editors. A starting point, not a lock. Saves editors from starting completely blank.

Allowed components

Which components editors can add to each container area. Enforced by policies. If a component is not in the allowed list, editors cannot add it no matter how hard they try.

Page component

Which Java/HTL component renders the outer HTML shell of every page made from this template — the doctype, head, header, footer, scripts.

Why templates matter for developers

As a developer, you:

  • Create the Page Component that the template references
  • Create the components that editors can add to pages
  • Set up the initial template structure (or the template author does this in the UI)
  • Configure policies to allow the right components

Once you set up the template correctly, a template author can modify it any time through the AEM UI without any code changes from you.

Part 2 — templates

Static templates — the legacy approach

Static templates are the original AEM template system (used before AEM 6.2). You will encounter them in older projects. Understanding them helps you recognise legacy code and understand why editable templates were invented.

How static templates work

A static template is a JCR node at /apps/mysite/templates/article-page with jcr:primaryType = cq:Template. A developer defines the template's properties in a .content.xml file in the project source. The template node has:

jcr:title
The name shown in the template picker when creating a page.
allowedPaths
A regex pattern of JCR paths where this template can be used. Example: /content/mysite(/.*)? means anywhere under /content/mysite/.
allowedChildren
Which templates can be used for child pages of pages made from this template.
ranking
The order this template appears in the picker. Lower number = shown first.
thumbnail
Path to an image shown in the template picker to preview the layout.
The problems with static templates
Static templates (legacy)
Why this is a problem
Template structure is hardcoded in XML files. Changing the layout requires a developer to edit code and deploy.
Marketing needs a new page layout urgently. They have to wait for a developer sprint to get it. Days of delay.
Allowed components are set via Design Mode — a developer or admin visits the page with ?wcmmode=design and clicks to enable components.
This is a separate workflow disconnected from template management. Easy to forget. Cannot be version-controlled properly.
All pages using a template share the exact same structure. No flexibility within the template for different page variations.
Need a slightly different article layout for opinion pieces? Need a separate template. Template proliferation over time.

Template types — the parent of every editable template

Every editable template is created from a "Template Type". A Template Type is a skeleton template (stored under /libs/ or /conf/) that defines the technical structure a template must have. Think of it like a class in object-oriented programming — the Template is an instance of the Template Type.

For AEM Sites, the standard Template Type is "Empty Page" — it creates a completely blank template with the correct structural nodes. AEM provides several built-in Template Types and you can create your own.

Who manages editable templates

Editable templates are managed by "Template Authors" — a user role in AEM with permissions to edit templates. Template authors are typically senior content editors or UX leads, not developers. They understand what page structures are needed and configure them visually.

Developers set up the initial Template Type and the Page Component. Template authors do everything else.

Real example — a template author's workflow
The UX lead at a media company decides they need a new "Video Article" template — like a regular article but with a full-width video embed at the top instead of a static hero image. She opens the template editor, duplicates the existing "Article" template, renames it "Video Article", changes the top structure area to use a Video Player component instead of the Hero Image component, and adjusts the allowed components in the body container. The new template is available to editors within 15 minutes. No developer involved. No code deployed.
Part 2 — templates

Template 3 layers — structure, initial content, policies

Every editable template has three distinct layers. Each layer has a different purpose and is controlled by different people. Understanding these three layers is the most important thing about editable templates.

Layer 1 — Structure (the locked parts)

The Structure layer defines components that are locked on every page made from this template. Editors cannot move, delete, or configure these components. They are typically used for:

  • Global site header (navigation, logo)
  • Global site footer (links, copyright)
  • Mandatory legal disclaimers that cannot be removed
  • The global Layout Container in which editors can add components

When template authors add components to the Structure layer, they can also lock individual component instances — making the configuration fixed (editors cannot change the dialog values either). Or they can unlock them — editors can reconfigure the component per page but cannot delete or move it.

Structure components are stored at /conf/mysite/settings/wcm/templates/article-page/structure/ in the JCR. They are the same JCR nodes as normal page components — just locked with a special property.
Layer 2 — Initial content (the pre-populated parts)

Initial content components appear on every new page created from this template, but editors CAN modify or delete them. They are starting point content, not locked content. Used for:

  • A pre-configured intro text block so editors know where to start
  • A default image placeholder pointing to a generic stock image
  • A standard CTA button pre-configured with the company's primary action

When an editor creates a page from a template with initial content, the initial components are copied to the new page's jcr:content. From that point, the page's copies are completely independent of the template's initial content. Changing initial content does NOT affect pages already created.

Layer 3 — Policies (the rules)

Policies control what editors can and cannot do. They are set per-container (per Layout Container) on the template. Each container in the template can have its own policy defining:

  • Which components are allowed in this container
  • Which style options are available for each component
  • Design dialog settings for each component type

Policies are stored separately from the template itself at /conf/mysite/settings/wcm/policies/. Multiple templates can share the same policy — if you have an "Article Page" and a "News Page" template, both can reference the same "Article Body Content" policy rather than duplicating it.

In the JCR — how the three layers map to nodes
/conf/mysite/settings/wcm/templates/article-page/ ├── structure/ ← Layer 1 — locked components │ └── jcr:content/ │ └── root/ ← the main layout container ├── initial/ ← Layer 2 — pre-populated components │ └── jcr:content/ │ └── root/ ├── policies/ ← Layer 3 — policy references per container └── jcr:content ← template metadata (title, thumbnail, status)

Part 2 — templates

Page component — the HTML document wrapper

Every page in AEM is rendered by a "Page Component". This component generates the complete HTML document — the doctype, head section, body, header, footer, and global scripts. Every page template references one Page Component.

What the Page Component is responsible for
  • The <!DOCTYPE html> declaration
  • The <html lang="en"> tag with the correct language
  • The entire <head> — title tag, meta tags, SEO tags, canonical URLs, Open Graph tags, clientlib CSS includes
  • The global site header (navigation, logo, search)
  • The main <main> area where the page's own components render
  • The global site footer
  • The clientlib JavaScript includes at the bottom of body
  • The AEM context hub script for personalization
  • Analytics and tag manager scripts
Where the Page Component lives
ui.apps/src/main/content/jcr_root/apps/mysite/components/page/base/ ├── .content.xml ← sling:resourceSuperType = core/wcm/components/page/v3/page ├── base.html ← the root HTML document template ├── head.html ← everything inside <head> ├── header.html ← global site header ├── footer.html ← global site footer └── bodyEnd.html ← scripts before </body>
How a template references the Page Component

In the template's JCR structure, the template's structure/jcr:content node has a sling:resourceType property pointing to the Page Component. When AEM renders a page, it first runs the Page Component to produce the outer HTML shell, then renders the page's own components inside the <main> area.

/conf/mysite/settings/wcm/templates/article-page/structure/jcr:content
sling:resourceType = "mysite/components/page/base" ↑ This is the Page Component path in /apps AEM runs base.html to produce the <html><head><body> shell
Extending Core Components' Page component

Your Page Component should not be built from scratch. It should extend Adobe's Core Components Page component via sling:resourceSuperType. This gives you all the SEO tags, multilingual support, accessibility features, and correct script loading for free. You only override the head.html, header.html, and footer.html to add your site-specific content.

page/base/.content.xml

Part 3 — layout container

What is the Layout Container

The Layout Container is the droppable area on a page where editors can drag and drop components. Without a Layout Container, editors cannot add any components to a page. It is the fundamental building block of page construction.

The simplest explanation

Imagine a page as a blank whiteboard. The Layout Container is the frame you draw on that whiteboard that says "you can draw things inside here". Outside the frame, nothing can be drawn. Inside the frame, you can draw anything that is allowed by the policy attached to that frame.

A page can have multiple Layout Containers — one for the main content area, one for a sidebar, one for a hero section. Each container can have its own policy allowing different sets of components and different layout rules.

Layout Container vs Parsys
Parsys (paragraph system — legacy)
Layout Container (modern)
Basic droppable area. No responsive grid. No column resizing. Components just stack vertically. Used in AEM before 6.0.
Advanced droppable area with a 12-column responsive grid built in. Editors can resize components, set different layouts per breakpoint. The standard for all modern AEM projects.
Where the Layout Container component comes from

The Layout Container is actually a Core Component called "Layout Container" or "Responsive Grid". Its sling:resourceType is wcm/foundation/components/responsivegrid (old) or core/wcm/components/container/v1/container (new). When template authors add a Layout Container to a template's structure, they are adding this Core Component. Developers do not need to build it — it is provided by AEM/Core Components.

Nested Layout Containers

Layout Containers can be nested — a Layout Container inside another Layout Container. This is how you create complex multi-column layouts. For example: a full-width outer container holds a two-column inner container, where one column has a text+image stack and the other has a sidebar component.

Part 3 — layout container

Parsys — the old droppable system

Parsys stands for "paragraph system". It is the original AEM component that creates a droppable area for editors. You will find it in legacy projects from before AEM 6.0. Modern projects use Layout Container instead, but understanding Parsys helps when working with older codebases.

How Parsys worked

A developer added a Parsys to a page component's JSP/HTL file with the tag <cq:include path="par" resourceType="foundation/components/parsys"/>. This created a droppable zone at the path par (or whatever name was given). Editors could drag and drop components into it.

The components in a Parsys were stored as child nodes of the Parsys node in the JCR — just like Layout Container stores its child components. The fundamental concept is the same: a container node that holds child component nodes.

What Parsys lacked
  • No responsive grid — components could not be resized or placed side by side. Everything was always full-width stacked vertically.
  • No policy system — allowed components were managed through Design Mode, a separate and inconvenient mechanism.
  • No template author tooling — structure was hardcoded in JSP files by developers.

These limitations are exactly what the Layout Container and editable templates solved.

If you see Parsys in a project

If you are working on an older AEM project and see foundation/components/parsys or wcm/foundation/components/parsys in the code, this is a legacy Parsys. The project predates AEM 6.2's editable templates and Layout Container. The functionality still works but is considered outdated. New development on such a project should still use editable templates and Layout Container where possible.

Part 3 — layout container

The 12-column responsive grid — complete explanation

The Layout Container uses a 12-column grid system identical to Bootstrap's grid. Every component inside a container can be set to span 1 to 12 columns. This is how editors create multi-column layouts without any CSS knowledge.

Why 12 columns?

12 is divisible by 1, 2, 3, 4, 6, and 12. This means you can create equal columns of almost any count: 2 columns (6+6), 3 columns (4+4+4), 4 columns (3+3+3+3), a 2/3 + 1/3 split (8+4), a 3/4 + 1/4 split (9+3). Almost any layout ratio is achievable without fractional columns.

Column span examples
12 columns
Full width of the container. Default for most components. A full-width hero banner.
6 + 6
Two equal columns. A text block on the left, an image on the right. Equal weight.
8 + 4
Main content area (wider) + sidebar (narrower). Classic article layout.
4 + 4 + 4
Three equal columns. Three feature cards side by side. Product comparison.
3 + 3 + 3 + 3
Four equal columns. Four team member cards. Four category tiles.
9 + 3
Wide main content with narrow sidebar. 75%/25% split.
Responsive breakpoints

Each component can have a different column span at each breakpoint. AEM uses three breakpoints:

Desktop (default)
Above 1024px. Set in Layout mode without selecting a specific breakpoint.
Tablet
768px – 1024px. Select the tablet icon in Layout mode's breakpoint selector.
Mobile
Below 768px. Select the mobile icon.

A typical pattern: three product cards at 4+4+4 on desktop (three columns side by side), 6+6+12 on tablet (two side by side, one below), 12+12+12 on mobile (all stacked vertically).

Real example
An editor building a credit card comparison page needs four cards side by side on desktop (4+4+4+4 = 3 columns each), two per row on tablet (6+6 = two columns each), and stacked on mobile (12 each). She selects all four card components, switches to Layout mode, drags each to 3 columns on Desktop breakpoint, then selects Tablet and adjusts the first two to 6 columns (the second two automatically drop to the next row at 6 columns each), then selects Mobile and sets each to 12. Done — fully responsive layout with zero CSS.

Part 4 — policies

What is a policy — from scratch

A policy is a set of rules attached to a component in a template that controls what editors can do with that component on pages. Policies enforce consistency — brand guidelines, technical constraints, and UX standards — without relying on editors to remember the rules.

The security guard analogy

Think of a policy as a security guard at the door of a Layout Container. The security guard has a guest list. Only people on the guest list are allowed inside. Editors are the guests. Components are the people trying to get in. If a component is not on the policy's allowed list, the editor literally cannot see it in the component browser when they are inside that container.

This is not a gentle suggestion — it is a hard technical enforcement. Even if an editor knows the component exists, the AEM UI will not let them add it to a container whose policy does not allow it.

Who creates and manages policies
Template authors
Create and edit policies through the template editor in the AEM UI. They are typically senior content people or UX leads who understand what should and should not be allowed on each type of page.
Developers
Create the components and define what policy-level settings are available (via the _cq_design_dialog). Also set up the initial allowed components list when deploying.
Editors
Cannot create or edit policies. They work within the constraints the template authors set.
Where policies are stored in the JCR

Style System in policies — one component, many looks

The Style System is a feature of policies that lets template authors define named visual styles for a component. Editors pick from these named styles. The correct CSS class is automatically applied. Zero code change needed to add a new visual variant.

The concept

Without the Style System, if you want a "Card" component to have three visual variants (Standard, Featured, Minimal), you need three separate components, three separate HTL files, and three separate dialog registrations. That is three times the code to maintain.

With the Style System, you have ONE Card component. A template author creates three named styles in the policy. Each style maps to one or more CSS class names. When an editor picks "Featured", AEM adds the CSS class card--featured to the component's wrapper div. Your CSS file makes it look featured. Same component, same HTL, same dialog — different visual outcome.

4
Add CSS to your clientlib
In your clientlib CSS: .card--featured { border: 2px solid gold; font-size: 1.2em; } and .card--minimal { background: none; padding: 0; }.
How editors use styles

In the page editor, the editor clicks on a Card component. The blue toolbar appears. There is now a paintbrush icon alongside the wrench and delete icons. When an editor selects a style, AEM stores the style ID as a cq:styleIds property on the component's JCR content node. On render, AEM looks up the style ID in the policy, finds the corresponding CSS class name, and adds it to the component's wrapper element.

Part 4 — policies

Design mode — the legacy policy system

Design mode is the predecessor to policies. It was the mechanism for controlling allowed components in static templates. You will encounter it in older AEM projects. Understanding it prevents confusion when you see it.

How Design mode worked

In Design mode, a developer or admin would visit a page with the URL parameter ?wcmmode=design. The page would switch to a special mode where each Parsys had a "Configure" icon. Clicking it opened a dialog where allowed components could be checked and unchecked.

These design settings were stored in the JCR under /etc/designs/mysite/ — separate from both the template and the page content. When a new page was created, it inherited the design settings from its parent path.

Problems with Design mode
  • Settings stored in /etc/designs/ were not version-controlled and easily lost during migrations
  • Editing required a special URL parameter — not discoverable for non-developers
  • No visual indication of what was allowed when in normal authoring mode
  • Design dialog settings were managed entirely separately from template structure
If you encounter Design mode today

If you see /etc/designs/mysite/ in a JCR or a project's source code, you are looking at a pre-AEM 6.2 project using static templates. Design mode still works in current AEM versions for backward compatibility but should not be used for new development. Migrate to editable templates and policies when possible.

An AEM component is a reusable piece of UI that editors drag onto pages. Each component has a specific purpose — a text block, a hero banner, a product card, a testimonial slider. Developers build components. Editors use them to build pages.

The Lego analogy

Think of AEM components as Lego bricks. Each brick has a specific shape and function. Developers design and manufacture the bricks. Part 6 — Sling resolution

What is Sling — the web request engine

Apache Sling is the web framework inside AEM. Every single HTTP request to AEM goes through Sling. Sling decides which script (which HTL file) renders each request. Understanding Sling is understanding how AEM renders pages.

The core principle — REST-ful content

Sling is built on one central idea: every URL maps to a JCR node, and the data on that node determines how it is rendered. This is called "Resource-oriented" or "Content-centric" design. There are no controller classes, no servlet mappings, no routing tables. The URL IS the data path.

In Spring MVC (a typical Java web framework), you would write a controller class with @RequestMapping("/articles/summer-sale") that handles that URL. In Sling, there is no such mapping. The URL /content/mysite/articles/summer-sale directly corresponds to the JCR node at that path. Sling finds the node, reads its sling:resourceType, and renders it with the matching script.

Why this matters for AEM developers

Because Sling's resolution is based on sling:resourceType, you never configure URL routes. Instead, you:

  • Set sling:resourceType = "mysite/components/card" on a content node
  • Create the rendering script at /apps/mysite/components/card/card.html
  • Sling automatically connects the two — whenever a node with that resourceType is requested, Sling runs that script

This is why the HTL filename must match the component folder name — Sling looks for a script named after the component name (last part of the resourceType path).

Sling script resolution search order

When Sling looks for a rendering script for a given resource, it searches in this order:

1
Check /apps first
Sling always looks in /apps/ before /libs/. This is how overlays work — your script in /apps/ takes precedence over AEM's default in /libs/.
2
Walk up the sling:resourceSuperType chain
If the script is not found in your component folder, Sling checks the parent component (defined in sling:resourceSuperType), then the parent's parent, and so on until it finds the script or runs out of parents.
3
Fall back to /libs
If no script is found in /apps/, Sling looks in /libs/ for the same path. This is how Core Components work — their scripts live in /libs/core/wcm/components/.

Part 6 — Sling resolution

Sling resolution — step by step

When a request arrives at AEM, here is the exact sequence of steps Sling follows to find the right rendering script. This is the most important algorithm in AEM to understand.

The full resolution sequence
1
Request arrives: /content/mysite/en/home.html
Sling receives the HTTP GET request. It parses the URL into: JCR path = /content/mysite/en/home, no selectors, extension = html.
2
Resolve the JCR node
Sling looks up /content/mysite/en/home in the JCR. It finds a cq:Page node. The actual content is on its child node jcr:content — Sling automatically uses this for rendering.
3
Read sling:resourceType
Sling reads the sling:resourceType property from /content/mysite/en/home/jcr:content. Value: mysite/components/page/base. This is the component that will render this node.
4
Determine the script name to look for
From the resourceType, Sling extracts the component name: base (last part of the path). With extension html and no selectors, it looks for a script named base.html first, then GET.html as fallback.
5
Search for the script in /apps first
Sling checks /apps/mysite/components/page/base/base.html. Found! Uses this script.
6
Execute the script
Sling runs base.html with the JCR node as the current resource. The HTL template can access all properties of the jcr:content node via ${properties.xxx} or via a Sling Model.
7
Return the HTML response
The rendered HTML string is returned as the HTTP response body. The browser receives the complete HTML page.
Script search priority when a script is NOT found

If Sling does not find base.html at /apps/mysite/components/page/base/, it continues searching in this priority order:

1st (most specific)
Script matching selector(s) + extension: base.article.wide.html
2nd
Script matching first selector + extension: base.article.html
3rd
Script matching extension only: base.html or GET.html
4th
Check parent via sling:resourceSuperType — repeat the same search on the parent component
5th
Check /libs at the same path pattern
Last resort
Default GET servlet — renders the resource as a simple dump of its properties
Part 6 — Sling resolution

Selectors — multiple renderings from one JCR node

Selectors are the parts of the URL between the resource path and the extension. They let you request different renderings of the same JCR content without creating separate pages. Same data, different presentation.

How selectors work

When a URL has a selector, Sling looks for a script whose name matches that selector. For the URL /content/mysite/article.print.html:

  • JCR path: /content/mysite/article
  • Selector: print
  • Extension: html
  • Sling looks for: article.print.html in the component folder first, then article.html as fallback

So if you create a file /apps/mysite/components/page/article/article.print.html, it will render for all requests with the .print. selector. Requests without the selector still use article.html. Two different renderings of the same page content.

Common use cases for selectors
.print.html
Printer-friendly version of a page — minimal CSS, no navigation, optimised for printing.
.amp.html
AMP (Accelerated Mobile Pages) version — special stripped HTML for Google's AMP framework.
.model.json
JSON representation of the component data. Used by AEM's SPA (Single Page Application) framework to deliver content to React/Angular apps.
.results.json
Search results endpoint. A search page with this selector returns JSON results for AJAX requests instead of a full HTML page.
.infinity.json
AEM's built-in infinite-depth JSON dump of a JCR node's entire subtree. Useful for debugging.
Multiple selectors

URLs can have multiple selectors: /content/mysite/article.article.wide.html has two selectors: article and wide. Sling searches for scripts in this order: article.article.wide.html → article.article.html → article.html. Most specific to least specific.

Real example
A newspaper site has article pages at /content/news/articles/story.html. They need: normal web version, printer-friendly version, AMP version for Google, and a JSON API for their mobile app. They create four scripts in the article component: story.html (web), story.print.html (print), story.amp.html (AMP), story.model.json (API). All four renderings serve the same JCR content — the editor writes the article once and all four channels are automatically served.
Part 6 — Sling resolution

sling:resourceType — the most important property in AEM

sling:resourceType is a single JCR property that connects content to code. It is the answer to "which component renders this node?". Every content node that needs to produce output must have this property set correctly.

What sling:resourceType does

When Sling encounters a JCR node during request processing, the first thing it does is read the sling:resourceType property. This property contains a path (relative to /apps or /libs) to the component that should render this node.

Example: a card component's content node at /content/mysite/en/home/jcr:content/root/card has sling:resourceType = "mysite/components/card". Sling finds the rendering script at /apps/mysite/components/card/card.html and uses it.

Where sling:resourceType is set
On page's jcr:content
Set to the Page Component path. Determines which Page Component wraps this page. Set via the template reference or manually: mysite/components/page/base
On component content nodes
Set automatically by AEM when an editor drags a component onto a page. AEM reads the sling:resourceType from the component's .content.xml and sets it on the new content node.
In component .content.xml
Not the resourceType of the component — instead this is the component's own identity. The component's folder path IS its resourceType. No explicit property needed here.
In dialogs and editConfigs
Used to specify which Granite UI component type to use for each field: sling:resourceType = "granite/ui/components/coral/foundation/form/textfield"
Debugging with sling:resourceType

The #1 debugging step for any rendering problem: open CRXDE, navigate to the content node of the misbehaving component, click the Properties tab, and check sling:resourceType. Ask:

  • Does the value match the folder path under /apps?
  • Does that folder actually exist in /apps?
  • Does a script with the right name (folder-name.html) exist inside that folder?

90% of "component not rendering" or "component rendering blank" issues are caused by a wrong or missing sling:resourceType.

sling:resourceSuperType — component inheritance

While sling:resourceType says "which component am I?", sling:resourceSuperType says "who is my parent?". Set in the component's .content.xml, it creates an inheritance chain that Sling follows when looking for scripts. If your component does not have card.html, Sling checks the parent, then the grandparent. This is how extending Core Components works.
