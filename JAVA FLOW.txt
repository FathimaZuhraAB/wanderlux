# ✅ COMPLETE JAVA PROGRAM FLOW (NO GAPS)

We’ll follow **exactly what happens**, starting from when *you write code* till *you see output on the screen*.

***

## 🧩 STEP 0: Installing Java (Foundation)

Before writing code, you install **JDK** on your system.

### What gets installed?

### ✅ JDK (Java Development Kit) includes:

*   **javac** → Java **compiler**
*   **java** → launcher to start JVM
*   **JRE**
*   Development tools (debugger, javadoc, etc.)

📌 **Important**:

*   JDK ⊃ JRE ⊃ JVM  
    (JDK contains JRE, JRE contains JVM)

***

## 📝 STEP 1: Writing Java Source Code

You write code in a text editor or IDE.

```java
class Hello {
    public static void main(String[] args) {
        System.out.println("Hello Java");
    }
}
```

✅ File is saved as:

    Hello.java

### What this is:

*   **Human‑readable**
*   **NOT executable**
*   OS **cannot run** `.java`

***

## 🧠 STEP 2: Compilation (JDK → javac)

Now you compile the file:

```bash
javac Hello.java
```

### 🔹 Who handles this?

✅ **javac** (Java Compiler)  
✅ Lives inside **JDK**

***

### 🔹 What compiler does internally (VERY IMPORTANT):

1.  **Lexical Analysis**
    *   Breaks code into tokens

2.  **Syntax Analysis**
    *   Checks grammar
    *   Missing semicolon? ❌ Error here

3.  **Semantic Analysis**
    *   Type checking
    *   Method existence
    *   Variable scope

4.  **Bytecode Generation**

✅ If **any error exists**, `.class` file is **NOT created**

***

### ✅ Output of Compilation

    Hello.class

### What `.class` file contains:

*   **Bytecode**
*   Platform‑independent
*   **NOT machine code**
*   Designed for **JVM**, not OS

📌 **Big point**:

> javac converts `.java` → `.class`  
> **javac NEVER produces machine code**

***

## ▶️ STEP 3: Running the Program (JRE → JVM)

You now run:

```bash
java Hello
```

⚠️ Notice:

*   You **do NOT** write `.class`
*   The **java command starts the JVM**

***

## ⚙️ STEP 4: JVM Starts (Inside JRE)

### JVM is loaded from:

✅ **JRE**

### JVM components loaded:

1.  **Class Loader subsystem**
2.  **Runtime Memory Areas**
3.  **Execution Engine**
4.  **Garbage Collector**
5.  **Security Manager (if enabled)**

***

## 📦 STEP 5: Class Loading (Class Loader)

### JVM asks:

> “Where is `Hello.class`?”

### Class Loader:

*   Loads `Hello.class`
*   Loads required classes:
    *   `java.lang.Object`
    *   `java.lang.System`
    *   `java.io.PrintStream`

✅ This happens **dynamically at runtime**

***

## 🔍 STEP 6: Bytecode Verification (Security Step)

Before execution, JVM checks:

*   Stack overflow possibility?
*   Illegal memory access?
*   Invalid bytecode?

✅ If verification fails → **program stops**
✅ This is why Java is considered **secure**

***

## 🧠 STEP 7: JVM Memory Allocation

JVM allocates memory:

### 🔹 Method Area

*   Class metadata
*   Bytecode of methods

### 🔹 Heap

*   Objects (`new` keyword)

### 🔹 Stack

*   Method calls
*   Local variables

### 🔹 PC Register

*   Keeps current instruction

### 🔹 Native Method Stack

*   For native (C/C++) methods

***

## ▶️ STEP 8: Execution Begins (Interpreter)

### JVM finds:

```java
public static void main(String[] args)
```

Execution starts **from main()**.

***

### 🔹 JVM Interpreter

*   Reads **bytecode instruction by instruction**
*   Converts each instruction to:
    *   OS‑specific machine instructions
*   Executes immediately

📌 Slower, but fast startup

***

## 🚀 STEP 9: JIT Compiler (Optimization)

While program is running:

*   JVM tracks frequently used methods (hot code)
*   JIT compiles those bytecodes into **native machine code**
*   Stores them in memory
*   Reuses them

✅ Result:

*   Faster execution
*   No recompilation of `.java`

📌 **Very important clarification**:

> JIT compiles **bytecode**, NOT `.java`

***

## 🧹 STEP 10: Garbage Collection

During execution:

*   JVM detects unused objects
*   Frees heap memory automatically
*   No `free()` like C++

✅ Happens silently in background

***

## 🖥️ STEP 11: Interaction with OS

Machine code generated by JVM:

*   Calls OS system calls
*   Uses hardware (CPU, RAM)
*   Displays output

<!---->

    Hello Java

✅ Program successfully runs 🎉

***

## ❌ WHY JRE CANNOT COMPILE `.java`

This answers your confusion **once and for all**:

### JRE contains:

✅ JVM  
✅ Libraries

### JRE does NOT contain:

❌ javac  
❌ Development tools

So:

    .java + JRE ❌ → cannot compile
    .java + JDK ✅ → compile success

Even though JVM exists inside JRE, **JVM is NOT a source code compiler**.
