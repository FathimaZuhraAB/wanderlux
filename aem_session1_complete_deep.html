Part 1 — The problem

The problem that existed before CMS

To understand why AEM exists, you first need to understand the painful reality of managing a website without a CMS. Every concept in AEM makes perfect sense once you understand what problem it is solving.

Imagine this scenario

It is 2005. You work at a bank. The bank's website has 300 pages — home loans, savings accounts, credit cards, branch locators, FAQs, about us pages. Every page is a static HTML file stored on a server.

Today the Reserve Bank changed the interest rate. The bank needs to update the interest rate on 47 different pages of the website. What happens?

1
A developer receives a ticket
The marketing manager emails a developer: "Please update the home loan rate from 8.5% to 9.2% on all pages."
2
Developer finds all the files
The developer SSHs into the server, runs a grep command to find all HTML files mentioning "8.5%", gets a list of 47 files.
3
Developer edits HTML files
Opens each file in a text editor, finds the rate, changes it, saves. Repeats 47 times. One mistake means a wrong number live on the website.
4
Developer deploys the files
Uploads the changed files to the production server via FTP or a deployment script. This takes more time.
5
Hours or days have passed
The whole process took 3–4 hours. During that time, the website showed incorrect information to customers. The developer wasted their entire morning on content changes instead of building features.

This happened every single time any content changed. New product? Developer. Typo fix? Developer. Update a photo? Developer. The developer was a bottleneck for everything.

The second big problem — inconsistency

When 47 different developers or even the same developer edits HTML files manually over years, the website becomes inconsistent. The footer looks different on old pages vs new pages. Font sizes vary. Some pages have outdated branding. There is no single source of truth for how things should look.

The third problem — no workflow

A junior marketer writes a new product description. It needs a legal review, a brand review, then a manager approval before going live. With static HTML, there is no process for this. The file either goes live or it doesn't. There is no "draft" state, no "waiting for approval" state, no notification system, no audit trail of who changed what when.

This is why CMS was invented
All three of these problems — developer bottleneck, inconsistency, no workflow — are exactly what a Content Management System solves. AEM solves all three at enterprise scale, for companies with thousands of pages across hundreds of countries.
Part 1 — CMS

What is a CMS — explained from scratch

A Content Management System is software that separates content from code. Editors manage content through a user interface. Developers manage code and templates. Neither needs the other for day-to-day work.

The fundamental idea

In a static website, content and code are mixed together. The HTML file contains both the page structure (HTML tags, CSS classes) AND the actual content (words, images). Changing the content means touching the code.

A CMS splits these into two separate things. The content lives in a database or repository. The templates (the HTML structure, the design) live separately as code. When a user requests a page, the CMS takes the template and fills it with the content from the database and renders the final HTML.

Content in database
"Summer Sale 2024"
+
Template (HTML + CSS)
<h1>{{title}}</h1>
=
Final HTML page
<h1>Summer Sale 2024</h1>

Now a marketing manager can log into the CMS, find the page, click on the title, type new text, and click Save. The template does not change. The code does not change. Only the content in the database changes. No developer needed.

The three layers of a CMS
Content repository

Where all content is stored. In traditional CMS it is a relational database (MySQL). In AEM it is the JCR — a tree-structured content repository where everything is a node with properties.

Authoring interface

The user interface where editors work. Can be form-based (fill in fields), WYSIWYG (click directly on the page), or both. AEM uses both simultaneously.

Delivery engine

Takes content from the repository, applies it to a template, and renders HTML for the visitor's browser. In AEM, this is Apache Sling doing the rendering.

Examples of CMS platforms
WordPress
The world's most used CMS. Powers 43% of all websites. Good for blogs, small business sites, simple content. Not built for enterprise scale.
Drupal
Open-source CMS. More flexible than WordPress. Used by governments and universities. Requires technical setup and management.
Contentful
Headless CMS. Content is stored as JSON and delivered via API. No built-in frontend. Your React or Vue app consumes the content.
Adobe AEM
Enterprise-grade WCM. Built for large companies with thousands of pages, multiple brands, multiple countries, complex workflows. This is what you are learning.
Real example — Zomato
Zomato's marketing team publishes new restaurant partner pages, festive offer banners, and blog articles every single day. Their marketing manager logs into their CMS, creates a new page, fills in the restaurant name, photos, menu highlights, and clicks Publish. The page is live in minutes. Zero developers involved. This is CMS working exactly as intended.
Part 1 — WCM

What is WCM — and how it differs from a basic CMS

WCM stands for Web Content Management. It is a CMS that has been built specifically for large, complex enterprise websites. Not just one website — hundreds of websites, in dozens of languages, across multiple brands, all from one platform.

Where basic CMS stops and WCM begins
Basic CMS (e.g. WordPress)
WCM (e.g. AEM)
One website. Maybe a few languages if you install a plugin. No concept of "sites" or "brands" — just one installation per site.
Hundreds of websites from one installation. Marriott manages 30 sub-brands (Sheraton, Westin, Courtyard) all from one AEM instance.
No built-in workflow. Content goes live when you click publish, no approval process.
Full workflow engine. Content goes through Draft → Review → Legal Approval → Brand Approval → Scheduled Publish — with email notifications at each step.
Basic media library. Upload images, use them. No metadata, no auto-resizing, no rights management.
Full DAM (Digital Asset Management). Auto-generates 15 image renditions, manages copyright expiry, watermarking, smart tagging with AI.
One language. Multilingual is possible with plugins but messy and unreliable at scale.
Built-in multilingual with translation workflows. Content is structured so translators can work in isolation without touching the original.
No personalization. Everyone sees the same page.
Adobe Target integration. A returning customer in Mumbai sees different content than a first-time visitor in Delhi.
Why enterprises need WCM specifically

Consider Unilever. They have 400+ brands (Dove, Lipton, Knorr, Surf). Each brand has a website. Each website runs in 50+ countries. Each country has different products, different pricing, different regulatory requirements, different languages. That is 20,000+ pages being updated daily by hundreds of editors across the globe.

A basic CMS cannot handle this. It would require 20,000 separate WordPress installations, each managed separately, each with different plugins and updates, each potentially going out of sync. This is a nightmare.

A WCM like AEM handles all 20,000 pages from one platform with shared templates, shared brand guidelines enforced by policies, shared assets in one DAM, and one unified authoring interface for all editors globally.

Real example — HDFC Bank
HDFC Bank has a website with over 2,000 pages in English and Hindi, covering personal banking, business banking, NRI banking, investments, insurance, and credit cards. Their compliance team must review every piece of content before it goes live. Their legal team must approve any mention of interest rates. Their brand team ensures consistent visual language. All of this is managed through AEM's workflow engine. A content editor cannot accidentally publish an unapproved interest rate — the workflow literally prevents it.
Part 1 — AEM

AEM is Adobe's enterprise WCM platform. The word "Experience" in the name is important — AEM is not just about managing text on pages. It manages the entire digital experience a customer has with a brand, across every channel they interact with.

AEM Sites

The web content management part. Create pages, manage templates, drag-drop components, run A/B tests. This is the main thing you are learning.

Part 1 — technology

What AEM is built on — the technology stack

AEM is not built from scratch. It is assembled from several well-known open-source Apache projects layered on top of each other. Understanding each layer helps you understand why AEM behaves the way it does.

The layers from bottom to top
1
Java Virtual Machine (JVM)
Everything runs inside a JVM. AEM is a Java application. When you start AEM by running java -jar aem-author-p4502.jar, you are starting a JVM that then starts all the other layers.
2
Apache Felix — the OSGi container
OSGi (Open Service Gateway Initiative) is a module system for Java. Think of it like a plugin system. AEM's Java code is split into many small "bundles" (JAR files). Felix manages these bundles — starts them, stops them, resolves their dependencies. You can install a new bundle (your code) without restarting the whole application. This is why deploying your code is fast — you just install a new bundle.
3
Apache Jackrabbit Oak — the JCR
The content repository. Stores all data — pages, components, configs, users, assets — as a tree of nodes. Think of it like a file system for AEM. All data is persistent here. Oak uses a segment store (binary files on disk) to store data efficiently.
4
Apache Sling — the web framework
Sling is the web request handling layer. When a browser requests /content/mysite/home.html, Sling figures out which component should render that URL. It reads the JCR node at that path, finds the sling:resourceType property, and calls the matching rendering script (your HTL file). This "resource resolution" is the core of how AEM works.
5
Adobe Granite — the AEM layer
Adobe's layer on top of Sling. Adds all the AEM-specific things: the authoring UI (the page editor, Sites console, DAM), workflows, user management, replication, the Dispatcher integration. This is what you see when you open AEM in a browser.
As a developer, you mostly work with Sling (resource resolution, Sling Models, Servlets) and Granite (component dialogs, author UI). You rarely need to interact with Felix or Oak directly, though you use the Felix console to check your bundles.

Why three layers? Why not just one?

This question is important. Why not just run one AEM server and let both editors and visitors use it?

Because a website that gets a million visitors per day cannot also be the place where editors make changes. The load would be enormous. More importantly, editors need to see draft content, unpublished changes, and work-in-progress. Visitors must never see any of that. They should only ever see approved, published content.

The three-layer architecture solves both problems cleanly. Author handles editing. Publish handles delivery. Dispatcher handles scale and security.

Real example — Flipkart Big Billion Days sale
When Flipkart announces their Big Billion Days sale, their content team has been building the campaign pages for weeks on Author — draft banners, product lists, countdown timers. None of this is visible to customers. One minute before the sale goes live, they publish everything from Author to Publish. Instantly, millions of customers can see the sale. The Dispatcher serves cached HTML to most of them without even touching AEM. The Author server is completely unaffected by the traffic surge.
Part 2 — Author layer

The Author layer — everything about it

The Author layer is where all content creation happens and where developers deploy their code. You will spend 100% of your development time working with the Author server.

Basic facts
Default URL
http://localhost:4502 on your local machine
Default login
admin / admin — ONLY on local development. Production uses proper credentials.
Who accesses it
Content editors, developers, template authors, admin staff. Never end-users or the public.
Network access
Behind a VPN or firewall. Never publicly accessible. If a stranger on the internet could reach your Author, they could delete everything.
Runmode
Starts with -r author runmode. This activates author-specific OSGi configs and disables some performance optimizations that are only needed on Publish.
What happens on Author
  • Content editors create pages, edit text, upload images, configure components
  • Template authors create and manage page templates and policies
  • Developers deploy code using Maven (mvn clean install -PautoInstallPackage)
  • Workflow approvals happen here (review, legal check, brand check)
  • Replication agents run here — they push content to Publish
  • User accounts and permissions are managed here
  • The entire AEM admin UI is available here
How to start AEM Author locally
1
Get the AEM JAR file
You need a file called aem-author-p4502.jar and a license.properties file. Get these from Adobe's download portal or from your company's repository. Put both in the same folder.
2
Run the JAR
In your terminal, navigate to that folder and run: java -jar aem-author-p4502.jar -gui. The -gui flag shows a small progress window.
3
Wait for startup
First startup takes 5–10 minutes. AEM is unpacking and initializing the repository. Subsequent starts take 1–2 minutes.
4
Open in browser
Go to http://localhost:4502. You'll see the AEM login page. Login with admin / admin.
The first time you run AEM, it creates a crx-quickstart/ folder next to the JAR file. This folder is the repository — it contains everything. Never delete this folder unless you want to start fresh.
Part 2 — Publish layer

The Publish layer — everything about it

The Publish layer is the AEM server that end-users interact with. It only contains content that has been explicitly activated from Author. Nothing draft, nothing in-progress ever appears here.

Basic facts
Default URL
http://localhost:4503 locally. In production it sits behind a load balancer and is never directly accessible.
Who accesses it
Only end-users (via Dispatcher). Developers access it directly only for testing. Never used for content creation.
UI available
Very limited. No page editor, no Sites console, no DAM UI. Just AEM rendering pages for visitors.
Runmode
Starts with -r publish. Disables author-only features, enables publish-specific caching and security configs.
Multiple instances
In production, you run 2–10 Publish instances behind a load balancer. If one crashes, others keep serving. Author only needs one instance (it's not public-facing).
How to start AEM Publish locally

Same as Author but use a different JAR name or a different port parameter. The simplest approach on local: copy the Author JAR to a new folder, rename it to aem-publish-p4503.jar, and start it. AEM reads the port from the filename.

start commandjava -jar aem-publish-p4503.jar -gui
What "published" means technically

When an editor clicks "Publish" on a page in Author, AEM's Replication Agent fires. It serializes the page's JCR nodes (the content tree under /content/mysite/en/home) and sends them over HTTP to the Publish server's replication endpoint. The Publish server receives this and writes the nodes into its own JCR repository.

From that point on, the Publish server has the content and can render it when visitors request it. The two repositories (Author's JCR and Publish's JCR) are separate — they do not sync automatically. Only explicitly published content moves from Author to Publish.

Real example — understanding the separation
An editor at a newspaper works on tomorrow's front page article all day on Author. They can preview it, show it to their editor-in-chief, revise it. The live newspaper website shows today's articles and has no idea the new article exists. At midnight, the editor clicks Publish. The article replicates to Publish and immediately becomes visible to readers. The Author work and the live website were completely isolated until that moment.
Part 2 — Dispatcher

The Dispatcher — cache, security, and scale

The Dispatcher is an Apache web server module that sits in front of Publish. It is not AEM software — it is a plugin for Apache httpd or Microsoft IIS. It does three things: caches pages, blocks dangerous URLs, and load-balances across Publish instances.

Job 1 — caching (the most important job)

When a visitor requests a page, the Dispatcher checks if it has a cached HTML file for that URL on disk. If yes, it serves the cached file directly — AEM is never involved. This is extremely fast and handles massive traffic.

If no cached file exists (or the cache was cleared), the Dispatcher forwards the request to Publish. Publish generates the HTML by reading from JCR and rendering templates. The Dispatcher then saves this HTML to disk as a cache file and returns it to the visitor.

Visitor requests /page.html
→
Cache file exists?
→
YES: serve cache instantly
Visitor requests /page.html
→
Cache file exists?
→
NO: forward to Publish
→
Publish renders HTML
→
Save to cache, return to visitor
Job 2 — cache invalidation (flushing)

When a page is published on Author, AEM also sends a "flush" request to the Dispatcher. This tells the Dispatcher to delete the cached HTML file for that page. The next visitor will get a fresh version from Publish.

This is called cache invalidation. Without it, visitors would always see the old cached version even after content is updated.

Job 3 — security filtering

The Dispatcher has a filter configuration that blocks requests to sensitive AEM URLs. These should never be accessible on the public internet:

  • /crx/de — CRXDE Lite. An attacker with access could read/write everything in JCR.
  • /system/console — OSGi console. An attacker could stop your bundles or change configs.
  • /bin/wcmcommand — Internal AEM commands
  • /libs/granite/security/... — Security-related endpoints

The Dispatcher blocks all of these with a URL filter — even if somehow the Publish server was directly reachable, these paths would still be blocked.

In local development you typically do not set up the Dispatcher. You work directly with Author on 4502. The Dispatcher is set up in staging and production environments by the infrastructure team.
Part 2 — Replication

Replication — how content moves from Author to Publish

Replication is the technical process of copying content from the Author JCR repository to the Publish JCR repository. It is triggered when an editor publishes a page. Understanding this process tells you exactly what happens when you click "Publish".

Step by step — what happens when you click Publish
1
Editor clicks "Publish" on a page
In the Sites console or the page editor toolbar, the editor activates a page. AEM starts the replication process.
2
AEM finds the Replication Agent
A Replication Agent is configured at /etc/replication/agents.author/publish. It knows the Publish server's URL and credentials. AEM uses this agent to send content.
3
AEM serializes the JCR nodes
The page and all its component content nodes under /content/mysite/en/home are serialized into a package (a zip file of JCR data).
4
Package is sent to Publish via HTTP
The Replication Agent sends this package to the Publish server's replication endpoint at http://publish:4503/bin/receive.
5
Publish installs the package
The Publish server receives the package and writes the JCR nodes into its own repository. The content is now on Publish.
6
Dispatcher cache is flushed
AEM also sends a flush request to the Dispatcher, telling it to delete the cached HTML for this page. The next visitor will get a fresh page.
Where to check replication status

Go to http://localhost:4502/etc/replication/agents.author.html. You can see all replication agents and test them. If the "Publish" agent shows an error, content is not reaching the Publish server.

Unpublishing a page

Unpublishing (deactivating) a page reverses replication — it removes the page's nodes from the Publish JCR and sends a flush request to the Dispatcher to delete the cached file. The page immediately disappears from the live website.

Part 3 — JCR

What is the JCR — Java Content Repository

The JCR is where AEM stores literally everything. Not just page content — your code (components, templates), your configuration, your users, your workflows, your assets — all of it lives in the JCR as a tree of nodes. If you understand the JCR, you understand AEM.

The simplest explanation possible

Think of the JCR like a file system, but instead of files and folders you have nodes. Each node can have:

  • A name (like a folder name)
  • A type (jcr:primaryType — what kind of node it is)
  • Properties — key/value pairs storing data (like title = "Home Page")
  • Child nodes (just like subfolders)

Every URL in AEM maps directly to a JCR path. The page at http://localhost:4502/content/mysite/en/home.html corresponds to the JCR node at /content/mysite/en/home. That's it. AEM's entire URL routing is just JCR paths.

What a page looks like in the JCR
/content/mysite/en/home ← cq:Page node (the page itself) └── jcr:content ← cq:PageContent node (the page's data) jcr:title = "Home" ← the page title property sling:resourceType = "mysite/components/page/base" cq:template = "/conf/mysite/settings/wcm/templates/home-page" └── root ← layout container node sling:resourceType = "wcm/foundation/components/responsivegrid" └── banner ← your banner component sling:resourceType = "mysite/components/banner" title = "Welcome to My Site" imageRef = "/content/dam/mysite/hero.jpg" └── card_1 ← a card component inside banner sling:resourceType = "mysite/components/card" title = "Summer Sale"

This tree is exactly what you see when you open CRXDE and navigate to /content/mysite/en/home. The property values here are exactly what editors typed into component dialogs. The sling:resourceType properties tell AEM which component (which HTL file) to use for rendering each node.

Why the JCR matters to you as a developer

When you write a Sling Model, it reads from the JCR. When an editor fills in a dialog, the values save to the JCR. When you create a component, you are creating JCR nodes. When you write an HTL template, it renders content from the JCR. Everything goes through the JCR.

The most important debugging skill in AEM is opening CRXDE, finding the relevant JCR node, and reading its properties. 90% of problems become obvious once you can see what is actually stored in the JCR.

Part 3 — nodes

JCR nodes, node types, and properties

Every piece of data in AEM is a node. Each node has a type that defines what it is. Each node has properties that store its data. Here are the node types you will see every day.

The most important node types
cq:Page
A web page. Found at paths like /content/mysite/en/home. Every page you create in the Sites console becomes a node with this type. It always has a child node called jcr:content that holds the actual page data.
cq:PageContent
The content node of a page. Always named jcr:content and always a child of a cq:Page node. Has properties like jcr:title, sling:resourceType, cq:template.
cq:Component
A component definition. Found at paths like /apps/mysite/components/card. When you create a component, you set this as the jcr:primaryType in your .content.xml file.
nt:unstructured
A generic, schema-less data node. Used for component content (the data an editor saves), dialog definitions, and many other things. Very flexible — can have any properties.
dam:Asset
A digital asset in the DAM. Found at paths like /content/dam/mysite/images/hero.jpg. Has child nodes for metadata and renditions.
cq:ClientLibraryFolder
A ClientLib node. Found at paths like /apps/mysite/clientlibs/site. Has categories and dependencies properties. Tells AEM how to bundle and serve CSS/JS.
The most important properties
sling:resourceType
The single most important property in AEM. On any content node, it tells AEM which component renders this node. Value is a path relative to /apps, e.g. mysite/components/card. Sling looks for the rendering script at /apps/mysite/components/card/card.html.
jcr:primaryType
The node type. Defines what kind of node this is. Every node must have this.
jcr:title
The human-readable name. On pages: the page title. On components: the name shown in the component browser. On templates: the name shown in the template picker.
sling:resourceSuperType
On a component node: the parent component to inherit from. AEM walks up this chain when looking for rendering scripts or dialog fields that are not found locally.
cq:template
On a page's jcr:content node: the path to the editable template used to create this page.
How this connects to your code
When you write a Sling Model with @ValueMapValue private String title;, it reads the title property from the JCR node. When you write ${model.title} in HTL, it calls getTitle() which returns that JCR value. When the page renders, the browser receives HTML with the editor's text in it. The whole system is just: editors write data to JCR nodes, your code reads it back and formats it as HTML.
Part 3 — paths

Key JCR paths — the map of where everything lives

The JCR tree has a consistent structure. Once you know this map, you always know where to look for anything in AEM.

The top-level paths and what they contain
/apps/
Your application code. Components, templates (static), ClientLibs, OSGi configs, i18n translation files. This is YOUR code. In CRXDE this is where you navigate to find component files.
/apps/mysite/components/
All your component folders. Each subfolder here is one component. Navigate here in CRXDE to see and edit component HTL files and dialogs.
/apps/mysite/clientlibs/
ClientLib nodes containing CSS and JS. Each subfolder is one library with a unique category ID.
/content/
All page content. What editors create. The site hierarchy here mirrors your site's URL structure.
/content/mysite/en/home
A specific page. Navigate here in CRXDE to see what values editors have set on components. The Properties tab shows all JCR properties of the selected node.
/content/dam/
Digital Asset Manager. All uploaded images, videos, PDFs, Content Fragment instances. Each asset is a dam:Asset node.
/conf/
Configuration data. Editable templates, Content Fragment models, site-specific OSGi configs, policies. Created through the AEM UI, not code.
/conf/mysite/settings/wcm/templates/
All editable page templates for your site. Created in Tools → Sites → Templates.
/libs/
AEM's built-in code. Core Components, default UI, default scripts. Never edit anything here. AEM overwrites /libs on every Service Pack update. Customize by copying to /apps.
/etc/replication/
Replication agents. The configuration that tells Author how to push content to Publish.
/var/
Runtime data. Workflow instances, audit logs, replication queues. You rarely need to look here.
/home/users/
AEM user accounts and groups. Manage these through Tools → Security.
The single most useful debugging habit: when something renders wrong, navigate to the component's content node in CRXDE (/content/mysite/.../jcr:content/root/yourcomponent), click the Properties tab, and check: is sling:resourceType pointing to the right component? Are the dialog values saved correctly? This answers 80% of rendering questions immediately.
Part 4 — WYSIWYG

WYSIWYG — What You See Is What You Get

WYSIWYG means the editing interface looks exactly like the final published page. You do not edit in a separate form and then switch to preview. You edit directly on the rendered page and see exactly what visitors will see.

What this means in AEM specifically

When you open the page editor in AEM, AEM renders the actual page — with all the real CSS, real images, real fonts, and real layout. It then overlays this with a thin authoring layer that adds blue selection handles and edit toolbars on each component. The page underneath is the real published appearance.

When you click on a text component and double-click to edit it, a Rich Text Editor appears directly on the page at the exact position of the text. You type your changes. The text on the page updates as you type. When you click away, the RTE disappears and you see the final appearance.

This is fundamentally different from older CMS approaches where you edited content in a separate form, then clicked "Preview" to see what it would look like — a slow and confusing workflow.

Two types of editing in AEM
Inline editing (in-place)
Dialog editing (configure)

Double-click directly on a text component. A Rich Text Editor toolbar appears at the top of the component. You type the new text directly on the page.

Used for: Text components, Title components — anything where the output is primarily text you can see on the page.

Click the component once (blue toolbar appears), then click the wrench icon. A dialog popup appears with form fields. Fill in the fields and click the checkmark to save.

Used for: Complex components with multiple settings — image paths, links, dropdowns, checkboxes. Things that are not just editable text.

Real example — editing a banner
A marketing manager at Myntra needs to update the homepage hero banner for their End of Reason Sale. She opens the homepage in the AEM page editor. Part 4 — modes

Edit mode, Layout mode, and Preview mode

The AEM page editor has three different modes. Each mode gives editors a different view and different capabilities. You switch between them using buttons in the top toolbar of the page editor.

Edit mode — the default working mode

When you first open a page in the editor, it is in Edit mode. This is the mode where:

  • Clicking a component selects it and shows the blue toolbar with Edit, Configure, Delete, Copy buttons
  • Dragging from the left component browser panel drops a new component onto the page
  • Double-clicking a text component activates the Rich Text Editor inline
  • You can reorder components by dragging them to new positions

In Edit mode you can see a faint blue outline around each component even when it is not selected. This helps you understand the page structure — where one component ends and another begins.

Layout mode — the responsive grid editor

Layout mode lets you resize components using the 12-column responsive grid. Switch to it using the toolbar. In Layout mode:

  • Each component shows resize handles on its right edge
  • Drag the handle to make the component span more or fewer columns (1–12)
  • A breakpoint selector in the toolbar lets you switch between Desktop, Tablet, and Mobile views
  • You can set different column spans for each breakpoint — a component might be 8 columns on desktop but 12 columns (full width) on mobile

This is how editors create two-column, three-column, and other layouts without any CSS knowledge.

Preview mode — see it as a visitor would

Preview mode hides all authoring overlays and toolbars. The page looks and behaves exactly as it will on the live website. Use it to:

  • Check that the layout looks correct before publishing
  • Test hover states, animations, and interactive elements
  • Verify that links go to the right destinations
  • Check how the page looks on mobile (use browser DevTools device simulation while in Preview mode)
Preview mode uses the Author server's renderer — it does not go through the Dispatcher or Publish. Very rarely there are rendering differences between Author preview and the live site, usually related to Dispatcher caching configurations.

Part 5 — CRXDE

CRXDE Lite — the developer's most important tool

CRXDE Lite is a browser-based IDE for the JCR. It lets you browse the entire content repository, inspect node properties, edit files directly, and run queries. As a developer, you will use this constantly for debugging and testing.

How to open it
URLhttp://localhost:4502/crx/de
The CRXDE interface — every panel
Left panel — tree
The full JCR tree. Click the triangle next to any node to expand it and see child nodes. Click a node to select it. The tree always starts at the root /.
Center panel — file editor
When you click a file node (like card.html), its content appears here in a code editor. You can edit it and press Ctrl+S to save directly to AEM without Maven. Very useful for quick fixes.
Bottom right — Properties tab
When you click any node, this panel shows all its JCR properties as a table: property name, type, value. This is where you see sling:resourceType, jcr:title, and all dialog-saved values.
Bottom right — Node tab
Shows the current node path and lets you create child nodes, set node types, and manage node hierarchy.
Query panel
Run JCR queries (XPath or SQL2) to find nodes matching criteria. E.g., find all components with a specific resourceType across the whole repository.
What you use CRXDE for as a developer
1
Debug rendering problems
Navigate to /content/mysite/en/home/jcr:content/root/mycomponent. Click the Properties tab. Check: does sling:resourceType point to the right component? Are the dialog values saved correctly? Is the component node even there?
2
Check deployed components
Navigate to /apps/mysite/components/card. Verify the folder is there. Click card.html to see the HTL code. Click _cq_dialog to see the dialog definition. This confirms your Maven deployment worked.
3
Quick edit without Maven
Click any .html file in /apps/mysite/components/. Edit the code in the center panel. Press Ctrl+S. The change is live immediately — no Maven build needed. Use this for rapid iteration on HTL changes.
4
Verify template structure
Navigate to /conf/mysite/settings/wcm/templates/. Click a template node. See its structure node, initial content node, and policies.

Part 5 — OSGi console

The OSGi console — the Java runtime manager

The OSGi (Felix) console is a separate web UI from AEM's normal interface. It shows you everything about the Java runtime — which bundles are running, what services are registered, what the current OSGi configurations are. As a developer, you check this every single time you deploy Java code.

The most important sub-pages
Bundleshttp://localhost:4502/system/console/bundles
Configshttp://localhost:4502/system/console/configMgr
Serviceshttp://localhost:4502/system/console/services
Log supporthttp://localhost:4502/system/console/slinglog
Bundles page — the most used page

Every piece of Java code in AEM runs inside an OSGi bundle. AEM's own code is hundreds of bundles. Your project's Java code is one bundle (your core module builds into one JAR = one bundle).

After every Maven deployment, go here and search for your bundle (e.g. "mysite.core"). You should see:

Active
Your bundle is running correctly. All Java imports resolved. Your Sling Models, Services, and Servlets are available. This is what you want.
Installed
The bundle JAR is in AEM but could not start. Usually means a Java import could not be resolved — you are importing a class from a package that no other bundle exports. Click your bundle name to see which imports are failing.
Resolved
The bundle was running but was stopped. Can happen after a dependency bundle was updated. Usually resolves on its own. Try clicking the Start button.
Configuration manager (configMgr)

Shows all OSGi service configurations. Each service that has a @Designate(ocd=...) annotation appears here with a form for its config values. You can:

  • Edit config values live without any deployment — changes take effect immediately
  • See what the current value of any service config is
  • Create new config instances for factory configurations

Use this during development to quickly test different config values. Once you find the right values, add them to your .cfg.json files in the project for permanent deployment.

Part 5 — Package manager

Package manager — install and export content

Packages are zip files that contain JCR content. The Package Manager lets you install packages (to add content or code to AEM) and build packages (to export JCR content for moving to another environment).

How to open it
URLhttp://localhost:4502/crx/packmgr
What a package contains

A package is essentially a zip file with a specific folder structure inside. It contains JCR nodes serialized as XML files (using Apache Jackrabbit's vault format). When you install a package, AEM reads these XML files and writes the nodes into the JCR at the correct paths.

Your Maven build produces packages. When you run mvn clean install -PautoInstallPackage, Maven builds a content package (zip) and calls the Package Manager API to install it into AEM. You can see your installed packages in the Package Manager UI.

Common tasks
1
Install a package
Click "Upload Package" → select your zip file → click OK → click "Install" next to the uploaded package → AEM extracts and installs all JCR nodes.
2
Install WKND
Download aem-guides-wknd.all-x.x.x.zip from github.com/adobe/aem-guides-wknd → upload here → install. WKND will be available at /content/wknd/us/en.
3
Export content to share
Click "New Group" → "New Package" → set Name and Group → under "Filters" add a path (e.g. /apps/mysite/components/card) → click Build → click Download to get the zip.

Part 6 — Setup

Local AEM setup — complete step by step

Setting up AEM on your local machine for the first time. Follow these steps exactly in order. Each step has a specific purpose.

What you need before you start
Java 11 (JDK)
AEM 6.5 requires Java 11. Not Java 8, not Java 17. Download from adoptium.net. After installing, run java -version in your terminal. Should say 11.x.x.
Maven 3.6+
Build tool. Download from maven.apache.org. After installing, run mvn -version. Should say 3.6.x or higher.
AEM JAR + license
Two files: aem-author-p4502.jar and license.properties. Get from Adobe's software distribution portal or your company's shared drive.
VS Code
Code editor. Download from code.visualstudio.com. Install extensions: "HTL" (Adobe), "AEM Sync" (Adobe), "XML Language Support".
Git
Version control. Download from git-scm.com. Your AEM project will be in a Git repository.
Setting up AEM
1
Create a folder for AEM
Create a folder like ~/aem/author/. Put both the JAR file and license.properties inside this folder. Both files must be in the same folder.
2
Start AEM Author
Open terminal, navigate to the folder: cd ~/aem/author. Run: java -jar aem-author-p4502.jar -gui. A small window appears showing startup progress.
3
Wait for startup
The first startup extracts AEM and initializes the repository. Takes 5–15 minutes depending on your machine. You'll see "repository started" in the terminal when ready.
4
Open AEM in browser
Go to http://localhost:4502. You see the AEM login screen. Login with admin / admin.
5
Explore the UI
You are now on the AEM start page. Click "Sites" to see the Sites console. There are no sites yet — it is empty until you install WKND or deploy your project.
Memory allocation (important)

AEM needs at least 4GB of RAM. By default Java allocates much less. Start AEM with explicit memory settings:

recommendedjava -Xmx4096m -jar aem-author-p4502.jar -gui

If AEM crashes with "OutOfMemoryError" or becomes very slow, this is why. Increase to -Xmx6144m if you have 16GB RAM.

Part 6 — Authoring process

The content authoring process — end to end

Understanding the complete authoring workflow — from a marketing request to a live page — shows how all the pieces you have learned connect together in the real world.

A full example — creating a product launch page
1
Marketing requests a new page
The product team sends a brief: "We need a landing page for our new credit card launching next Monday. It needs a hero banner, three feature cards, an FAQ section, and an 'Apply Now' form." This is a content request — no developer needed.
2
Content editor opens Sites console
Navigates to http://localhost:4502/sites.html/content/mybank/personal/cards/. This is where new credit card pages go.
3
Creates the page
Clicks Create → Page → selects "Campaign Landing Page" template → sets title "New Platinum Card" and name "new-platinum-card" → clicks Create → clicks Open to go to the editor.
4
Builds the page in the editor
Opens the component browser (left panel). Drags a "Hero Banner" component into the main Layout Container. Clicks Configure → fills in headline "Introducing the Platinum Card", uploads the hero image, sets the button text "Apply Now" and link to the form. Saves.
5
Adds more components
Drags three "Feature Card" components below the banner. Configures each one with a benefit icon, title, and description. Drags an FAQ component and adds questions and answers. Switches to Layout mode and makes the feature cards appear side by side (4 columns each on desktop, 12 columns on mobile).
6
Previews and submits for review
Switches to Preview mode to check the page looks correct. Then uses the workflow feature to submit the page for Legal Review and Brand Approval.
7
Approval workflow
The legal team receives a notification. They open the page, read all the copy. The interest rate claims are correct so they approve. The brand team checks the images and color usage. They approve. The manager gets a final notification.
8
Scheduled publish
The editor schedules the page to publish at midnight Sunday. AEM's scheduler fires at midnight, replicates the page to Publish, and the Dispatcher serves it immediately.
9
The page is live
Customers visiting www.mybank.com/personal/cards/new-platinum-card see the new page. The URL, layout, content, and design are all exactly as the editor built them — without a single developer involved.
This is the value of AEM
From content brief to live page — the entire process above happened with zero developer involvement. The developer's job was done earlier: building the "Hero Banner" component, the "Feature Card" component, and the "Campaign Landing Page" template. Once built, these become a toolkit that non-technical teams can use to build any number of pages, any time, instantly.
