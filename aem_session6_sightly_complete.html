Part 1 — what & why

Why HTL — the problem JSP had

HTL (HTML Template Language) replaced JSP in AEM 6.1. Before understanding HTL, you need to understand why JSP was a problem — because every design decision in HTL was made to solve a specific JSP problem.

What JSP looked like (and why it was bad)
old card.jsp — the JSP way
<!-- Problem 1: Java code mixed inside HTML --> <% String title = properties.get("title", ""); String link = properties.get("ctaLink", ""); boolean show = StringUtils.isNotBlank(title); %> <!-- Problem 2: No auto-escaping — developer must remember to escape --> <h2><%= xssAPI.encodeForHTML(title) %></h2> <!-- Problem 3: Easy to forget escaping — instant XSS vulnerability --> <a href="<%= link %>">Click</a> ← missing encode = XSS hole

JSP mixed Java scriptlets directly inside HTML. This caused three big problems: the templates were hard to read (HTML and Java interleaved), business logic crept into templates making code hard to maintain, and developers frequently forgot to escape output creating XSS security vulnerabilities.

What HTL does differently — solving all three problems
JSP problem
HTL solution
Java code mixed inside HTML. Hard to read and maintain.
No Java allowed in templates at all. Logic goes in Sling Models. Templates are pure HTML with data-sly-* attributes.
Developers must manually remember to call xssAPI.encodeForHTML() on every output.
Auto-escaping is the default. ${model.title} is automatically HTML-escaped. You cannot accidentally create an XSS vulnerability.
JSP syntax is not valid HTML. HTML editors, linters, and design tools cannot process JSP files.
HTL is valid HTML. data-sly-* attributes are valid HTML5 custom data attributes. Any HTML tool can process an HTL file.
Why this matters
A bank's AEM site had 200 JSP templates. A security audit found 14 XSS vulnerabilities — all caused by missing encodeForHTML() calls. After migrating to HTL, those vulnerabilities became structurally impossible — HTL escapes everything by default. The developers could not create those vulnerabilities even if they tried. This is the single most important security improvement HTL brought to AEM.
Part 1 — what & why

HTL basics — how it works

HTL is just HTML with two additions: expressions (${...}) and block statements (data-sly-* attributes). That is the entire language. Everything else is the HTML you already know.

The two constructs of HTL
Expressions
${ ... }
Output a value from the model or JCR. Written inside attribute values or between HTML tags. Example: ${model.title} outputs the value returned by getTitle(). Auto-escaped for security.
Block statements
data-sly-*
HTML attributes that add logic to an element — conditionals, loops, variable assignments, includes. They are processed by AEM's template engine and removed from the final HTML output. Example: data-sly-test="${model.title}" hides the element if title is null/empty.
How AEM processes an HTL file
1
A page is requested
Browser requests /content/mysite/en/home.html. Sling resolves the component via sling:resourceType.
2
AEM finds and compiles the HTL file
AEM reads card.html, compiles it into a Java class on first request, and caches the compiled version.
3
data-sly-use loads the model
data-sly-use.model="com.mysite.core.models.CardModel" — AEM instantiates CardModelImpl, runs @PostConstruct, and binds it to the variable model.
4
Expressions are evaluated
${model.title} calls model.getTitle() and escapes the return value as HTML text. The expression is replaced with the escaped string in the output.
5
Block statements are processed
data-sly-test conditionals are evaluated. data-sly-list loops are expanded. data-sly-include files are inlined. All data-sly-* attributes are removed from the final HTML.
6
Clean HTML is returned
The browser receives pure HTML with no trace of HTL syntax. No data-sly-* attributes visible in browser DevTools.
File naming rule — most commonly broken rule

The HTL filename MUST match the component folder name. If your component folder is /apps/mysite/components/card/, the HTL file must be named card.html. Not template.html. Not component.html. Exactly card.html. When Sling resolves a resource with sling:resourceType="mysite/components/card", it looks for a script named after the last segment of the path: card. With extension .html, it becomes card.html.

Part 1 — what & why

The <sly> tag — the invisible wrapper

The <sly> tag is a special HTL-only element that is always removed from the final HTML output. It exists only to carry data-sly-* attributes when you do not want an actual HTML element.

The problem <sly> solves

Every data-sly-* attribute must be placed on an HTML element. But sometimes you need logic that should not produce any HTML element at all. For example, you want to include a file without wrapping it in a <div>:

problem — unwanted wrapper div
<!-- This produces an unwanted <div> in the HTML output --> <div data-sly-include="footer.html"></div> <!-- Renders: <div>...footer content...</div> --> <!-- CORRECT: use <sly> — produces no wrapper element --> <sly data-sly-include="footer.html"/> <!-- Renders: ...footer content... (no wrapper) -->
When to use <sly>
  • With data-sly-include to include a file without a wrapper element
  • With data-sly-resource to render a sub-resource without a wrapper
  • With data-sly-call to call a template without a wrapper
  • With data-sly-use when you only need to load an object but not output anything
  • With data-sly-set to define a variable without outputting anything
  • With data-sly-list inside a <ul> when the list element itself comes from the include
Part 2 — expression language

Expression syntax — everything about ${ }

HTL expressions are written as ${...}. They can appear in attribute values or in element content. Every expression is automatically escaped based on context.

Basic expression forms
expression syntax forms
<!-- Simple property access --> ${model.title} ${properties.jcrTitle} ${currentPage.name} <!-- Dot notation for nested access --> ${model.author.name} ${currentPage.properties.navTitle} <!-- Bracket notation for dynamic property names --> ${model['dynamic-key']} ${properties['jcr:title']} ← use brackets for property names with colons or hyphens <!-- String concatenation --> ${'Hello, ' + model.name + '!'} <!-- Ternary operator --> ${model.isActive ? 'active' : 'inactive'} <!-- Null coalescing / default value --> ${model.title || 'Default Title'} <!-- Boolean NOT --> ${!model.isEmpty} <!-- Comparison --> ${model.count == 0} ${model.count != 0} ${model.count > 3} <!-- Options (@ syntax) for context and formatting --> ${'some/path' @ context='uri'} ${model.price @ format='currency', locale='en_IN'} ${model.date @ format='dd MMM yyyy'}
Expression options — the @ syntax

The @ symbol inside an expression introduces options. Options modify how the expression value is processed or formatted.

context
Override the auto-detected escape context. ${link @ context='uri'}. Required for all href and src values.
default
Fallback value if the expression evaluates to null or empty. ${model.title @ default='Untitled'}
join
Join array elements with a separator. ${model.tags @ join=', '} → "technology, news, aem"
i18n
Translate a string through AEM's i18n translation system. ${'Add to Cart' @ i18n}
locale
Override locale for formatting. ${model.price @ format='currency', locale='en_IN'}
timezone
Override timezone for date formatting. ${model.date @ format='HH:mm', timezone='Asia/Kolkata'}
Part 2 — expression language

Operators in HTL expressions

HTL supports a subset of JavaScript/Java operators inside expressions. Knowing which operators work and how they behave prevents common bugs.

All supported operators
operators — all types with examples
// ── LOGICAL operators ──────────────────────────────────────────── ${a && b} ← AND: true if both truthy ${a || b} ← OR: first truthy value, or last value ${!a} ← NOT: inverts truthiness // ── COMPARISON operators ───────────────────────────────────────── ${a == b} ← equal (type coerced) ${a != b} ← not equal ${a < b} ← less than (use &lt; in attributes) ${a > b} ← greater than ${a <= b} ← less than or equal ${a >= b} ← greater than or equal // ── TERNARY operator ───────────────────────────────────────────── ${condition ? 'yes' : 'no'} ${model.count > 0 ? model.count : 'none'} // ── STRING concatenation ───────────────────────────────────────── ${'Hello ' + model.name} ${'card card--' + model.theme} // ── ARITHMETIC ─────────────────────────────────────────────────── ${model.price * 1.18} ← multiply (rarely used in templates) ${model.count + 1} ← add // ── FALSY values in HTL (data-sly-test hides element for these) ── // null, undefined, false, 0, "", [], {} // Empty string "" is falsy — useful: data-sly-test="${model.title}" // hides element when title is null OR empty string // ── TRUTHY values ──────────────────────────────────────────────── // Any non-empty string, any non-zero number, true, // non-empty array, non-empty object
The OR operator as a default value

The || operator returns the first truthy value, not just true. This makes it a natural default value operator in HTL:

using || for defaults
<!-- If model.title is null or empty, use "Untitled" --> <h2>${model.title || 'Untitled'}</h2> <!-- If ctaText is not set, use "Learn More" --> <a href="${model.link @ context='uri'}"> ${model.ctaText || 'Learn More'} </a> <!-- Chained defaults --> <title>${currentPage.navigationTitle || currentPage.title || 'My Site'}</title>
Part 2 — expression language

Global objects — what is available in every HTL file

HTL automatically provides a set of global objects in every template file without any data-sly-use. These give you access to the current page, JCR properties, request, and authoring context.

properties
The JCR properties of the current resource (the component's content node). ${properties.title} reads the title property directly — without a Sling Model. Use for simple components that do not need computed logic.
pageProperties
The JCR properties of the current page's jcr:content node. ${pageProperties['jcr:title']}. Useful for page components that need to read page-level metadata.
inheritedPageProperties
Like pageProperties but also searches parent pages if the property is not found on the current page. Used for inherited configurations like analytics IDs.
currentPage
The AEM com.day.cq.wcm.api.Page object for the current page. Gives access to currentPage.name, currentPage.title, currentPage.path, currentPage.depth, currentPage.parent, currentPage.listChildren().
currentNode
The current JCR node as a javax.jcr.Node object. Less commonly used than currentResource.
currentResource
The Sling Resource object for the current component node. Use to get resource path (currentResource.path), resource type (currentResource.resourceType), and child resources.
request
The SlingHttpServletRequest. Access URL parameters (request.getParameter('q')), selectors, suffix, and request attributes.
response
The SlingHttpServletResponse. Rarely used directly in HTL templates.
wcmmode
The current WCM mode. Boolean flags: wcmmode.edit (author edit mode), wcmmode.preview (preview mode), wcmmode.disabled (publish/live mode). Use to show authoring helpers only in edit mode.
resolver
The Sling ResourceResolver. Use to map paths: ${resolver.map('/content/mysite/en/home')} returns the shortened URL.
Using globals in practice
global objects in use
<!-- Page title from currentPage (no model needed) --> <title>${currentPage.title} | My Site</title> <!-- Show helper text only in author edit mode --> <p class="author-hint" data-sly-test="${wcmmode.edit && !model.title}"> Please configure this component. </p> <!-- Read a JCR property directly without Sling Model --> <h1>${properties['jcr:title']}</h1> <!-- Current resource path (useful for debug or AJAX calls) --> <div data-path="${currentResource.path}">
Part 2 — expression language

Contexts — automatic XSS protection

Every HTL expression is automatically escaped based on the context where it appears. Context means: where in the HTML is this value being placed? Different locations in HTML require different escaping to be safe.

All contexts — what they do
text (auto)
HTML content between tags. Auto-detected. Encodes < → &lt;, > → &gt;, & → &amp;. Most common. Used automatically when expression is between tags.
html
Allows HTML markup to pass through unescaped. Use ONLY for content from AEM's RTE (Rich Text Editor). Never for user-submitted data. ${model.bodyText @ context='html'}
attribute (auto)
HTML attribute value. Auto-detected when expression is inside an attribute. Encodes quotes and angle brackets.
uri
URLs used in href, src, action, data-*. URL-encodes, AND blocks javascript: protocol. Required: ${model.link @ context='uri'}
styleToken
CSS class names. Allows only valid CSS identifier characters (letters, digits, hyphens, underscores). Blocks CSS injection. ${model.theme @ context='styleToken'}
scriptToken
JavaScript identifiers. Allows only valid JS identifier characters. Use in inline scripts.
scriptString
String value inside a JavaScript string literal. Escapes quotes and backslashes for safe inclusion in JS.
number
Numeric values in attributes. Allows only digits and decimal point. ${model.width @ context='number'}
attributeName
When setting a dynamic attribute name. Validates against a whitelist of allowed attribute name characters.
elementName
Used with data-sly-element. Validates the tag name against allowed HTML element names.
unsafe
No escaping at all. Use ONLY for completely static developer-controlled strings. NEVER for anything editors or users can input.
Part 2 — expression language

Display context decisions — which context to use where

HTL auto-detects context in most cases, but some require explicit specification. Here are the decision rules for every common situation.

context decisions — complete reference
<!-- ✓ BETWEEN TAGS: auto-detected as text context --> <h2>${model.title}</h2> <p>${model.description}</p> <!-- ✓ HREF / SRC / ACTION: MUST use uri context --> <a href="${model.link @ context='uri'}"> <img src="${model.imageUrl @ context='uri'}"/> <form action="${model.endpoint @ context='uri'}"> <!-- ✓ CSS CLASSES: use styleToken for dynamic class names --> <div class="card card--${model.theme @ context='styleToken'}"> <!-- ✓ RICH TEXT from RTE: use html context --> <div class="rte">${model.bodyText @ context='html'}</div> <!-- ✓ REGULAR HTML ATTRIBUTES: auto-detected --> <img alt="${model.altText}"/> ← text context auto <input placeholder="${model.hint}"/> <!-- ✓ DATA ATTRIBUTES with URLs --> <div data-endpoint="${model.apiUrl @ context='uri'}"> <!-- ✗ WRONG — no context on href: javascript: URLs can pass through --> <a href="${model.link}"> ← missing @ context='uri' <!-- ✗ WRONG — html context on user-submitted content --> <div>${userInput @ context='html'}</div> ← XSS risk
Part 3 — basic block statements

data-sly-use — loading objects into the template

data-sly-use is the most important HTL block statement. It loads an external object — a Sling Model, a Java class, or a JavaScript object — and binds it to a variable name you can use in the template.

Syntax — the variable name comes from the attribute
data-sly-use syntax
<!-- Pattern: data-sly-use.VARIABLENAME="CLASSNAME" VARIABLENAME = how you reference the object in this template CLASSNAME = fully-qualified Java class or JavaScript file path --> <!-- Load a Sling Model --> <div data-sly-use.model="com.mysite.core.models.CardModel"> ${model.title} <!-- "model" is the variable name --> </div> <!-- Load multiple objects --> <div data-sly-use.card="com.mysite.core.models.CardModel" data-sly-use.page="com.mysite.core.models.PageHelper"> ${card.title} ${page.breadcrumb} </div> <!-- On a <sly> tag (no wrapper in output) --> <sly data-sly-use.nav="com.mysite.core.models.NavigationModel"/> <nav>${nav.title}</nav> <!-- Note: nav is available even after the <sly> element --> <!-- Pass options to the Use object (advanced) --> <div data-sly-use.list="com.mysite.core.models.ListModel" data-sly-use.list.limit="5" data-sly-use.list.path="/content/mysite">
How HTL finds the class

When you write data-sly-use.model="com.mysite.core.models.CardModel", HTL first checks if this is a fully-qualified Java class name. If so, it uses SlingHttpServletRequest.adaptTo() or Resource.adaptTo() to get an instance. If the string ends in .js, it loads it as a JavaScript Use object. If it is a relative path, it resolves it relative to the current HTL file.

Part 3 — basic block statements

data-sly-text — setting element content

data-sly-text sets the text content of an HTML element, replacing all its child nodes with the evaluated value. Unlike inline expressions, it makes placeholder text possible in the source HTML.

data-sly-text — all usage patterns
<!-- Pattern 1: Replace element content --> <h2 data-sly-text="${model.title}"> Placeholder headline (replaced at runtime) </h2> <!-- Renders: <h2>Actual Title from JCR</h2> --> <!-- The placeholder text is NEVER shown — data-sly-text overwrites it --> <!-- Pattern 2: Expression directly (no data-sly-text needed) --> <h2>${model.title}</h2> <!-- Both are equivalent for simple cases --> <!-- When to prefer data-sly-text: When your source HTML should have meaningful placeholder text for design purposes, static mockups, or frontend development --> <button data-sly-text="${model.ctaLabel}">Add to Cart</button> <!-- "Add to Cart" is the placeholder used in static mockups. At runtime it is replaced with whatever the model returns. --> <!-- data-sly-text with html context for rich text --> <div class="rte" data-sly-text="${model.bodyHtml @ context='html'}"> <p>Body content goes here</p> </div>
Part 3 — basic block statements

data-sly-attribute — setting attributes dynamically

data-sly-attribute sets or removes HTML attributes at runtime. Use it when you need to conditionally set an attribute or set multiple attributes from model values.

data-sly-attribute — all patterns
<!-- Pattern 1: Set a single attribute --> <a data-sly-attribute.href="${model.link @ context='uri'}"> Click here </a> <!-- Equivalent to href="${model.link @ context='uri'}" --> <!-- Pattern 2: Conditionally set an attribute --> <a href="${model.link @ context='uri'}" data-sly-attribute.target="${model.newTab ? '_blank' : ''}" data-sly-attribute.rel="${model.newTab ? 'noopener noreferrer' : ''}"> <!-- When value is empty string, AEM removes the attribute entirely --> <!-- So rel="" becomes no rel attribute in the output HTML --> <!-- Pattern 3: Set a map of attributes from model --> <div data-sly-attribute="${model.attributeMap}"> <!-- model.getAttributeMap() returns a Map<String,String> All map entries become HTML attributes on this element --> <!-- Pattern 4: Remove an attribute --> <button disabled="disabled" data-sly-attribute.disabled="${model.isEnabled ? '' : 'disabled'}"> <!-- disabled="" removes the attribute (button is enabled) disabled="disabled" adds the attribute (button is disabled) --> <!-- Pattern 5: aria attributes --> <button data-sly-attribute.aria-expanded="${model.isExpanded}" data-sly-attribute.aria-controls="${model.controlsId}"> Toggle </button>
Part 3 — basic block statements

data-sly-test — conditional rendering

data-sly-test is HTL's conditional. If the expression is falsy (null, false, empty string, 0, empty array), the entire element and all its children are NOT rendered — they are completely absent from the HTML output, not just hidden with CSS.

data-sly-test — all patterns
<!-- Pattern 1: Show element only if value exists --> <h3 data-sly-test="${model.title}"> ${model.title} </h3> <!-- If model.title is null or "" — the <h3> is NOT in the HTML output at all --> <!-- Pattern 2: Negation — show when empty/null --> <p class="empty-state" data-sly-test="${!model.title}"> No title configured </p> <!-- Pattern 3: Store result in a variable for reuse --> <sly data-sly-test.hasContent="${model.title || model.description}"/> <!-- Now use hasContent elsewhere in the template --> <div class="card" data-sly-test="${hasContent}">...</div> <p class="empty" data-sly-test="${!hasContent}">Empty</p> <!-- Pattern 4: Complex conditions --> <div data-sly-test="${model.isActive && model.hasPermission}"> Visible only when both conditions are true </div> <!-- Pattern 5: Author-mode helper (show only in edit mode) --> <div class="author-hint" data-sly-test="${wcmmode.edit && !model.title}"> Please add a title. </div> <!-- Pattern 6: Comparing values --> <span class="badge" data-sly-test="${model.count > 0}"> ${model.count} </span>
There is no "else" in HTL. To show alternative content, use two data-sly-test attributes — one for the positive case and one with ! (NOT) for the negative case. Or store the result with data-sly-test.variable and reuse it.
Part 3 — basic block statements

data-sly-list — looping over collections

data-sly-list repeats the element it is on for each item in a collection. The collection can be a Java List, an array, a Collection, or anything Iterable.

data-sly-list — all patterns
<!-- Pattern 1: Basic list --> <ul> <li data-sly-list.item="${model.navItems}"> <!-- "item" is the loop variable — access item's properties --> <a href="${item.url @ context='uri'}">${item.label}</a> </li> </ul> <!-- Pattern 2: The itemList object for loop metadata --> <ul> <li data-sly-list.item="${model.items}"> <!-- itemList is auto-created: {item}List gives loop metadata --> <!-- itemList.index = 0-based position (0, 1, 2...) --> <!-- itemList.count = 1-based position (1, 2, 3...) --> <!-- itemList.first = true on first item --> <!-- itemList.last = true on last item --> <!-- itemList.odd = true on odd items (1st, 3rd...) --> <!-- itemList.even = true on even items (2nd, 4th...) --> <!-- itemList.middle = true on items that are not first or last --> <li class="item ${itemList.first ? 'item--first' : ''} ${itemList.last ? 'item--last' : ''} ${itemList.odd ? 'item--odd' : 'item--even'}"> <span class="item__number">${itemList.count}</span> ${item.title} </li> </li> </ul> <!-- Pattern 3: Custom loop variable name --> <div data-sly-list.product="${model.products}"> <!-- Loop var is "product", metadata is "productList" --> <div class="product ${productList.first ? 'product--featured' : ''}"> <h3>${product.name}</h3> <p>${product.price}</p> </div> </div> <!-- Pattern 4: Empty collection check before the loop --> <sly data-sly-test.hasItems="${model.items}"/> <ul data-sly-test="${hasItems}"> <li data-sly-list.item="${model.items}">${item.title}</li> </ul> <p data-sly-test="${!hasItems}">No items found.</p>
Part 3 — basic block statements

data-sly-repeat — repeating the element itself

data-sly-repeat is like data-sly-list but with one key difference: it repeats the element it is placed on, not just its content. Use it when the element IS the repeated item, not a container for it.

data-sly-list vs data-sly-repeat — the key difference
<!-- data-sly-LIST: repeats CHILDREN of the element --> <ul> <li data-sly-list.item="${model.items}">${item.title}</li> </ul> <!-- Output: <ul> <li>Item 1</li> <li>Item 2</li> <li>Item 3</li> </ul> --> <!-- data-sly-REPEAT: repeats the ELEMENT ITSELF --> <div class="card" data-sly-repeat.card="${model.cards}"> <h3>${card.title}</h3> </div> <!-- Output: <div class="card"><h3>Card 1</h3></div> <div class="card"><h3>Card 2</h3></div> <div class="card"><h3>Card 3</h3></div> (no wrapper — just three separate divs) --> <!-- data-sly-repeat also has the cardList metadata object --> <tr data-sly-repeat.row="${model.rows}" class="${rowList.odd ? 'row--odd' : 'row--even'}"> <td>${rowList.count}</td> <td>${row.name}</td> </tr>
Use data-sly-list when the repeated items go inside a container element (like <ul> or <select>). Use data-sly-repeat when the element itself is the thing being repeated (like a grid of <div> cards or table rows).
Part 3 — basic block statements

data-sly-set — defining local variables

data-sly-set creates a template-local variable from an expression. Use it to avoid computing the same expression multiple times, to create readable aliases for long expressions, or to pre-compute a value used in several places.

data-sly-set — patterns and use cases
<!-- Pattern 1: Create a variable from an expression --> <sly data-sly-set.greeting="${'Hello, ' + model.name + '!'}"/> <h2>${greeting}</h2> <p aria-label="${greeting}">...</p> <!-- greeting is computed once, used twice --> <!-- Pattern 2: Readable alias for a long expression --> <sly data-sly-set.isPromo="${model.badge == 'promo' && model.isActive}"/> <div class="card ${isPromo ? 'card--promo' : ''}"> <span class="badge" data-sly-test="${isPromo}">Promo</span> </div> <!-- Pattern 3: Avoid long repeated property chains --> <sly data-sly-set.author="${model.article.metadata.author}"/> <span>By ${author.name}</span> <img src="${author.photo @ context='uri'}" alt="${author.name}"/> <!-- Pattern 4: Boolean flag for readability --> <sly data-sly-set.showCta="${model.ctaLink && model.ctaText}"/> <div class="card__footer ${showCta ? 'has-cta' : ''}"> <a href="${model.ctaLink @ context='uri'}" data-sly-test="${showCta}"> ${model.ctaText} </a> </div>
Part 4 — advanced block statements

data-sly-include — including other HTL files

data-sly-include inlines the content of another HTL file at the current position. The included file has access to the same global objects and the same data-sly-use variables from the including file.

data-sly-include — patterns
<!-- Pattern 1: Include a sibling file (relative path) --> <sly data-sly-include="card-header.html"/> <!-- Looks for card-header.html in the SAME component folder --> <!-- Pattern 2: Include from a subfolder --> <sly data-sly-include="partials/footer.html"/> <!-- Pattern 3: Include with a wrapper element --> <footer class="page-footer" data-sly-include="footer.html"/> <!-- Renders: <footer class="page-footer">...footer content...</footer> --> <!-- Pattern 4: Conditional include --> <sly data-sly-include="${model.showPromo ? 'promo-banner.html' : 'default-banner.html'}"/> <!-- What the included file can access: - All global objects: properties, currentPage, wcmmode, etc. - All data-sly-use variables defined BEFORE the include tag - Its own data-sly-use objects defined within itself -->
data-sly-include vs data-sly-resource
data-sly-include
data-sly-resource
Includes another HTL file's source as a partial/fragment. Same resource context. Used for breaking up a large template into smaller maintainable files.
Renders a JCR resource (a different node) using its own sling:resourceType rendering script. Creates a completely new resource context. Used for sub-components.
Part 4 — advanced block statements

data-sly-resource — rendering sub-resources

data-sly-resource renders a JCR resource using that resource's own sling:resourceType rendering script. It is how you embed one component inside another component's template.

data-sly-resource — all patterns
<!-- Pattern 1: Render a child resource by its name --> <sly data-sly-resource="image"/> <!-- Renders the "image" child node of the current resource. AEM finds /content/mysite/.../mycomponent/image, reads its sling:resourceType, and renders it. --> <!-- Pattern 2: Override the resource type --> <sly data-sly-resource="${'image' @ resourceType='mysite/components/image'}"/> <!-- Renders the "image" child node using the specified resourceType, ignoring any resourceType stored on the node. --> <!-- Pattern 3: Add selectors --> <sly data-sly-resource="${'image' @ selectors='thumbnail'}"/> <!-- Renders image with the ".thumbnail" selector — calls image.thumbnail.html in the image component --> <!-- Pattern 4: Render with wrapper element --> <div class="card__image" data-sly-resource="image"/> <!-- Renders: <div class="card__image">...image output...</div> --> <!-- Pattern 5: Use absolute path --> <sly data-sly-resource="${'/content/mysite/shared/header'}"/> <!-- Renders the shared header from an absolute JCR path -->
Part 4 — advanced block statements

data-sly-template — defining reusable template blocks

data-sly-template defines a named block of HTML that can be called multiple times from anywhere in the same file with different parameters. Like a function in programming. The block is not rendered where it is defined — only when called with data-sly-call.

data-sly-template — define and call
<!-- DEFINE a template at the top of the file --> <!-- It is NOT rendered here. Template elements with data-sly-template are always removed from the output. --> <template data-sly-template.badge="${@ text, type}"> <!-- Parameters declared with @ in the definition --> <!-- type and text are available inside the template body --> <span class="badge badge--${type @ context='styleToken'}"> ${text} </span> </template> <!-- CALL the template with specific parameter values --> <sly data-sly-call="${badge @ text='New', type='success'}"/> <sly data-sly-call="${badge @ text='Sale', type='warning'}"/> <sly data-sly-call="${badge @ text=model.badgeLabel, type=model.badgeType}"/> <!-- TEMPLATE WITH NO PARAMETERS --> <template data-sly-template.spinner> <div class="spinner" aria-hidden="true"> <div class="spinner__ring"></div> </div> </template> <sly data-sly-call="${spinner}"/> <!-- TEMPLATE INSIDE A LOOP --> <template data-sly-template.cardItem="${@ item}"> <div class="card"> <h3>${item.title}</h3> <p>${item.description}</p> </div> </template> <div class="card-grid"> <sly data-sly-list.item="${model.items}"> <sly data-sly-call="${cardItem @ item=item}"/> </sly> </div>
Calling templates from another file

You can define templates in one file and call them from another using data-sly-use:

cross-file template calls
<!-- In templates/shared.html — a library of reusable templates --> <template data-sly-template.icon="${@ name, size}"> <svg class="icon icon--${name @ context='styleToken'} icon--${size @ context='styleToken'}"> <use href="#icon-${name @ context='styleToken'}"/> </svg> </template> <!-- In card.html — import and call the template --> <sly data-sly-use.shared="templates/shared.html"/> <sly data-sly-call="${shared.icon @ name='arrow', size='sm'}"/>

Part 5 — Use API

Use API — what it is and why it exists

The Use API is the mechanism HTL uses to load logic objects into templates. It is the bridge between the template (HTML) and the business logic (Java or JavaScript). Without it, templates would be limited to only the global objects HTL provides.

Why the Use API was invented

HTL deliberately has no scripting capabilities — no if statements based on complex business logic, no method calls, no variable assignments inside the template. This is by design: templates should only render data, not compute it.

But complex components need computed data — formatted prices, filtered lists, resolved URLs, permission checks. The Use API provides a clean way to get that computed data into the template without breaking the separation of concerns.

The two Use API options
Java Use API
A Java class (Sling Model) loaded via data-sly-use. The class can do anything Java can do — query JCR, call APIs, format data, check permissions. Most powerful and recommended for production use.
JavaScript Use API
A JavaScript file loaded via data-sly-use. Runs server-side in AEM using the Rhino JavaScript engine (AEM 6.5) or Nashorn. Simpler than Java for light logic but less powerful. Good for prototyping or very simple computed values.
Part 5 — Use API

Java Use API — Sling Models as Use objects

The Java Use API in AEM is implemented through Sling Models. Any class annotated with @Model can be loaded via data-sly-use. HTL calls the public getter methods as expressions in the template.

How HTL connects to a Sling Model
the connection between HTL and Sling Model
<!-- In card.html --> <div data-sly-use.model="com.mysite.core.models.CardModel"> ${model.title} <!-- calls model.getTitle() --> ${model.description} <!-- calls model.getDescription() --> ${model.hasLink} <!-- calls model.isHasLink() --> ${model.items.size} <!-- calls model.getItems() then .size() --> </div> // In CardModel.java (interface) public interface CardModel { String getTitle(); // HTL: ${model.title} String getDescription(); // HTL: ${model.description} boolean isHasLink(); // HTL: ${model.hasLink} — note: "is" not "get" List<Item> getItems(); // HTL: ${model.items} } // HTL → Java naming convention: // ${model.title} → getTitle() (capitalize, prepend get) // ${model.firstName} → getFirstName() // ${model.hasLink} → isHasLink() (boolean uses "is" prefix) // ${model.empty} → isEmpty() (boolean, AEM uses "is" prefix) // ${model.items.size} → getItems().size() (chained)
What the Sling Model receives at runtime

When HTL processes data-sly-use.model="com.mysite.core.models.CardModel", it calls request.adaptTo(CardModel.class) or resource.adaptTo(CardModel.class) depending on the model's adaptables configuration. Sling finds the registered implementation (CardModelImpl) and calls its constructor, injects all @ValueMapValue and other annotated fields, then calls @PostConstruct. The resulting object is bound to model.

Part 5 — Use API

JavaScript Use API — server-side JS in HTL

The JavaScript Use API lets you write a JavaScript file that runs server-side in AEM and provides computed values to the HTL template. It is simpler to set up than a full Sling Model but less powerful. Good for prototypes, simple logic, or cases where a Sling Model is overkill.

How to create a JavaScript Use object
card.js — JavaScript Use object in same folder as card.html
"use strict"; // All global HTL objects are available in the JS Use object: // - properties (JCR properties of current resource) // - pageProperties (JCR properties of current page) // - currentPage (com.day.cq.wcm.api.Page) // - currentNode (javax.jcr.Node) // - request (SlingHttpServletRequest) // - resource (org.apache.sling.api.resource.Resource) // - resolver (ResourceResolver) use(function() { // "this" gives access to all HTL globals var props = this.properties; var page = this.currentPage; var request = this.request; // Read JCR properties var title = props.get("title", ""); var price = props.get("price", 0); var isActive = props.get("active", false); // Compute derived values var formattedPrice = price > 0 ? "₹" + price.toFixed(2) : "Free"; var greeting = title ? "Hello, " + title + "!" : "Welcome!"; // Access page information var pagePath = page ? page.getPath() : ""; // Return object — properties become available as ${model.xxx} return { title: title, formattedPrice: formattedPrice, greeting: greeting, isActive: isActive, pagePath: pagePath }; });
Using the JavaScript Use object in HTL
card.html — loading the JS Use object
<!-- Load card.js from the same folder (no path = same folder) --> <div data-sly-use.model="card.js" class="card"> <h2>${model.greeting}</h2> <p class="price">${model.formattedPrice}</p> <span class="status" data-sly-test="${model.isActive}">Active</span> </div> <!-- Load a JS file from a different path --> <div data-sly-use.nav="helpers/navigation.js"> ${nav.currentSection} </div>
The JavaScript Use API is limited. It cannot use npm modules. It runs in Java's Rhino/Nashorn JS engine which is not a browser environment. For complex logic (JCR queries, OSGi service calls, permissions), use a Sling Model instead.
Part 5 — Use API

Use API patterns — when to use what

Choosing between the Java Use API (Sling Model), JavaScript Use API, and global objects is a common decision. Here are the guidelines.

Use global properties directly
For the simplest components that just read 1–3 JCR properties with no transformation. ${properties.title} directly. No Sling Model needed.
Use JavaScript Use API
For simple components with light computed logic — string formatting, simple conditions, basic calculations. When you want to prototype quickly without writing Java.
Use Java Sling Model
For all production components. Supports unit testing, OSGi service injection, complex JCR queries, multifield reading, permission checks, and any logic requiring Java APIs. The default choice for all serious development.
Multiple Use objects in one template
loading multiple Use objects
<!-- You can load multiple Use objects on any element --> <div data-sly-use.card="com.mysite.core.models.CardModel" data-sly-use.nav="com.mysite.core.models.NavigationModel" data-sly-use.utils="utils.js"> ${card.title} ${nav.currentSection} ${utils.formattedDate} </div> <!-- Or load them on <sly> tags before use --> <sly data-sly-use.card="com.mysite.core.models.CardModel"/> <sly data-sly-use.nav="com.mysite.core.models.NavigationModel"/> <h1>${card.title}</h1> <nav>${nav.currentSection}</nav>

Debugging HTL — what to do when things go wrong

HTL fails silently in many cases — expressions return null without errors, templates render blank output, models return nothing. Here is how to diagnose and fix every common HTL problem.

Problem 1 — ${model.title} renders nothing
1
Check that the value is in the JCR
CRXDE → navigate to the component's content node → Properties tab → is title property there with a value? If not, the dialog did not save it.
2
Check the Sling Model loaded
Add ${model} or ${model.class.name} to the template temporarily. If it outputs nothing, the model is null. Check: is the bundle Active? Does Sling-Model-Packages include your model package?
3
Check the @Model resourceType
In your Java class, @Model(resourceType="mysite/components/card"). Does this exactly match your component folder path relative to /apps? It is case-sensitive.
4
Check the getter name
If you write ${model.title}, Java must have a method named exactly getTitle(). If the getter is getCardTitle(), use ${model.cardTitle}.
Problem 2 — data-sly-test always hides the element
debugging conditionals
<!-- Temporarily output the value to see what it is --> <p style="color:red;font-size:12px"> DEBUG: model.title = [${model.title}] | model.hasLink = [${model.hasLink}] | model.items.size = [${model.items.size}] </p> <!-- Output [null] means the getter returned null Output [] means an empty string Both are falsy in data-sly-test -->
Problem 3 — wrong HTML in output (XSS issues)

If link URLs are garbled (e.g. href="&#x2F;content&#x2F;") you forgot @ context='uri'. If rich text is showing as literal HTML entities (&lt;p&gt;) you need @ context='html'. If CSS classes have characters stripped, use @ context='styleToken'.
