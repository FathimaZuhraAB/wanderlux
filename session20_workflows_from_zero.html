What problem does Session 20 solve?
Every company has business processes — steps that must happen in a certain order, involving multiple people, before something is approved or published. Without automation, this is done manually by email, which is slow, error-prone, and has no audit trail.

AEM Workflows automate these business processes directly inside your CMS.
The scenario we use throughout this session
You work for a large bank that uses AEM for its website. Every page about a financial product — loans, credit cards, savings accounts — must go through a strict approval process before it goes live. Why? Because incorrect financial information on a bank's website can cause legal and regulatory problems.

The process is:

1. Author writes the content and submits it for review
2. Legal team checks it for compliance issues
3. Marketing manager approves the final look
4. System automatically publishes it live
5. Compliance officer gets an email confirmation

Without workflows, this is 5 emails and 3 days. With AEM Workflows, it is automated, tracked, and takes hours.

Key terms — the vocabulary of AEM Workflows
Every AEM workflow conversation uses these exact words. Learn them before anything else — they are the foundation that everything else builds on.
Real life analogy first
🏗️
Think of building a house. There is a blueprint that describes all the steps: pour foundation → build walls → roof → plumbing → electrical → inspection → move in.

The blueprint is just a plan — it doesn't build anything by itself. When a specific family wants their house built, a construction instance starts. The house under construction is the payload — the thing being worked on. Each task assigned to a tradesperson is a work item — the electrician gets a task, the plumber gets a task.

Workflow Model = blueprint. Workflow Instance = one actual construction. Payload = the house. WorkItem = one task assigned to one tradesperson.
Every term defined clearly
Workflow Model
The blueprint. The template. The diagram. It defines the steps in order, who does what, and what happens at each step. You create it once in the Workflow Model editor (a visual drag-and-drop designer). For our bank: the model defines the Legal → Marketing → Publish → Notify sequence. The model itself does nothing until it is triggered.
Workflow Instance
One running execution of a workflow model. When author Priya submits the "Credit Card page" for review, a new workflow instance is created. When author Ravi submits the "Loan page", another separate instance starts. Both run the same model independently at the same time. Each instance has its own state, its own current step, its own history.
Payload
The thing the workflow is processing. For a content approval workflow, the payload is the JCR path of the page being reviewed — e.g. /content/bank/en/credit-card. Every step in the workflow can read this path to know what content to act on. In asset processing workflows, the payload is the asset path.
WorkItem
One task inside a running workflow instance. When the workflow reaches a "Participant Step" (a step that needs human action), a WorkItem is created and placed in the assigned person's AEM Inbox. The legal officer opens their inbox, sees "Please review: Credit Card page", approves it, and the workflow moves to the next step. WorkItems only exist for human steps — automated steps have no WorkItem.
Workflow Data
A key-value bag that travels with the workflow instance. Each step can read from and write to this bag. Example: the Legal step writes "legalApproved=true" into workflow data. The Marketing step reads it and adds "marketingApproved=true". The final step reads both values to confirm all approvals before publishing.
Launcher
A rule that automatically starts a workflow when something happens. Example: "When any node is created or modified under /content/dam, automatically start the DAM Update Asset workflow." Without launchers, someone would have to manually start every workflow. You manage launchers at Tools → Workflow → Launchers.
How these terms connect in one picture
Modelblueprint (created once)
Launchertriggers it
Instanceone running job
Payloadpage being processed
WorkItemstasks in inboxes
The Model exists permanently. A Launcher fires when a page is submitted. That creates an Instance which works on the Payload (the page path). As it reaches human steps, WorkItems land in people's AEM Inboxes.

Workflow step types
A workflow model is made of steps connected in a sequence. There are different types of steps — some run code automatically, some wait for a human, some split into multiple paths. Understanding these is essential for designing real workflows.
Step type 1: Process Step — fully automated
⚙️
What it is: A step that runs Java code automatically with zero human involvement. It starts, runs your code, and immediately moves to the next step.

Real life analogy: A stamp machine on an assembly line. It doesn't wait for anyone — it automatically stamps every item that passes through and moves it along.

Real AEM examples: Send an email notification, activate (publish) a page, update a metadata property, call an external REST API, generate a PDF, validate that required fields are filled in.
The bank's "Publish Page" step is a Process Step — when both approvals are done, it automatically calls AEM's Replication API to publish the page. No human clicks publish. It just happens.
Step type 2: Participant Step — waits for a human
👤
What it is: A step that pauses the workflow and assigns a task (WorkItem) to a specific user or group. The workflow waits — could be hours or days — until that person acts.

Real life analogy: A form that needs a manager's signature. It sits on their desk until they sign it. Nothing moves forward until the signature is there.

Real AEM examples: "Legal team, please review this page." The workflow stops. A WorkItem appears in the Legal group's AEM Inbox. Any member of the Legal group can open it, read the page, and click Approve or Reject.
AEM Inbox — what the legal officer sees
Review required: Credit Card Cashback page
Assigned by workflow: Bank Content Approval • Submitted by: priya.author@bank.com
Review required: Home Loan Interest Rates page
Assigned by workflow: Bank Content Approval • Submitted by: ravi.author@bank.com
Step type 3: Dynamic Participant Step — human assigned by code
Same as Participant Step but the assignee is decided by Java code at runtime, not hardcoded in the model.

Example: A page about "Home Loans" should go to the Home Loans team. A page about "Credit Cards" should go to the Cards team. A Dynamic Participant Step runs a small piece of Java code that reads the page's category and returns the right group name. The WorkItem is then sent to that group's inbox automatically.

Step type 4: OR Split — take one path based on a condition
🔀
What it is: An OR Split evaluates conditions and routes the workflow down exactly ONE branch — like an if/else in your code.

Real life analogy: A passport control officer checks your visa. If you have a valid visa → go to arrivals. If your visa is expired → go to the immigration office. You go to one place — not both.

Bank example: After the legal review, did they approve or reject?
— If approved → go to Marketing review
— If rejected → go back to Author with comments
How the condition is defined: Each branch of an OR Split has an ECMA Script condition. For the "Approved" branch: workflowData.getMetaDataMap().get("legalDecision","").equals("approved"). The workflow evaluates each branch's script and takes the first one that returns true.
Step type 5: AND Split — run multiple branches at the same time
⚡
What it is: An AND Split runs multiple branches in parallel simultaneously and waits until ALL of them complete before moving forward.

Real life analogy: A house inspection requires three specialists to check it simultaneously: a structural engineer, an electrician, and a plumber. All three work at the same time. Only after all three sign off does the sale proceed.

Bank example: The page needs two independent reviews simultaneously — Legal AND Compliance — to save time. Both review in parallel. When both are done, the workflow merges back and moves to Marketing.
AND Split vs OR Split in one sentence: AND Split = "do all of these paths and wait for all" (parallel). OR Split = "do only one of these paths based on a condition" (branching).

Workflow Console — managing your workflows
The Workflow Console is your control room for all workflow activity. It is where you design workflow models, monitor running instances, debug failures, and configure launchers. Go to Tools → Workflow in AEM navigation.
The four sections of the Workflow Console
1
Models — design your workflow blueprints
Path: Tools → Workflow → Models. This is where you design workflows using a visual drag-and-drop editor. You see all existing models (DAM Update Asset, Request for Activation, etc.). Click "Create" to build a new one. Drag step types from the left panel, connect them with arrows, configure each step by double-clicking it.

Important: editing a model creates a draft version. You must click "Sync" to make the changes take effect for new workflow instances. Existing running instances continue on the old version.
2
Instances — monitor everything running right now
Path: Tools → Workflow → Instances. Shows all currently running workflow instances. For each you can see: which model it uses, what the payload path is, which step it is currently on, how long it has been running, and who it is waiting for.

Critical for debugging: if a workflow is stuck (running for 3 days on the same step), come here to investigate. You can Terminate a stuck instance, Suspend it to pause it, or Resume a suspended one.
3
Archive — history of completed workflows
Path: Tools → Workflow → Archive. Shows all completed (or failed) workflow instances. You can see the full execution history: which steps ran, what happened at each, timestamps. Useful for audits — "show me the approval history for this page before it went live."

Warning: the archive grows indefinitely. In production you must schedule a "Purge Workflow" job to delete old archive entries, otherwise the repository fills up and AEM slows down.
4
Launchers — rules that auto-start workflows
Path: Tools → Workflow → Launchers. A launcher watches for JCR events (node created, modified, deleted) and automatically starts a workflow when the event matches. Example: "When a node of type dam:Asset is created under /content/dam, start the DAM Update Asset workflow." The DAM Update Asset workflow never needs to be manually started because a launcher handles it.
Most common production problem: stuck workflow instances
🚨
The problem: You go to Workflow Instances and see 2,000 instances all stuck on the same step, all showing "Running" for days.

Common causes:
— A custom process step threw an uncaught exception and the workflow has no error handling
— A Participant Step was assigned to a user who left the company and nobody can complete it
— An external API call in a process step is timing out indefinitely

How to fix:
1. Go to Tools → Workflow → Instances
2. Filter by the affected model
3. Select all stuck instances
4. Click Terminate (kills them cleanly)
5. Fix the root cause in your code
6. Redeploy and test with one instance first
Triggering a workflow manually from Java code
Sometimes you need to start a workflow from your own code — for example, a scheduled job that reviews all pages older than 1 year and starts a "review and refresh" workflow on each one.

@Reference WorkflowService workflowService — OSGi injects the WorkflowService automatically. This is AEM's central workflow engine service. You use it to get a WorkflowSession which is your handle into the workflow system — similar to how ResourceResolver is your handle into the JCR.
L7-9
workflowService.getWorkflowSession(session) — creates a WorkflowSession tied to the given JCR Session. The resolver.adaptTo(Session.class) extracts the underlying JCR session from your ResourceResolver. The WorkflowSession carries the same user permissions as that session — so your service user needs appropriate permissions to start workflows.
L11-12
wfSession.getModel("/var/workflow/models/bank-content-approval") — loads a WorkflowModel by its path in the JCR. All workflow models are stored under /var/workflow/models/. The path is the model's ID — you can find it in the Workflow Models console by hovering over the model name. If the path is wrong, this returns null and the next line throws a NullPointerException.
L14-15
wfSession.newWorkflowData("JCR_PATH", pagePath) — creates the WorkflowData object that carries the payload. "JCR_PATH" is the payload type — it tells the workflow engine that the payload is a JCR node path (as opposed to TEXT or other types). pagePath is the actual path of the page to be reviewed, e.g. "/content/bank/en/credit-card". Every step in the workflow will receive this path as its payload.
L17
wfSession.startWorkflow(model, data) — actually starts a new workflow instance using the given model and payload data. AEM creates the Instance node under /var/workflow/instances/, runs the first step, and the workflow begins. This call returns immediately — the workflow runs asynchronously in the background.

Custom workflow process step — writing your own automated step
This is the most important coding concept in Session 20. A custom process step is a Java class that AEM runs automatically as one step in a workflow. You write the logic — send email, publish page, validate content, call an API — and AEM calls it at the right moment.
The interface you must implement
🔌
To create a custom process step, your Java class must implement the WorkflowProcess interface. This interface has one method: execute(). AEM calls this method when your step runs in a workflow.

You also register your class with a process.label property. This label appears as a selectable option in the Workflow Model editor's process step configuration — so workflow designers can choose your step by name without knowing any Java.
Full example: "Validate and Publish" process step for the bank
This step checks that the page has a title, description, and author field filled in. If all checks pass, it publishes the page. If any check fails, it terminates the workflow with an error message.

L29-32
execute(WorkItem, WorkflowSession, MetaDataMap) throws WorkflowException — this is the ONE method from the WorkflowProcess interface that you must implement. AEM calls it when this step runs. The throws WorkflowException declaration is important — if you throw a WorkflowException, AEM marks the workflow instance as "failed" and stops it. This is how you signal that something went wrong and the workflow should not continue.
L34-35
workItem.getWorkflowData().getPayload().toString() — getting the payload path. Every workflow step receives the WorkItem which contains the WorkflowData which contains the Payload. For a content workflow the payload is a JCR path string. You call toString() because getPayload() returns a generic Object type — it could theoretically be other things, but for JCR_PATH type payloads it is always a path String.
L38-39

L57-61
Validation check before publishing — this is the core value of a custom process step. The standard AEM "Activate Page" action doesn't validate anything. Your custom step can enforce business rules before publishing. Here: if title, description, or author is empty, throw WorkflowException. Don't wrap it or log it again. catch(Exception e) — catches any other unexpected exception (LoginException, RepositoryException, RuntimeException). Wraps it in a WorkflowException so AEM properly marks the instance as failed with the error details.

How workflows get triggered
There are three ways a workflow can start in AEM. Understanding all three is important because in real projects you use different methods for different situations.
Method 1: Manually by an editor in the AEM UI
🖱️
How: An author opens a page in Sites console → selects the page → clicks the "Manage Publication" or "Request Approval" button → chooses the workflow model → clicks Start.

When to use: For content approval workflows. The author consciously decides "I am done with this page and want it reviewed." This is the most common trigger for human-driven approval processes.

What happens: AEM creates a workflow instance with the selected page as the payload. The first step begins immediately.
Method 2: Automatically via a Launcher
⚡
How: You configure a Launcher rule at Tools → Workflow → Launchers. You define: "When a node of type X is created/modified under path Y, start workflow model Z."

When to use: For asset processing (DAM Update Asset workflow). Every upload automatically triggers it. Also used for auto-expiry workflows — "when the jcr:content/expiryDate property is modified, start the Review Before Expiry workflow."

What happens: AEM watches for JCR events matching your launcher rule. The moment the event fires, a new workflow instance starts automatically — no human action needed.
Key launcher properties to know:
Event Type: Node Created / Node Modified / Node Removed
Node Type: e.g. dam:Asset or cq:Page
Path: e.g. /content/dam — only watch this subtree
Workflow Model: which model to start
Condition: optional ECMA script for fine-grained filtering
Method 3: Programmatically from Java code
You already saw the full code for this on the previous tab. Here's when to use each approach:
Manual (UI)

Content approval, translation requests, legal review — anything a human initiates consciously when their work is done.

Launcher

Asset processing, auto-expiry, sync triggers — anything that should happen automatically whenever content changes in a certain way.

Programmatic

Batch operations, scheduled jobs, external system triggers — when your code makes a business decision that a workflow should start.

Passing custom data into a workflow — MetaDataMap
Sometimes your step needs more information than just the payload path. MetaDataMap is the workflow's data bag — you can store and read any key-value pair from it across all steps.

Common mistakes with workflows
!
Not closing the ResourceResolver inside a process step
Just like scheduled jobs, workflow process steps must use try-with-resources for ResourceResolvers. Workflows can run hundreds of times — each leaked resolver slowly kills AEM.
!
Swallowing exceptions instead of throwing WorkflowException
If your process step catches an exception and doesn't re-throw it as WorkflowException, the workflow thinks the step succeeded and happily moves to the next step — including publishing broken content. Always throw WorkflowException on failure.
!
Not purging the workflow archive
Every completed workflow is stored in the JCR archive. After months in production, this can grow to millions of nodes and slow AEM significantly. Always configure a periodic Purge Workflow scheduled job to delete old archive entries.
