Session 4 — custom components

Component anatomy — understanding the whole picture first

Before writing a single line of code, understand what a component actually is, what files it needs, and how data flows through the entire system from editor to browser.

What is a component

A component is a reusable piece of UI. Editors drag it onto a page. They fill in a dialog form to configure it. AEM saves those values to the JCR. A Java class (Sling Model) reads those values. An HTML template (HTL) renders them as HTML. The browser displays the result.

Think of it as a form with logic. The dialog is the form. The Sling Model is the logic. The HTL file is the output template.

Data flow — from editor click to browser HTML
Editor types "Summer Sale" in dialog title field
→
AEM saves title="Summer Sale" to JCR node
→
Sling Model reads it with @ValueMapValue
→
HTL outputs ${model.title}
→
Browser sees <h2>Summer Sale</h2>
The four required files — every component needs these
.content.xml

Registers the component with AEM. Sets the name editors see in the component browser, which group it belongs to, and which parent component it extends.

card.html (HTL)

The HTML output template. Filename MUST match the component folder name. Runs when the component is rendered on a page.

_cq_dialog/.content.xml

Defines the "Configure" dialog editors fill in. Each form field saves a JCR property. The most important file for editor experience.

CardModel.java (Sling Model)

Java class that reads JCR properties and prepares them for the HTL template. The bridge between stored data and rendered output.

Real example
A bank's "Interest Rate Card" component shows a product name, rate percentage, and apply button. The developer builds it once — four files. Editors use it 200 times across the site, each configured with a different product and rate through the dialog. One component, 200 pages, zero code duplication.

Component folder — every file and its job

Every component is a folder inside /apps/mysite/components/. The folder name becomes the component's identifier. Here is every file that can go inside, what it does, and when you need it.

Location in your project
ui.apps/src/main/content/jcr_root/apps/mysite/components/ └── card/ ← component folder name = component ID ├── .content.xml ← required: component registration ├── card.html ← required: HTL rendering template ├── _cq_dialog/ ← required: author configure dialog │ └── .content.xml ├── _cq_editConfig/ ← optional: author toolbar behaviour │ └── .content.xml ├── _cq_design_dialog/ ← optional: template policy settings │ └── .content.xml ├── _cq_htmlTag/ ← optional: customise wrapper element │ └── .content.xml └── thumbnail.png ← optional: icon in component browser (80×80px)
Each file — what it does and when you need it
.content.xml
The component node itself. Required. Sets jcr:primaryType=cq:Component, jcr:title (browser name), componentGroup (browser group), sling:resourceSuperType (parent). Without this, AEM does not know the component exists.
card.html
The HTL rendering file. Required. Must match the folder name exactly. AEM runs this file when the component is requested. Loads the Sling Model and renders HTML.
_cq_dialog/
The configure dialog. Required for any component that has editable content. Defines form fields — textfields, selects, pathfields. Each field saves a JCR property when the editor saves the dialog.
_cq_editConfig/
Author toolbar behaviour. Optional but recommended. Controls which buttons appear in the blue edit bar, what text shows when the component is empty, and what refreshes after editing.
_cq_design_dialog/
Template-level policy form. Optional. Opened by template authors (not page editors) through the template editor. Settings apply to ALL instances of this component on ALL pages using that template.
thumbnail.png
80×80 PNG icon shown in the component browser. Optional but strongly recommended — it helps editors identify components visually in a long browser list.
Start with just .content.xml, card.html, and _cq_dialog/.content.xml. Add editConfig and design dialog later when needed. Never create empty placeholder files.

.content.xml — the component definition file

This file is the component's identity. It tells AEM the component exists, what it is called, where it appears in the browser, and which parent component it extends. Without this file, the component is invisible.

.content.xml — complete annotated example
<?xml version="1.0" encoding="UTF-8"?> <jcr:root xmlns:cq="http://www.day.com/jcr/cq/1.0" xmlns:jcr="http://www.jcp.org/jcr/1.0" xmlns:sling="http://sling.apache.org/jcr/sling/1.0" jcr:primaryType="cq:Component" ↑ REQUIRED — tells AEM this is a component definition node. Without this exact value, AEM ignores the folder. jcr:title="Card" ↑ The name editors see in the component browser. Make it human-readable: "Product Card", "Hero Banner", "FAQ Item". jcr:description="A card with image, title, description and CTA button" ↑ Tooltip shown when editors hover over the component in the browser. Write a clear one-sentence description of what it does. componentGroup="My Site - Content" ↑ Which tab/group in the component browser. ALL your custom components should share the same group name. This name must EXACTLY match what you allow in the template policy. Case-sensitive. One typo and your component disappears from the browser. sling:resourceSuperType="core/wcm/components/teaser/v2/teaser" ↑ Parent component to inherit from. Inheriting from Core Components means you get accessibility, image handling, and policies for free. Your component only overrides what it needs to change. Remove this line if you are building 100% from scratch. />
The componentGroup rule — the most common mistake

The value in componentGroup must exactly match what is allowed in the template's Layout Container policy. If you write "My Site - Content" here but the policy allows "my site - content", your component will be invisible in the browser — no error, just gone. Always copy-paste the group name, never type it by hand.

Other optional properties
cq:isContainer
Set to {Boolean}true if this component can hold other components (like a custom Layout Container).
allowedParents
Restricts where this component can be dropped. Example: ["*/parsys","*/responsivegrid"] — only inside containers.
allowedChildren
For container components — which component types can be dropped inside this component.
noDecoration
Set to {Boolean}true to suppress AEM's auto-generated wrapper div around the component output.

HTL — every attribute explained with examples

HTL (HTML Template Language) is AEM's templating language. It replaced JSP. It is HTML with special data-sly-* attributes that add logic. All output is automatically XSS-escaped. No Java code inside the template — only in the Sling Model.

Why HTL instead of JSP
JSP (old)
HTL (modern)
Java scriptlets mixed directly in HTML. Easy to create XSS vulnerabilities. Business logic and presentation entangled.
No Java in templates. Auto-escaping prevents XSS by default. Logic goes in Sling Models. Clean separation of concerns.
A complete card.html with every attribute
card.html — every HTL attribute demonstrated
<!-- ① data-sly-use: load a Sling Model and bind to variable "model" --> <div data-sly-use.model="com.mysite.core.models.CardModel" class="card"> <!-- ② data-sly-test: only renders element if condition is truthy --> <div class="card__image-wrap" data-sly-test="${model.imageUrl}"> <img src="${model.imageUrl @ context='uri'}" alt="${model.imageAlt}" loading="lazy"/> </div> <!-- ③ data-sly-test with stored result for reuse --> <div class="card__body"> <span class="card__eyebrow" data-sly-test="${model.eyebrow}"> ${model.eyebrow} </span> <!-- ④ data-sly-element: change the tag name dynamically --> <h3 data-sly-element="${model.headingLevel}" class="card__title"> ${model.title} </h3> <!-- If model.headingLevel = "h2", renders <h2> instead of <h3> --> <!-- ⑤ data-sly-text: set element text content (replaces inner HTML) --> <p class="card__description" data-sly-text="${model.description}"> Placeholder text removed at runtime </p> <!-- ⑥ data-sly-attribute: set HTML attributes dynamically --> <a class="card__cta" data-sly-attribute.href="${model.ctaLink @ context='uri'}" data-sly-attribute.target="${model.newTab ? '_blank' : ''}" data-sly-attribute.rel="${model.newTab ? 'noopener' : ''}" data-sly-test="${model.hasLink}"> ${model.ctaText} </a> </div> </div> <!-- ⑦ data-sly-list: loop over a collection --> <ul class="tag-list" data-sly-test="${model.tags}"> <li data-sly-list.tag="${model.tags}"> <!-- tagList.index=0-based, tagList.count=1-based, tagList.first, tagList.last, tagList.odd, tagList.even --> <span class="tag ${tagList.first ? 'tag--first' : ''}"> ${tagList.count}. ${tag} </span> </li> </ul> <!-- ⑧ data-sly-repeat: repeat the element itself (not parent) --> <div class="card" data-sly-repeat.item="${model.cards}"> <h3>${item.title}</h3> </div> <!-- Renders multiple <div class="card"> elements, one per card --> <!-- ⑨ data-sly-include: include another HTL file as a partial --> <sly data-sly-include="partials/card-footer.html"/> <!-- ⑩ data-sly-resource: render another JCR resource by its own template --> <sly data-sly-resource="${'image' @ resourceType='mysite/components/image'}"/> <!-- ⑪ data-sly-template + data-sly-call: reusable template functions --> <template data-sly-template.badge="${@ text, type}"> <span class="badge badge--${type @ context='styleToken'}">${text}</span> </template> <!-- Call it: --> <sly data-sly-call="${badge @ text='New', type='success'}"/> <!-- ⑫ data-sly-set: define a local variable --> <sly data-sly-set.greeting="${'Hello, ' + model.name}"/> <p>${greeting}</p> <!-- ⑬ data-sly-unwrap: keep children but remove the wrapper element --> <div data-sly-unwrap="${!wcmmode.edit}"> children here <!-- wrapper div removed in publish mode --> </div>

HTL contexts — XSS safety explained

HTL automatically escapes all output to prevent Cross-Site Scripting (XSS). But the type of escaping depends on WHERE in the HTML the value appears. You must specify the right context for each output location.

All contexts and when to use each
text (default)
HTML element content. Auto-applied when you write ${model.title} between tags. HTML-encodes < > &. Most common context.
html
Use when the value contains trusted HTML markup you want to render. ${richText @ context='html'}. Only use for content from AEM's own RTE — never for user-submitted data.
uri
Use for href and src attribute values. URL-encodes the value AND blocks javascript: protocol. Without this, a malicious link could inject JavaScript.
attributeName
Use when setting dynamic attribute names (not values). Validates the name against a whitelist of safe HTML attributes.
styleToken
Use for CSS class names. Allows only valid CSS identifier characters. Prevents CSS injection.
scriptToken
Use for JavaScript identifiers. Only allows valid JS identifier characters.
number
Use for numeric values in attributes. Only allows digits and decimal points.
unsafe
No escaping at all. NEVER use for content that comes from editors or users. Only for completely static developer-controlled strings.
Correct context usage in practice
HTL context examples
<!-- CORRECT: uri context for links --> <a href="${model.link @ context='uri'}">Click</a> <!-- CORRECT: html context for rich text --> <div class="rte-content">${model.bodyText @ context='html'}</div> <!-- CORRECT: styleToken for dynamic CSS classes --> <div class="card card--${model.theme @ context='styleToken'}"> <!-- WRONG: missing uri context — javascript: URLs pass through --> <a href="${model.link}">Unsafe</a> <!-- WRONG: using html context for unknown/user content --> <div>${userComment @ context='html'}</div> ← XSS vulnerability

Dialog — structure and all field types

The dialog is the popup form that appears when an editor clicks the wrench (Configure) icon on a component. It is a single XML file that defines a Granite UI form. Every field in it saves a JCR property when the editor saves.

The name="./property" rule — the most important rule

Every dialog field has a name property. This tells AEM where to save the value in the JCR. The ./ prefix means "relative to this component's content node". So name="./title" saves the value as a title property at the component's JCR path.

For example: a card on the homepage saves to /content/mysite/en/home/jcr:content/root/card. The dialog's name="./title" saves to /content/mysite/en/home/jcr:content/root/card/title. Your Sling Model's @ValueMapValue private String title; reads that exact property. The field name in XML and the Java field name must match.

When to use REFRESH_PAGE vs REFRESH_SELF
REFRESH_PAGE — reload the whole page
REFRESH_SELF — reload only this component
Use when your component affects page structure, other components, or generates navigation. Slower but ensures everything is consistent.
Use for most components that only affect their own output. Much faster — the page does not flash. Editors prefer this.

_cq_design_dialog — template-level settings

The design dialog is opened by template authors (not page editors) through the template editor policy panel. Settings here apply to all instances of this component on all pages using that template.

Design dialog vs regular dialog
Regular dialog (_cq_dialog)
Design dialog (_cq_design_dialog)
Per-instance settings. Each component on each page has different values. The editor sets these when they configure a specific component on a specific page.
Template-wide settings. Applied to ALL instances of this component on ALL pages using this template. Set by template authors once in the template editor.
Where values are stored

Regular dialog values → /content/mysite/en/page/jcr:content/root/card (per page, per component instance)

Design dialog values → /conf/mysite/settings/wcm/policies/ (shared across all pages using the template)

Sling Model — complete production example

The Sling Model is the Java class that bridges the JCR (where data is stored) and the HTL template (where data is rendered). It reads JCR properties, computes derived values, and exposes everything as Java getters.

core/src/main/java/com/mysite/core/models/impl/CardModelImpl.java
package com.mysite.core.models.impl; import com.mysite.core.models.CardModel; import org.apache.sling.api.resource.Resource; import org.apache.sling.models.annotations.*; import org.apache.sling.models.annotations.injectorspecific.*; import com.day.cq.wcm.api.Page; import org.apache.commons.lang3.StringUtils; import javax.annotation.PostConstruct; @Model( adaptables = { Resource.class, SlingHttpServletRequest.class }, adapters = { CardModel.class }, resourceType = "mysite/components/card", ↑ Must exactly match the component folder path relative to /apps defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL ↑ Missing properties return null instead of throwing an exception ) public class CardModelImpl implements CardModel { // ── JCR properties from the dialog ────────────────────────────── @ValueMapValue private String title; // reads the "title" JCR property saved by name="./title" in dialog @ValueMapValue private String description; @ValueMapValue private String ctaLink; @ValueMapValue @Default(values = "Read More") private String ctaText; // @Default provides a fallback if editor left the field empty @ValueMapValue private String theme; @ValueMapValue private boolean openInNewTab; @ValueMapValue(name = "jcr:title") private String pageTitle; // @ValueMapValue(name=) overrides the property name to read // ── AEM/Sling objects ──────────────────────────────────────────── @SlingObject private Resource currentResource; // the JCR node this component renders @SlingObject private ResourceResolver resourceResolver; @ScriptVariable private Page currentPage; // the AEM Page object this component is on // ── Child resource ─────────────────────────────────────────────── @ChildResource private Resource image; // injects the child JCR node named "image" under this component // ── Computed fields (not injected — calculated in @PostConstruct) ─ private boolean hasLink; private String themeClass; private String formattedTitle; // ── Lifecycle ──────────────────────────────────────────────────── @PostConstruct protected void init() { // Called AFTER all @ValueMapValue / @SlingObject fields are injected. // Safe to use all injected values here. // Put ALL computed logic here — not in getters. hasLink = StringUtils.isNotBlank(ctaLink); themeClass = StringUtils.isNotBlank(theme) ? "card--" + theme : ""; formattedTitle = StringUtils.isNotBlank(title) ? title.trim() : ""; } // ── Getters (called by HTL as ${model.xxx}) ────────────────────── // HTL calls getTitle() when you write ${model.title} // HTL calls isHasLink() when you write ${model.hasLink} @Override public String getTitle() { return formattedTitle; } @Override public String getDescription() { return description; } @Override public String getCtaLink() { return ctaLink; } @Override public String getCtaText() { return ctaText; } @Override public boolean isHasLink() { return hasLink; } @Override public String getThemeClass() { return themeClass; } @Override public boolean isOpenInNewTab() { return openInNewTab; } }

Inheritance — sling:resourceSuperType

Inheritance lets your component extend another. You get all its files (dialog, HTL, logic) automatically and only override what you need to change. This is the correct way to build on top of Adobe's Core Components.

How it works

Set sling:resourceSuperType in your component's .content.xml. When AEM resolves a file for your component, it checks your folder first. If the file is not found, it walks up to the parent and uses that file. Then the grandparent. And so on.

Script resolution order for inherited components
# Your component: /apps/mysite/components/fancy-teaser # sling:resourceSuperType = "core/wcm/components/teaser/v2/teaser" # When AEM looks for "fancy-teaser.html": 1. Check /apps/mysite/components/fancy-teaser/fancy-teaser.html ← YOUR file 2. Not found → check /apps/core/wcm/components/teaser/v2/teaser/teaser.html 3. Not found → check its parent, and so on # When AEM looks for "_cq_dialog": 1. Check /apps/mysite/components/fancy-teaser/_cq_dialog/ ← YOUR dialog 2. Not found → use Core Teaser's dialog (from /libs/core/...) # Result: you only need to create the files that DIFFER from the parent. # Everything else is inherited automatically.
Practical example — adding one field to Core Teaser

A bank needs a Teaser component with an extra "Disclaimer Text" field for regulatory compliance. Without inheritance, you copy 1000 lines of Core Teaser code. With inheritance:

mysite/components/bank-teaser/ ├── .content.xml ← sling:resourceSuperType="core/wcm/components/teaser/v2/teaser" └── _cq_dialog/ └── .content.xml ← your dialog with the extra disclaimer field added

That is all. Two files. AEM inherits: the HTL template, image handling, CTA logic, accessibility, SEO features, and all other dialog fields from Core Teaser. You only added the one new field. When Adobe updates Core Teaser, you automatically pick up the improvements.

Real example
A pharmaceutical company extends Core Teaser for their drug product cards. They add: a "Dosage Information" text field, a "Prescription Required" checkbox, and a "Side Effects" multifield. Four files total. Core Teaser handles everything else — image rendering, link handling, accessibility. The team focused on the business requirement, not rebuilding what Adobe already built.

Overlay and override — customising without touching /libs

The /libs folder contains AEM's own code. It is overwritten during every Service Pack upgrade. The rule is absolute: never edit anything in /libs. Overlay and override are the two safe mechanisms to customise AEM's built-in behaviour.

The golden rule

AEM resolves scripts and resources from /apps before /libs. If the same path exists in both /apps and /libs, AEM always uses the /apps version. This is why overlaying works — you copy a file to the same path under /apps and AEM uses your copy.

Overlay — same path in /apps shadows /libs
overlay example — customising AEM's 404 error page
# Original in /libs — DO NOT TOUCH this file: /libs/sling/servlet/errorhandler/404.jsp # Overlay in /apps — create this file with your customised version: /apps/sling/servlet/errorhandler/404.jsp # AEM resolves /apps first. # Your /apps version is used. /libs version is completely ignored. # When Adobe updates /libs, your customisation is unaffected. # Another example — overlaying an AEM admin UI component: # Original: /libs/cq/gui/components/authoring/page/toolbar/... # Overlay: /apps/cq/gui/components/authoring/page/toolbar/...
Override — pointing sling:resourceType to your component

Override means changing a content node's sling:resourceType to point to YOUR component instead of the default. Used when you want to completely replace a component's rendering for specific content.

override example — page component
# In your template's structure/jcr:content node: sling:resourceType = "mysite/components/page/base" # Instead of AEM's default page: # sling:resourceType = "core/wcm/components/page/v3/page" # Now YOUR page component renders all pages using this template. # The /libs Core page component is completely bypassed.
When to use which
Use overlay when
Use override/inheritance when
Customising AEM's own UI — admin screens, RTE plugins, author clientlibs. Things in /libs with no inheritance mechanism available.
Building your own components that extend Core Components. Extend via sling:resourceSuperType and only override specific files in your component folder.

Style system — one component, many looks

The Style System lets template authors define named visual styles for a component. Editors pick a style from a paint palette. AEM adds the corresponding CSS class to the component wrapper. Zero code change to add a new visual variant.

The full setup — developer, template author, and editor roles
1
Developer adds CSS classes to the clientlib
In your component's CSS: .card--featured { border: 2px solid gold; font-size: 1.2em; } and .card--minimal { background: none; padding: 0; }
2
Template author creates the styles in the policy
Template editor → click the Card component → Policy icon → Styles section → Add Style Group "Theme" → add: "Standard" (class: empty), "Featured" (class: card--featured), "Minimal" (class: card--minimal) → save
3
Editor picks a style on a specific component instance
On the page, click the Card → click the paintbrush icon in the toolbar → pick "Featured" → AEM adds card--featured to the wrapper div automatically
What AEM adds to the HTML
rendered HTML when "Featured" style is selected
<!-- Before style selected: --> <div class="card">...</div> <!-- After "Featured" style selected: --> <div class="card card--featured">...</div> <!-- AEM stores the selection in JCR as: cq:styleIds = ["style-id-for-featured"] On render, looks up the class name from the policy. -->

A telecom company has a "Plan Card" component. The developer builds one component. How Granite UI works

Every Granite UI component is itself an AEM component — it has a sling:resourceType. When AEM renders a dialog, it reads the XML nodes and renders each field by looking up its sling:resourceType. The full Granite UI component library path is: granite/ui/components/coral/foundation/. Everything under form/ is a form field. Everything under layouts/ is a layout container (tabs, accordion, columns). Everything under containers/ is a structural wrapper.

Touch UI dialog — what it is and how it appears

A Touch UI dialog is the popup form that slides in from the right of the page editor when an editor clicks the Configure (wrench) icon on a component. It is defined entirely as a JCR node tree in a single XML file.

What editors see

When an editor clicks the wrench on a component, a panel slides in from the right side of the screen (or a modal appears for some configurations). It shows a form with labeled fields — text inputs, dropdowns, path browsers, checkboxes. The editor fills in the fields and clicks the checkmark (top right of the dialog) to save. AEM immediately writes the values to the JCR content node.

What you create as a developer

One XML file at _cq_dialog/.content.xml. This XML defines the complete node tree that AEM interprets to render the dialog form. Every node in the XML becomes a JCR node. Every JCR node with a Granite UI sling:resourceType gets rendered as a UI component.

What happens when an editor saves the dialog
1
Editor fills in all fields and clicks the checkmark button
2
AEM's authoring servlet receives a POST request with all field values
3
AEM writes each value as a JCR property at the component's content node path. The property name is the last part of the field's name attribute (after ./).
4
AEM refreshes the component on the page using the listener configured in editConfig (REFRESH_PAGE or REFRESH_SELF)
5
The component re-renders using the new JCR property values — the editor sees the updated content immediately

Here is every Granite UI field type you will use, what it renders, and when to use each one. All paths start with granite/ui/components/coral/foundation/form/.

textfield
Single-line text input. Use for short text: title, label, URL slug, code. Properties: maxlength, emptyText (placeholder), required, validation.

Data source — dynamic dropdown options

A data source makes a Select field's options dynamic — fetched from a servlet, JCR query, or external API at dialog open time. Instead of hardcoding options in XML, the options come from live data.

When to use a data source

Use a data source when the dropdown options cannot be hardcoded because they change over time or come from external data:

  • A "Product Category" dropdown populated from the product catalogue in the JCR
  • A "Country" dropdown fetched from a configuration node
  • An "Author" dropdown populated from AEM user accounts
  • A "Layout Variant" dropdown populated from a remote API
Step 1 — reference the data source in the dialog
_cq_dialog/.content.xml — select with datasource

Step 2 — create the data source servlet in core

Multifield — what it is and why you need it

Multifield is a special Granite UI component that lets editors add, remove, and reorder a repeating group of fields. It solves the problem of "I need a list of items where each item has multiple properties".

The problem multifield solves

Without multifield: you need to know in advance how many items there will be and hardcode that many sets of fields. An editor who needs 8 team members but you only added 5 sets of fields is stuck.

With multifield: you define the fields for ONE item. Editors add as many items as they need using the "+ Add" button. They can remove items with the trash icon. They can reorder items by dragging the handle. The number of items is completely flexible at authoring time.

What multifield looks like to an editor

The editor sees a section in the dialog with a label (e.g. "Team Members") and an empty list. A blue "+ Add" button is at the bottom. When they click it, one set of sub-fields appears (e.g. Name, Role, Photo, LinkedIn URL). Clicking "+ Add" again adds another set. Each set has a trash icon to delete it and a drag handle to reorder.

How multifield data is stored in JCR

With composite={Boolean}true (the standard way), each added item becomes a child JCR node. For a component at /content/mysite/.../jcr:content/root/teamblock, adding three team members creates:

/content/mysite/en/home/jcr:content/root/teamblock/ └── team/ ← the multifield property group ├── item0/ ← first added item │ name = "Priya Sharma" │ role = "CTO" │ linkedin = "/in/priyasharma" ├── item1/ ← second item │ name = "Rohan Mehta" │ role = "Lead Engineer" └── item2/ ← third item name = "Anita Desai" role = "UX Designer"

Creating a multifield — complete XML

Here is the complete XML to create a multifield dialog section where each item has a name, role, and link. This is a common pattern for team members, social links, FAQ items, and product features.

_cq_dialog/.content.xml — multifield section
<!-- Add this inside your dialog's items section --> <team sling:resourceType="granite/ui/components/coral/foundation/form/multifield" fieldLabel="Team Members" fieldDescription="Add team members. Use the drag handle to reorder." composite="{Boolean}true" ↑ true = each item stored as a child node (use this always) false = stored as a flat string array (limited, avoid) name="./team" ↑ The JCR path where items are stored (creates a child node "team")> <!-- FIELD: this is the template for each repeated item --> <field jcr:primaryType="nt:unstructured" sling:resourceType="granite/ui/components/coral/foundation/container" name="./team"> <items jcr:primaryType="nt:unstructured"> <!-- Sub-field 1: member name --> <name sling:resourceType="granite/ui/components/coral/foundation/form/textfield" fieldLabel="Name" name="./name" required="{Boolean}true"/> <!-- Sub-field 2: member role --> <role sling:resourceType="granite/ui/components/coral/foundation/form/textfield" fieldLabel="Role / Title" name="./role"/> <!-- Sub-field 3: photo from DAM --> <photo sling:resourceType="granite/ui/components/coral/foundation/form/pathfield" fieldLabel="Photo" name="./photo" rootPath="/content/dam"/> <!-- Sub-field 4: LinkedIn URL --> <linkedin sling:resourceType="granite/ui/components/coral/foundation/form/textfield" fieldLabel="LinkedIn URL" name="./linkedin"/> </items> </field> </team>
Key properties on the multifield node
composite
Always set to {Boolean}true. This stores each item as a separate JCR child node. Without it, items are stored as a flat string array which is very limiting.
name
The path of the container node — ./team creates a child node called "team" under the component's content node. All item nodes are created inside this container.
field node
This is the template for one item. It must be a container type. The sub-field names inside use ./fieldname which is relative to each item's own node.

Reading multifield data in the Sling Model

The multifield stores items as child nodes under a container node. In your Sling Model you use @ChildResource to get the container, then iterate over its children, adapting each child to a model.

Option 1 — nested Sling Model (recommended)

Create a separate Sling Model for each multifield item. The parent model adapts each child resource to the item model.

TeamMember.java — interface for each item
package com.mysite.core.models; public interface TeamMember { String getName(); String getRole(); String getPhoto(); String getLinkedin(); boolean isHasLinkedin(); }
TeamMemberImpl.java — item model implementation
@Model( adaptables = Resource.class, adapters = { TeamMember.class }, defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL ) public class TeamMemberImpl implements TeamMember { @ValueMapValue private String name; @ValueMapValue private String role; @ValueMapValue private String photo; @ValueMapValue private String linkedin; private boolean hasLinkedin; @PostConstruct protected void init() { hasLinkedin = StringUtils.isNotBlank(linkedin); } @Override public String getName() { return name; } @Override public String getRole() { return role; } @Override public String getPhoto() { return photo; } @Override public String getLinkedin() { return linkedin; } @Override public boolean isHasLinkedin() { return hasLinkedin; } }
TeamBlockModelImpl.java — parent model reading the multifield
@Model(adaptables = Resource.class, ...) public class TeamBlockModelImpl implements TeamBlockModel { @SlingObject private Resource currentResource; private List<TeamMember> members = new ArrayList<>(); @PostConstruct protected void init() { // Get the "team" container node Resource teamNode = currentResource.getChild("team"); if (teamNode != null) { // Iterate over all child nodes (item0, item1, item2, ...) for (Resource itemResource : teamNode.getChildren()) { // Adapt each child to the TeamMember model TeamMember member = itemResource.adaptTo(TeamMember.class); if (member != null && StringUtils.isNotBlank(member.getName())) { members.add(member); } } } } @Override public List<TeamMember> getMembers() { return members; } }
