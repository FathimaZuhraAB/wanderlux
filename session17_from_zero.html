✅ Quick Quiz
🤔
Before anything — what is AEM?
AEM (Adobe Experience Manager) is like a very powerful website-building and content-management system — like WordPress, but for huge companies like banks, airlines, and retail chains.

Inside AEM, everything — pages, images, settings — is stored in a big database called the JCR (Java Content Repository). Think of it like a giant folder system on a computer, but stored in a database.

Your job as an AEM developer is to write Java code that talks to this database — reading pages, writing data, processing things in the background.
🎯 The scenario we will use throughout this session
You work for a hospital. The hospital uses AEM to manage its content. You are asked to build a background program that runs every night at midnight. This program:

1. Reads all patient appointment pages from the database
2. Counts how many appointments are scheduled for tomorrow
3. Writes a summary report so doctors can see it in the morning

This background program is called a "scheduled job" or OSGi service.
❓ Three problems you immediately face
1
Who does this background program log in as?
It needs to access the database to read appointment pages. It needs a user account. But whose? You can't use the admin account — that's too dangerous. → This is solved by Service Users
2
The program runs at midnight. You're asleep. How do you know if it worked?
You need the program to write a diary of everything it did — "I read 200 appointments... I wrote the report... ERROR: disk is full." In the morning you read this diary. → This is solved by Logging
3
You have 3 computers — your laptop, a test server, and the live hospital server. Each uses a different database address. How does the same code know which database to connect to?
You can't hardcode the database address into your code — it changes per computer. → This is solved by Runmodes
🔑
Service Users
A service user is a fake account that only your background program uses. It has no password, cannot log into AEM's website, and can only access the specific folders you allow.
🏥 Real life analogy
🏥
In a real hospital, a cleaning robot works at night. It has an access card that only opens: the patient wards (to clean) and the supply closet (to get cleaning supplies).

It cannot open the pharmacy, the operating theatre, the CEO's office, or the server room.

If the robot malfunctions and starts going to the wrong places — it physically cannot get in. The damage is contained.

A service user is that access card. Your background program can only go where you explicitly allow it.
❌ Why not just use the admin account?
❌ Using admin (WRONG)

Admin can read, write, DELETE everything in the entire system. One bug in your code and it could delete all 10,000 patient pages. Like giving the cleaning robot the master key to every door in the hospital — including the medicine cabinet and the patient records room.

✅ Using service user (RIGHT)

Can only access exactly what you allow. A bug can only affect those specific folders. In AEM Cloud Service (the modern version), admin access in code is completely blocked. You MUST use service users.

🔢 4 steps to create and use a service user
1
Create the user account
Just like creating a user on any system. Give it a name. This name will identify it everywhere. Name it clearly — what does it do? For our hospital program: appointment-report-service
2
Give it permissions (what doors can it open?)
Tell AEM: this user can READ from /content/hospital/appointments and can READ+WRITE to /var/hospital/reports. Nothing else.
3
Create a "nickname" mapping
Your Java code doesn't use the user's name directly. It uses a nickname like "report-subservice". A config file maps this nickname to the real user. This way if you rename the user, you only change one config file — not all your code.
4
Use the nickname in your Java code
Your Java code says "give me a connection using nickname report-subservice". AEM looks up the nickname, finds the real user, opens a database connection as that user.
📝 Step 2: The permissions file — read every line
This file tells AEM what the service user can access. You write it once and deploy it with your code.
📁 File name: org.apache.sling.jcr.repoinit.RepositoryInitializer-hospital.xml
1<!-- This is an XML file. XML uses tags like <this> -->
2<jcr:root jcr:primaryType="sling:OsgiConfig"
3 scripts="[
4 create service user appointment-report-service,
5 set ACL for appointment-report-service
6 allow jcr:read on /content/hospital/appointments
7 allow jcr:read\,jcr:write on /var/hospital/reports
8 end
9 ]"/>
L2
jcr:primaryType="sling:OsgiConfig" — This is the magic label that tells AEM: "this is a settings file". Without this exact label, AEM ignores the entire file. Think of it like writing "OFFICIAL DOCUMENT" at the top of a form — without it, nobody processes it.
L3
scripts="[...]" — scripts is the name of the setting. Everything inside the square brackets [ ] is a list of instructions AEM runs at startup. Like a checklist AEM goes through every time it starts.
L4
create service user appointment-report-service — "Make a service user account called appointment-report-service." If this user already exists (e.g. AEM was restarted), this line is safely skipped. AEM doesn't create duplicates. The comma at the end means "next instruction follows".
L5
set ACL for appointment-report-service — ACL = Access Control List = the list of permissions. This line says "I'm about to define what this user can do. Everything until the word 'end' is their permission list."
L6
allow jcr:read on /content/hospital/appointments — Three parts: allow = give this permission (not take away). jcr:read = permission to read content, like opening a file to view it but not edit it. on /content/hospital/appointments = ONLY on this path. Trying to read /content/hospital/doctors would return empty — access denied.
L7
allow jcr:read,jcr:write on /var/hospital/reports — On the reports folder, allow both reading AND writing. jcr:write = creating, editing, deleting content. The backslash \ before the comma is just XML formatting — it means "literal comma". The program needs write access here because it creates the report file.
L8
end — Closes the "set ACL" block. REQUIRED. If you forget this word, AEM's parser crashes and the service user gets NO permissions at all. Your code will fail silently — getting empty results instead of data.
📝 Step 3: The nickname mapping file
This file connects a short nickname (used in Java) to the real service user account.
📁 File name: com.adobe.granite.auth.impl.ServiceUserMapperImpl.amended-hospital.xml
1<jcr:root jcr:primaryType="sling:OsgiConfig"
2 servicemapping="[
3 com.hospital.myapp:report-subservice=appointment-report-service
4 ]"/>
L3
This one line has 3 parts separated by : and =

com.hospital.myapp — Your project/bundle name. Think of it as "which application owns this mapping". This prevents conflicts if two different apps both happen to use the nickname "report-subservice".

:report-subservice — The nickname. You make this up. Your Java code will use exactly this string. Think of it like a codename — "Operation Report-Subservice" → actually means "use the appointment-report-service account".

=appointment-report-service — The actual service user account name from Step 1. Must be spelled exactly the same. One typo here = the program crashes with "login failed" at midnight.
📝 Step 4: The Java code — with every line explained simply
This is your actual background program. Read it like reading English — I'll explain every single line.
📁 AppointmentReportJob.java
1import org.apache.sling.api.resource.ResourceResolver;
2import org.apache.sling.api.resource.ResourceResolverFactory;
3import org.apache.sling.api.resource.LoginException;
4import org.osgi.service.component.annotations.Component;
5import org.osgi.service.component.annotations.Reference;
6import org.slf4j.Logger;
7import org.slf4j.LoggerFactory;
8import java.util.Collections;
9
10@Component(service = Runnable.class,
11 property = {"scheduler.expression=0 0 0 * * ?"})
12public class AppointmentReportJob implements Runnable {
13
14 private static final Logger log =
15 LoggerFactory.getLogger(AppointmentReportJob.class);
16
17 @Reference
18 private ResourceResolverFactory resolverFactory;
19
20 public void run() {
21 log.info("Appointment report job started");
22
23 try (ResourceResolver resolver =
24 resolverFactory.getServiceResourceResolver(
25 Collections.singletonMap(
26 ResourceResolverFactory.SUBSERVICE,
27 "report-subservice"))) {
28
29 Resource appts = resolver.getResource(
30 "/content/hospital/appointments");
31
32 if (appts == null) {
33 log.error("Cannot read appointments - check permissions");
34 return;
35 }
36
37 log.info("Report written successfully");
38 resolver.commit();
39
40 } catch (LoginException e) {
41 log.error("Login failed - check service user mapping", e);
42 }
43 }
44}
L1-8
import statements — Like telling your program "I need these tools". Think of imports like choosing which tools to bring before starting a job. ResourceResolver = the database connection tool. Logger = the diary-writing tool. Collections = a helper for creating lists/maps. You must declare which tools you need before using them.
L10
@Component(service = Runnable.class) — @Component means "register this class with AEM's system (OSGi)". Without @Component, AEM doesn't even know this class exists. service = Runnable.class means "tell AEM this is a Runnable — something that can be run/executed". AEM's scheduler looks for all Runnables and schedules them.
L11
scheduler.expression=0 0 0 * * ? — This is a cron expression. It's a time schedule written in a specific format. Read it as 6 numbers/symbols: 0=second 0, 0=minute 0, 0=hour 0, *=every day, *=every month, ?=any day of week. Translation: run at 00:00:00 (midnight) every single day.
L14-15
Logger log = LoggerFactory.getLogger(...) — This creates your "diary writer". Every class needs exactly one of these. private = only this class uses it. static = shared across all uses of this class (more efficient). final = never changes. LoggerFactory.getLogger(AppointmentReportJob.class) = create a diary writer that labels every entry with this class's name, so you know where each diary entry came from.
L17-18
@Reference ResourceResolverFactory resolverFactory — @Reference means "AEM, please find this service and give it to me automatically." You don't create resolverFactory yourself — AEM injects it. Think of it like an employee saying "I need a laptop" — IT department (AEM/OSGi) brings one to you. ResourceResolverFactory is the factory that creates database connections (ResourceResolvers).
L23-27
try (ResourceResolver resolver = ...getServiceResourceResolver(...)) — Two concepts here:

try(...) = try-with-resources. The ResourceResolver is a database connection. When you're done, you MUST close it. This special syntax auto-closes it when the block ends — even if something crashes. If you forget to close it, AEM leaks connections until it runs out of memory and crashes.

Collections.singletonMap(SUBSERVICE, "report-subservice") = Create a tiny dictionary with one entry: {key: "sling.service.subservice", value: "report-subservice"}. This is how you tell AEM which nickname/subservice to look up. AEM finds "report-subservice" → looks in the mapping file → finds it maps to "appointment-report-service" → opens a connection as that user.
L29-30
resolver.getResource("/content/hospital/appointments") — "Open the appointments folder in the database." The resolver runs AS the service user, so it can only open what that user is allowed to see. If the service user doesn't have read permission here, this returns null (empty) — not an error, just nothing.
L32-35
if (appts == null) { log.error(...); return; } — Always check if what you got is null (empty). If the path doesn't exist OR the user doesn't have permission, you get null. Logging the error tells you exactly what went wrong. return = stop here, don't continue with null data (which would crash on the next line).
L38
resolver.commit() — Think of this like pressing Save in a document editor. All your writes to the database are held in memory until you call commit(). Without commit(), when the connection closes, all your writes disappear as if they never happened.
L40-41
catch (LoginException e) — If the service user login fails (typo in mapping file, user doesn't exist, wrong bundle name), this block runs instead of the normal code. log.error("...", e) — the e at the end is CRITICAL. It tells the logger to also write the full error details and the line number where it crashed. Without it, you'd only see "Login failed" with no info about what actually went wrong.

Logging — Your Code's Diary
Your program runs at midnight while you sleep. Logging is how your program writes a detailed diary of everything it does, so you can read it next morning and know exactly what happened.
✈️ Real life analogy
✈️
Every airplane has a black box. It records everything: altitude, speed, every button pressed, every warning triggered, every engine reading — with timestamps.

If a flight has a problem, investigators pull out the black box and read the entire flight history. They can reconstruct exactly what happened, second by second.

Your log file is the black box for your background job. Every night it records what happened. When something goes wrong, you read the log and reconstruct the problem.
5 levels of log messages — from quiet to loud
Think of it like writing notes in a diary with different importance levels:
🔬
TRACE — Every tiny micro-detail. "Processing item 1... Processing item 2..." Almost never used. Way too noisy.
🔍
DEBUG — Developer details. "The appointment count is 47. Cache was empty so fetched fresh." Use while writing code. Turn OFF in production.
📋
INFO — Important milestones. "Report job started." "Report completed. 200 appointments processed." Always on — even in production.
⚠️
WARN — Something unusual, but the program kept running. "Doctor Smith has no email — skipped notification." Always on in production.
🚨
ERROR — Something broke and needs fixing now. "Could not write report: disk is full." Always on. Always investigate immediately.
📝 How to write logs — line by line
📁 Inside any Java class
1// Step 1: Set up the logger — do this ONCE per class, outside all methods
2private static final Logger log =
3 LoggerFactory.getLogger(AppointmentReportJob.class);
4
5// Step 2: Use it inside your methods
6log.info("Report job started");
7
8// To include a variable value, use {} as a placeholder
9log.info("Processed {} appointments today", count);
10// {} gets replaced by the variable 'count' — output: "Processed 47 appointments today"
11
12// Multiple variables: use multiple {} placeholders
13log.info("Done. Processed={}, Skipped={}", processed, skipped);
14
15// For errors: ALWAYS pass the exception as last argument
16log.error("Failed to write report", e);
17// The 'e' automatically adds the full crash details to the log
L2-3
private static final Logger log = LoggerFactory.getLogger(AppointmentReportJob.class)

private = only this class uses this diary writer
static = create it once and share it — don't create a new one for every method call
final = once created, never replaced
Logger = the type (think of it as "type: diary")
log = the variable name (you'll write log.info(), log.error() etc.)
LoggerFactory.getLogger(AppointmentReportJob.class) = create a diary writer that stamps every entry with this class's name. In the actual log file, each entry will show "AppointmentReportJob" so you know which class wrote it.
L9
log.info("Processed {} appointments today", count)

Why use {} instead of just writing the number directly?

If you write log.info("Processed " + count + " appointments"), Java builds that string every time this line runs — even on days when info logging is turned off. That wastes memory and CPU.

The {} approach is lazy — Java only builds the string when the message will actually be written. In production with INFO turned off, zero string building happens.
L16
log.error("Failed to write report", e)

The e is the exception (error) object. When you pass it as the last argument to log.error(), the logger automatically appends the full crash report — including which exact line of code failed, and every method that was called leading up to the crash. This is called a stack trace.

Without e: you see "Failed to write report". You have no idea where it crashed.
With e: you see "Failed to write report" + 20 lines showing exactly where and why it crashed.
❌ Never do these:
System.out.println("debug: " + value) — this doesn't go to log files. Completely useless in AEM.
log.error(e.getMessage()) — you lose the stack trace. Use log.error("msg", e) instead.
log.debug("value = " + value) — use log.debug("value = {}", value) instead.

Runmodes — Right Settings for Each Environment
Your code is the same on every computer. But the settings (database address, log level, API keys) are different per environment. Runmodes make AEM automatically load the right settings based on where it's running.
🏥 Hospital analogy
👨‍⚕️
A medical student (your code) rotates through three hospitals:

🏠 Training hospital (dev) — uses mannequins, practice equipment, fake medicine. Mistakes don't hurt anyone.

🏥 Teaching hospital (stage/QA) — real patients, but supervised. Extra logging, no live billing.

🏨 Real hospital (prod) — fully live, real patients, real money, no extra logging.

The same doctor (same code). Different protocols and equipment (different config). The hospital they're in is their runmode.
📁 How folder names control which settings load
You create multiple folders. AEM reads only the folders that match the active runmodes:
📁 Folder structure in your project
apps/hospital-app/osgiconfig/
│
├── config/ ← loaded on ALL computers always
│ └── EmailService.xml ← same email settings everywhere
│
├── config.author/ ← loaded ONLY on author servers
│ └── WorkflowConfig.xml
│
├── config.author.dev/ ← loaded ONLY on author + dev computer
│ └── DatabaseConfig.xml ← points to: localhost/hospitaldb_dev
│
├── config.author.stage/ ← loaded ONLY on author + stage server
│ └── DatabaseConfig.xml ← points to: stage-server/hospitaldb_stage
│
└── config.author.prod/ ← loaded ONLY on author + prod server
└── DatabaseConfig.xml ← points to: prod-server/hospitaldb_live
You deploy the same package to all three environments. AEM looks at its active runmodes and loads only the matching folders. The DatabaseConfig.xml in config.author.dev loads on your laptop. The one in config.author.prod loads on the live server. Your Java code doesn't know or care — it just reads the database URL from the config and gets the right one for its environment.
🚀 How to tell AEM which runmode it's on
📁 When starting AEM — command line
1# -r means "runmodes". Comma separates multiple runmodes.
2# On your developer laptop:
3java -jar aem-quickstart.jar -r author,dev
4# AEM loads: config/ + config.author/ + config.author.dev/
5
6# On the staging test server:
7java -jar aem-quickstart.jar -r author,stage
8# AEM loads: config/ + config.author/ + config.author.stage/
9
10# On the live production server:
11java -jar aem-quickstart.jar -r author,prod
12# AEM loads: config/ + config.author/ + config.author.prod/
L3
java -jar aem-quickstart.jar — this starts AEM. Like double-clicking an application, but from the command line. -r author,dev — the -r flag passes the runmodes. "author" tells AEM it's an author instance (where editors create content). "dev" tells it this is a development environment. Both are active simultaneously, so folders like config.author.dev/ match and get loaded.
📝 A real settings file — line by line
Each DatabaseConfig.xml is identical in structure but has different values per environment:
📁 config.author.dev/com.hospital.app.DatabaseConfig.xml
1<jcr:root jcr:primaryType="sling:OsgiConfig"
2 db.url="jdbc:mysql://localhost:3306/hospital_dev"
3 db.max.connections="{Long}5"
4 db.debug.mode="{Boolean}true"/>
L1
jcr:primaryType="sling:OsgiConfig" — the magic label again. Without this, AEM ignores the file. The filename com.hospital.app.DatabaseConfig.xml tells AEM which Java service this settings file belongs to. AEM matches the filename to the Java class and feeds these settings into it.
L2
db.url="jdbc:mysql://localhost:3306/hospital_dev" — a custom setting called "db.url". In the dev version, this points to your laptop's local database. In the prod version (same property name, different file), this points to the live hospital server. Your Java code reads "db.url" — it doesn't know which environment it's on. AEM already loaded the right value.
L3
{Long}5 — The number 5, but with a type label. In XML, everything is text by default. If your Java code expects a number (int/long) and XML gives it the text "5", it could misinterpret it. The {Long} prefix tells AEM: "convert this text into an actual number before giving it to the Java class." Other types: {Boolean}true, {Double}3.14
L4
{Boolean}true — In dev, debug mode is on (extra logging). In the prod version of this file, this would be {Boolean}false — no debug mode on the live server. Same setting name, different values, loaded automatically based on runmode.

Felix Console — AEM's Control Room
The Felix Console is a webpage inside AEM at /system/console. It shows you the live internal state of everything running inside AEM. When something breaks, this is where you go to diagnose it.
🚗 Real life analogy
🔧
When your car has a problem, a mechanic plugs an OBD diagnostic device into it. The screen immediately shows: "Engine sensor 4 is faulty. Coolant temperature too high. Fuel injector 2 misfiring."

Without the OBD device, the mechanic has to guess — maybe check the engine manually, test parts one by one. Could take hours.

Felix Console is your OBD device for AEM. Instead of guessing why your code isn't working, Felix Console immediately shows you the problem.
6 pages you must know inside Felix Console
1
/system/console/bundles — "Did my code deploy?"
Shows all code bundles (JAR files) installed in AEM. Your code is one bundle. If it shows Active (green) = your code is loaded and running. If it shows Installed (red) = your code failed to start. Click on it to see WHY — it'll show which missing dependency caused the failure. Like checking if a new app installed successfully on your phone.
2
/system/console/components — "Is my @Component working?"
Shows every @Component class. Each has a status: Active (working) or Unsatisfied (broken). Unsatisfied means one of your @Reference fields couldn't be filled — AEM couldn't find the service you asked for. Click on your component to see exactly which @Reference is missing. This is the most-used debugging page in AEM development.
3
/system/console/configMgr — "Did my settings load?"
Shows all OSGi configuration values currently active. Search for your service name and click it — you'll see all the property values loaded from your XML files. If your runmode config isn't applying, you'll see default values here instead of your XML values. You can also edit values temporarily here without redeploying.
4
/system/console/slinglog — "Change log levels live"
Shows all loggers. You can add a new logger for your package at DEBUG level, watch your log file fill up with detailed messages, understand the problem, then remove the debug logger. No code changes, no redeployment. Like turning on extra security cameras in just one room when you suspect something is happening there.
5
/system/console/requests — "Which code handled my request?"
Shows the last 100 HTTP requests to AEM. For each request, shows which Java class or script handled it and how long it took. Useful when a page shows blank or wrong content — you can see exactly which code was chosen to render it.
6
/system/console/jmx — "Is AEM healthy?"
Live metrics — how many items are in queues, how much cache is used, how many workflows are running, are there any blocked threads. Use this when AEM feels slow or something seems stuck in the background.
🐛 Real debugging story — 2 hours → 2 minutes
A developer built a new scheduled job. It wasn't running. He spent 2 hours:
— Re-reading his code ❌ (code was fine)
— Checking cron expression ❌ (was correct)
— Restarting AEM ❌ (didn't help)

A senior developer came over and opened /system/console/components.
Searched "AppointmentReportJob" → Status: Unsatisfied
Clicked on it → ✗ ResourceResolverFactory not satisfied
Searched "ResourceResolverFactory" → Status: Installed (not Active)
Clicked on it → Missing import: org.apache.sling.api.resource

Added one line to pom.xml (the Maven dependency). Deployed. Fixed in 4 minutes.

The Felix Console showed in 30 seconds what 2 hours of code-reading never revealed.
