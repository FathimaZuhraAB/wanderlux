What problem does Session 21 solve?
You write Java code. You deploy it to AEM. It crashes in production at 2 AM — a page breaks for thousands of users. You stay up all night fixing it.

Unit testing means you catch that crash on your laptop before it ever reaches production. Tests run in milliseconds, need no AEM running, and tell you exactly what broke and why.
The scenario we use throughout this session
You work for the same bank from Session 20. You write a Sling Model called LoanCalculatorModel — it reads loan amount, interest rate, and tenure from a page's properties and calculates the monthly EMI (Equated Monthly Instalment).

Before you deploy this to the live bank website, you must be 100% sure:

— It correctly reads the properties from the page
— The EMI calculation is mathematically correct
— It handles edge cases: zero interest rate, missing properties, negative values
— The external "CreditScoreService" it depends on is called correctly

All of this is verified by unit tests — without needing a running AEM instance.

What is JUnit? What is a unit test?
A unit test is a small Java program that calls your code and checks whether the result is what you expected. JUnit is the framework that makes writing these checks easy.
Real life analogy
🏭
Imagine a factory that makes calculators. Before any calculator leaves the factory, a tester sits down with it and runs a checklist:

✓ Does 2 + 2 = 4? PASS
✓ Does 10 ÷ 0 show an error? PASS
✓ Does the display turn on? PASS
✗ Does the % button work? FAIL — ship it back

JUnit is that tester's checklist. You write the checks once. Every time you change your code, JUnit runs all checks automatically and tells you if anything broke.
Why unit tests? — The cost of finding bugs at different stages
❌ Bug found in PRODUCTION

Real users are affected. Bank displays wrong EMI to customers. Legal liability. Emergency fix at 2 AM. Reputation damage. Cost: very high.

✅ Bug found by unit test

Found on your laptop in milliseconds before you even commit the code. Fix it immediately. No users affected. Cost: zero.

Tests also act as documentation. A new developer joining your team reads your tests and immediately understands: "This method must return 0 when interest rate is 0", "This method must throw an exception when loan amount is negative." No other document is needed.
Your first JUnit test — the simplest possible example
Before AEM, before Sling Models — just plain Java. The EMI calculator class and its test:
EmiCalculator.java — the class we want to test
1public class EmiCalculator {
2
3 // Calculate monthly EMI
4 // principal = loan amount, annualRate = yearly interest %, months = tenure
5 public double calculateEmi(double principal,
6 double annualRate,
7 int months) {
8 if (annualRate == 0) return principal / months;
9 double r = annualRate / 12 / 100;
10 return principal * r * Math.pow(1+r, months)
11 / (Math.pow(1+r, months) - 1);
12 }
13}
EmiCalculatorTest.java — tests for the above class
1import org.junit.jupiter.api.Test;
2import static org.junit.jupiter.api.Assertions.*;
3
4class EmiCalculatorTest {
5
6 EmiCalculator calc = new EmiCalculator();
7
8 @Test
9 void testNormalLoan() {
10 double emi = calc.calculateEmi(100000, 10, 12);
11 assertEquals(8791.59, emi, 0.01);
12 }
13
14 @Test
15 void testZeroInterestRate() {
16 double emi = calc.calculateEmi(120000, 0, 12);
17 assertEquals(10000.0, emi, 0.01);
18 }
19
20 @Test
21 void testResultIsNotNull() {
22 double emi = calc.calculateEmi(50000, 8, 24);
23 assertNotNull(emi);
24 assertTrue(emi > 0);
25 }
26}
L1
import org.junit.jupiter.api.Test — JUnit 5 (Jupiter) is the version used in modern AEM projects. The @Test annotation comes from this import. Without the import, Java doesn't know what @Test means.
L2
import static org.junit.jupiter.api.Assertions.* — the * means "import everything from Assertions". This gives you assertEquals, assertNotNull, assertTrue, assertFalse, assertThrows etc. The word static means you can call them without writing Assertions.assertEquals every time — just assertEquals.
L6
EmiCalculator calc = new EmiCalculator() — you create a real instance of the class you want to test. In unit testing, you test one class at a time. This line creates your "subject under test" — the object all the test methods will call.
L8
@Test — this annotation tells JUnit: "this method is a test. When you run the test suite, call this method and check if it passes or throws an exception." Without @Test, JUnit ignores the method entirely. Every test method must have @Test.
L9
void testNormalLoan() — test method naming convention: start with "test" or use a descriptive name that says what scenario is being tested. "testNormalLoan" = testing a standard loan calculation. The method takes no parameters and returns void. JUnit calls it automatically.
L10
double emi = calc.calculateEmi(100000, 10, 12) — calling the actual method we want to test. Loan ₹1,00,000 at 10% annual interest for 12 months. We store the result in emi. This is the "Act" step — actually calling the code.
L11
assertEquals(8791.59, emi, 0.01) — the core of the test. Three arguments: expected value (8791.59 — what we calculated by hand or formula), actual value (emi — what our code produced), delta (0.01 — acceptable difference for floating point numbers). If emi is not within 0.01 of 8791.59, the test FAILS and JUnit shows you: "Expected 8791.59 but was 8800.00". You immediately know something is wrong in the formula.
L14-18
testZeroInterestRate() — testing an edge case. What happens when interest is 0%? Simple division should give ₹10,000/month. This test ensures the special case handling on line 8 of EmiCalculator works correctly. Without this test, a future developer might remove the zero-check thinking it's unnecessary — causing a division error in production.
L24
assertTrue(emi > 0) — checks a boolean condition. Here: EMI must always be positive. assertTrue(condition) passes if condition is true, fails if false. Other useful assertions: assertFalse(x), assertNull(x), assertNotNull(x), assertThrows(Exception.class, () -> methodThatShouldThrow()).
What test results look like in your IDE (IntelliJ / Eclipse)
testNormalLoan() — PASSED (3ms)
testZeroInterestRate() — PASSED (1ms)
testResultIsNotNull() — FAILED: expected true but was false

AEM Mocks — testing AEM components without running AEM
Your Sling Models use ResourceResolver, ValueMap, Page objects, the JCR repository. These only exist inside a running AEM instance. So how do you test them without AEM?

The AEM Mocks framework (io.wcm.testing.aem-mock) creates a fake in-memory version of AEM's environment — a fake JCR, fake ResourceResolver, fake pages — all running in milliseconds inside your test.
Real life analogy
✈️
Pilots train in flight simulators. The simulator is not a real plane — it's a fake environment that looks and behaves exactly like a real plane. The pilot can practice emergency procedures, test reactions, make mistakes — all without risking a real aircraft or real passengers.

AEM Mocks is the flight simulator for your AEM code. It gives your Java code a fake AEM environment to run in. Your code doesn't know or care that it's fake — it behaves exactly as it would on a real AEM server.
The AemContext — the heart of AEM Mocks
AemContext is the main object from the AEM Mocks framework. It gives you a complete fake AEM environment: a fake JCR repository, a fake ResourceResolver, a fake request, fake Sling services. You create test content in it, then run your code against it.

This tests our LoanCalculatorModel — a Sling Model that reads loan properties from a page and exposes them to HTL.
src/test/resources/loan-page.json — fake page content for the test
1{
2 "jcr:primaryType": "cq:Page",
3 "jcr:content": {
4 "jcr:primaryType": "cq:PageContent",
5 "sling:resourceType": "bank/components/loan-calculator",
6 "jcr:title": "Home Loan Calculator",
7 "loanAmount": 500000,
8 "interestRate": 8.5,
9 "tenureMonths": 240
10 }
11}
Why JSON?
Instead of creating JCR nodes in Java code (verbose and complex), AEM Mocks lets you describe your test content in a simple JSON file. This JSON is loaded into the fake in-memory JCR. It represents a real page exactly as it would exist in AEM. The structure mirrors the JCR node structure: jcr:content is a child node with all the page properties. Keep these JSON files in src/test/resources/ — they are not deployed to AEM, only used during tests.
LoanCalculatorModelTest.java — full test class with every line explained
1import io.wcm.testing.mock.aem.junit5.AemContext;
2import io.wcm.testing.mock.aem.junit5.AemContextExtension;
3import org.junit.jupiter.api.Test;
4import org.junit.jupiter.api.extension.ExtendWith;
5import static org.junit.jupiter.api.Assertions.*;
6
7@ExtendWith(AemContextExtension.class)
8class LoanCalculatorModelTest {
9
10 private final AemContext ctx = new AemContext();
11
12 @Test
13 void testModelReadsLoanProperties() {
14
15 // ARRANGE — set up the fake AEM environment
16 ctx.load().json("/loan-page.json", "/content/bank/home-loan");
17 ctx.currentResource("/content/bank/home-loan/jcr:content");
18
19 // ACT — run the model
20 LoanCalculatorModel model =
21 ctx.request().adaptTo(LoanCalculatorModel.class);
22
23 // ASSERT — check the results
24 assertNotNull(model);
25 assertEquals(500000, model.getLoanAmount());
26 assertEquals(8.5, model.getInterestRate(), 0.001);
27 assertEquals(240, model.getTenureMonths());
28 assertTrue(model.getMonthlyEmi() > 0);
29 }
30
31 @Test
32 void testMissingLoanAmount_returnsZero() {
33 ctx.load().json("/empty-page.json", "/content/bank/empty");
34 ctx.currentResource("/content/bank/empty/jcr:content");
35 LoanCalculatorModel model =
36 ctx.request().adaptTo(LoanCalculatorModel.class);
37 assertEquals(0, model.getLoanAmount());
38 assertEquals(0.0, model.getMonthlyEmi(), 0.001);
39 }
40}
L7
@ExtendWith(AemContextExtension.class) — this tells JUnit 5: "before and after each test, use AemContextExtension to manage the AemContext lifecycle." AemContextExtension automatically resets the fake AEM environment between tests so one test's content doesn't leak into another test. Without this, test 2 would still see the fake page created in test 1.
L10
private final AemContext ctx = new AemContext() — creates the fake AEM environment. final means the reference never changes. private means only this test class uses it. The AemContext gives you: a fake ResourceResolver (ctx.resourceResolver()), a fake SlingHttpServletRequest (ctx.request()), and methods to load JSON content into the fake JCR.
L16
ctx.load().json("/loan-page.json", "/content/bank/home-loan") — loads the JSON file into the fake JCR at the given path. After this line, your test's fake AEM has a page at /content/bank/home-loan with all the properties from the JSON. The path /loan-page.json is relative to src/test/resources/ — that's where JUnit looks for test resource files.
L17
ctx.currentResource("/content/bank/home-loan/jcr:content") — sets the "current resource" for the fake request. In real AEM, when a browser opens a page, Sling sets the current resource to that page's jcr:content node. Your Sling Model's @ValueMapValue fields read from this current resource. This line simulates "a user just opened this page in their browser."
L20-21
ctx.request().adaptTo(LoanCalculatorModel.class) — runs your actual Sling Model. ctx.request() returns the fake SlingHttpServletRequest. adaptTo(LoanCalculatorModel.class) instantiates your model against the fake request — exactly as AEM would when rendering a real page. Your model's @ValueMapValue fields get populated from the fake JSON page, your @PostConstruct method runs, and you get back a fully populated model object.
L24-28
The assertions — checking every property

assertNotNull(model) — the model must not be null. If adaptTo returns null, all other assertions would throw NullPointerException. Always check not-null first.
assertEquals(500000, model.getLoanAmount()) — loan amount read from JSON must equal 500000. If the @ValueMapValue injection is wrong (wrong property name), this fails.
assertEquals(8.5, model.getInterestRate(), 0.001) — the 0.001 delta handles floating point precision issues in Java.
assertTrue(model.getMonthlyEmi() > 0) — the calculated EMI must be positive.
L31-39
testMissingLoanAmount_returnsZero() — tests the "empty page" scenario. What does the model do when no loan properties are set? It must return 0 for amount and EMI — not throw a NullPointerException. This is called testing the edge case or the "sad path." In banking code, this is critical — a crash on missing data could display wrong information to customers.

Mockito — creating fake versions of services
Your LoanCalculatorModel calls an external CreditScoreService to check the applicant's credit score. In tests, you cannot call the real credit score system — it costs money per API call, it's slow, and it might return different results each time.

Mockito creates a fake (mock) version of CreditScoreService that you control completely — you tell it exactly what to return for each input.
Real life analogy
🎭
Movie stunt doubles. When filming a dangerous scene, the director doesn't put the real actor at risk. A stunt double — someone who looks similar — does the dangerous part. The scene is filmed, everything works, and the real actor is safe.

A Mockito mock is a stunt double for your services. Your code thinks it's calling the real CreditScoreService. It's actually calling a fake one you created — that returns exactly what you need for your test, instantly, with no side effects.
Complete Mockito test — every line explained
LoanCalculatorModelMockitoTest.java
1import org.junit.jupiter.api.Test;
2import org.junit.jupiter.api.extension.ExtendWith;
3import org.mockito.InjectMocks;
4import org.mockito.Mock;
5import org.mockito.junit.jupiter.MockitoExtension;
6import static org.mockito.Mockito.*;
7import static org.junit.jupiter.api.Assertions.*;
8
9@ExtendWith(MockitoExtension.class)
10class LoanCalculatorModelMockitoTest {
11
12 @Mock
13 private CreditScoreService creditScoreService;
14
15 @InjectMocks
16 private LoanCalculatorModel loanModel;
17
18 @Test
19 void testEligible_whenCreditScoreIsHigh() {
20
21 // ARRANGE — tell the mock what to return
22 when(creditScoreService.getScore("PAN123"))
23 .thenReturn(780);
24
25 // ACT
26 boolean eligible = loanModel.isEligibleForLoan("PAN123");
27
28 // ASSERT
29 assertTrue(eligible);
30 verify(creditScoreService, times(1)).getScore("PAN123");
31 }
32
33 @Test
34 void testNotEligible_whenCreditScoreIsLow() {
35 when(creditScoreService.getScore("PAN456"))
36 .thenReturn(420);
37 boolean eligible = loanModel.isEligibleForLoan("PAN456");
38 assertFalse(eligible);
39 }
40
41 @Test
42 void testWhenCreditServiceThrowsException() {
43 when(creditScoreService.getScore("PANERR"))
44 .thenThrow(new RuntimeException("API timeout"));
45 boolean eligible = loanModel.isEligibleForLoan("PANERR");
46 assertFalse(eligible);
47 // model must gracefully return false, not crash!
48 }
49}
L9
@ExtendWith(MockitoExtension.class) — tells JUnit to use Mockito's extension. This activates the @Mock and @InjectMocks annotations. Without this line, @Mock does nothing — your fields stay null.
L12-13
@Mock CreditScoreService creditScoreService — Mockito creates a fake (mock) object of type CreditScoreService and assigns it to this field. You don't create the object — Mockito does. The mock has the same methods as the real service, but they return default values (0, null, false) unless you tell them what to return using when(...).thenReturn(...). The real CreditScoreService code is NEVER called.
L15-16
@InjectMocks LoanCalculatorModel loanModel — Mockito creates a real instance of LoanCalculatorModel AND automatically injects the mock creditScoreService into it. So your real model code runs, but when it calls creditScoreService.getScore(), it calls the fake mock — not the real API. This is the key: real code under test, fake dependencies.
L22-23
when(creditScoreService.getScore("PAN123")).thenReturn(780) — this is "stubbing" — programming the mock's behaviour. Read it as: "WHEN getScore is called with "PAN123", THEN return 780." This is the entire power of Mockito. You can simulate any scenario: high score (780), low score (420), exception thrown — without needing the real API. The mock only returns 780 when called with exactly "PAN123". Any other argument returns 0 (default).
L30
verify(creditScoreService, times(1)).getScore("PAN123") — verifies that your code actually CALLED the mock. Three things checked: (1) creditScoreService was called, (2) exactly times(1) — once, not twice or zero times, (3) specifically the getScore method with argument "PAN123". If your model has a bug that skips the credit check, this verification fails — even if the final result happened to be correct by coincidence.
L42-48
testWhenCreditServiceThrowsException() — the most important test. What happens when the external credit score API times out? Your model must NOT crash and throw the exception up to the page renderer (which would show a 500 error to a bank customer). It must catch the exception and return false (not eligible). This test simulates the API being down and checks your error handling. Without this test, you would not know this scenario works until a real API outage in production.

AEM Coding Best Practices
These are the rules that senior AEM developers follow — not because someone forced them, but because breaking these rules always causes problems later: code that is hard to maintain, bugs that are hard to find, performance issues in production.
Sling Model best practices
✅
Always use DefaultInjectionStrategy.OPTIONAL
If a @ValueMapValue property doesn't exist in the JCR (maybe an author left the field blank), and you don't use OPTIONAL, AEM throws an exception and the entire component fails to render. With OPTIONAL, missing properties just get the default value (null, 0, "") — the model still works, and you handle null gracefully in the getter.
Correct Sling Model declaration
1// CORRECT — with OPTIONAL strategy and proper adaptables
2@Model(
3 adaptables = SlingHttpServletRequest.class,
4 defaultInjectionStrategy = DefaultInjectionStrategy.OPTIONAL
5)
6public class LoanCalculatorModel {
7
8 @ValueMapValue
9 @Default(intValues = 0)
10 private int loanAmount;
11
12 @ValueMapValue
13 @Default(doubleValues = 0.0)
14 private double interestRate;
15
16 // All heavy computation goes in @PostConstruct, not getters
17 private double monthlyEmi;
18
19 @PostConstruct
20 protected void init() {
21 if (loanAmount > 0 && interestRate > 0) {
22 monthlyEmi = calculateEmi(loanAmount, interestRate, tenureMonths);
23 }
24 }
25
26 // Getter just returns the pre-calculated value — no logic here
27 public double getMonthlyEmi() { return monthlyEmi; }
28 public int getLoanAmount() { return loanAmount; }
29 public double getInterestRate(){ return interestRate; }
30}
L4
defaultInjectionStrategy=DefaultInjectionStrategy.OPTIONAL — with OPTIONAL, if a property doesn't exist on the JCR node, Sling just skips it and leaves the field at its default Java value (0 for int, null for String, false for boolean). Without OPTIONAL, a missing property causes the entire model instantiation to fail — adaptTo() returns null — and your component renders nothing.
L8-14
@ValueMapValue with @Default — @ValueMapValue reads a property from the resource's ValueMap. The property name is derived from the field name: field loanAmount reads property loanAmount from JCR. @Default provides an explicit default value when the property is missing — cleaner than checking for 0/null in every getter.
L19-24
@PostConstruct void init() — this method is called by Sling automatically AFTER all @ValueMapValue fields have been injected. This is where you do all your calculation work — reading from injected fields and computing derived values. NEVER put logic in getters — HTL templates call getters multiple times. If you calculate EMI inside getMonthlyEmi(), it recalculates on every call. @PostConstruct runs once; getters just return stored values.
Java best practices for AEM
🔒
Always close ResourceResolvers — use try-with-resources
Every unclosed resolver = one leaked JCR session. After hundreds of leaked sessions, AEM's memory fills up and it crashes. Always use try(ResourceResolver r = ...) {}
📝
Use constants for JCR property names
Never write "jcr:title" as a raw string in multiple places. If you ever rename the property, you have to find every occurrence. Instead: private static final String PROP_TITLE = "jcr:title" — change in one place, everything updates.
🚫
Never use admin session in code
Always use service users (Session 17). Admin has access to delete the entire repository. Your service user should have only the minimum permissions needed.
⚠️
Never use System.out.println — always use Logger
System.out doesn't write to log files. In production you'll never find your debug output. Always use log.debug("message {}", value).
🏗️
Keep business logic in Java, keep templates in HTL
No Java code in HTL files. No business logic in Sling Models (only data preparation). HTL = presentation only. Sling Model = data preparation only. Java service = business logic. This separation makes each part testable and maintainable independently.

HTL Best Practices & ACS Commons
HTL (HTML Template Language, also called Sightly) is AEM's templating language for rendering HTML. ACS Commons is a free library of ready-made utilities that solve common AEM problems so you don't have to build them yourself.
HTL best practices — the golden rules
Good vs bad HTL examples
1<!-- CORRECT: use data-sly-use to bind your Sling Model -->
2<sly data-sly-use.model="com.bank.aem.models.LoanCalculatorModel"/>
3
4<!-- Display only if loanAmount exists -->
5<div data-sly-test="${model.loanAmount}">
6 <p>Loan Amount: ${model.loanAmount}</p>
7 <p>Monthly EMI: ${model.monthlyEmi}</p>
8</div>
9
10<!-- Loop through a list -->
11<ul>
12 <li data-sly-list="${model.loanTypes}">
13 ${item.name} — ${item.rate}%
14 </li>
15</ul>
16
17<!-- WRONG: never put Java scriptlets in HTL -->
18<% double emi = 500000 * 0.085 / 12; %> <!-- NEVER -->
L2
data-sly-use.model="..." — instantiates your Sling Model and binds it to the variable model. After this, all your model's public getter methods become accessible as ${model.propertyName}. The full class name ensures AEM finds the right class even if multiple models have the same simple name.
L5
data-sly-test="${model.loanAmount}" — conditionally renders this element. If model.loanAmount is 0, null, empty, or false, the entire div and its contents are removed from the HTML output. Equivalent to an if statement. Prevents showing "Loan Amount: 0" when the field was never filled in.
L12
data-sly-list="${model.loanTypes}" — iterates over a List returned by your model. For each item in the list, renders the li element. The variable item is automatically available inside the loop (you can also name it: data-sly-list.loan="${model.loanTypes}" makes it loan). Equivalent to a for-each loop.
L18
NEVER use scriptlets <% %> — JSP-style scriptlets mix Java and HTML in the same file. They are impossible to test, hard to read, and AEM's security model restricts them. Always do all logic in the Sling Model and only display results in HTL.
ACS Commons — free tools that solve common AEM problems
ACS Commons (Adobe Consulting Services Commons) is an open-source library of ready-made AEM utilities. Instead of building common features from scratch, you use these — saving days of work.
🔧
Ensure Authorizable — create users/groups on deployment
Automatically creates service users and groups when your package deploys. Instead of manually creating users in every environment, you declare them in a config file and ACS Commons handles creation. Replaces the manual service user creation we discussed in Session 17 — same result, simpler config.
🖼️
Named Image Transforms — consistent image resizing
Instead of hardcoding image dimensions in your components, you define named transforms: "hero-image" = 1920×600 cropped. Your component references "hero-image" by name. When the design changes to 1920×800, you update one config and every hero image across the site updates — no code change needed.
📊
MCP (Manage Controlled Processes) — bulk operations
ACS Commons MCP provides a framework for running large bulk operations in AEM safely — bulk page moves, asset migrations, content reorganisation. Instead of writing your own batch job from scratch, you implement MCP's interfaces and get: a progress bar UI, pause/resume, error reporting, and rollback capability for free.
🚨
Error Page Handler — custom 404 and 500 pages
ACS Commons Error Page Handler lets you create custom error pages per section of your website. The bank's /en/ section shows a branded English error page. The /ta/ section shows a Tamil error page. Without ACS Commons, you'd need a custom Sling error handler servlet — ACS Commons does this with just configuration.
🔍
ACL Packager — export and import permissions
Exports all your AEM user permissions (ACLs) as a content package. You can import this package into a fresh environment and all permissions are instantly recreated — no manual re-configuration. Essential when setting up staging or production environments.
